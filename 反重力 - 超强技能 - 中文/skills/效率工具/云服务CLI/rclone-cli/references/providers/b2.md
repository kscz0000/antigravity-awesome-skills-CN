# Backblaze B2

B2 是 [Backblaze 的云存储系统](https://www.backblaze.com/cloud-storage)。

路径指定为 `remote:bucket`（或 `lsd` 命令使用 `remote:`）。也可以包含子目录，例如 `remote:bucket/path/to/dir`。

## 配置

以下是一个创建 b2 配置的示例。首先运行

```console
rclone config
```

这将引导你完成交互式设置过程。要进行身份验证，你需要账户 ID（短的十六进制数字）和主应用密钥（长的十六进制数字），或者使用应用密钥（推荐方式）。有关生成和使用应用密钥的更多详情，请参见下文。

此远程存储名为 `remote`，现在可以像这样使用

查看所有存储桶

```console
rclone lsd remote:
```

创建新存储桶

```console
rclone mkdir remote:bucket
```

列出存储桶内容

```console
rclone ls remote:bucket
```

将 `/home/local/directory` 同步到远程存储桶，删除存储桶中多余的文件。

```console
rclone sync --interactive /home/local/directory remote:bucket
```

### 应用密钥

B2 支持多个[针对 B2 存储桶的不同访问权限的应用密钥](https://www.backblaze.com/docs/cloud-storage-application-keys)。

你也可以在 rclone 中使用这些密钥；你需要使用 1.43 或更高版本的 rclone。

按照 Backblaze 文档创建具有所需权限的应用密钥，然后将 `applicationKeyId` 作为 `account`，将 `Application Key` 本身作为 `key`。

注意，你必须将 *applicationKeyId* 作为 `account` —— 不能使用主账户 ID。如果你尝试这样做，B2 将返回 401 错误。

### --fast-list

此远程存储支持 `--fast-list`，允许你用更多内存换取更少的事务数。详见 [rclone 文档](/docs/#fast-list)。

### 修改时间

修改时间以 `X-Bz-Info-src_last_modified_millis` 元数据的形式存储在对象上，采用 Backblaze 标准的自 1970-01-01 以来的毫秒数。其他工具应能将其用作修改时间。

修改时间在同步中使用并完全支持。注意，如果需要更新对象的修改时间，则会创建该对象的新版本。

### 受限文件名字符

除了[默认受限字符集](/overview/#restricted-characters)之外，以下字符也会被替换：

| 字符 | 值 | 替换为 |
| ---- |:---:|:-----:|
| \    | 0x5C | ＼    |

无效的 UTF-8 字节也会被[替换](/overview/#invalid-utf8)，因为它们不能用于 JSON 字符串。

请注意，Backblaze 自 2020 年 5 月起开始允许文件名中包含 \ 字符。Rclone 没有更改其编码，因为这可能导致同步重新传输文件。如果你想让 rclone 不替换 \，请参阅下面的 `--b2-encoding` 标志，并从字符串中移除 `BackSlash`。这可以在配置中设置。

### SHA1 校验和

文件的 SHA1 校验和在上传和下载时都会被检查，并将在同步过程中使用。

大于 `--b2-upload-cutoff` 中限制的大文件将分块上传，其 SHA1 将按照 Backblaze 建议以 `X-Bz-Info-large_file_sha1` 存储在对象上。

要使大文件上传时带有 SHA1 校验和，源端需要支持 SHA1 校验和。本地磁盘支持 SHA1 校验和，因此从本地磁盘传输大文件将具有 SHA1。有关哪些远程存储支持 SHA1 的详情，请参见[概览](/overview/#features)。

不支持 SHA1 的源端，特别是 `crypt`，将上传不带 SHA1 校验和的大文件。这可能会在未来修复（参见 [#1767](https://github.com/rclone/rclone/issues/1767)）。

小于 `--b2-upload-cutoff` 的文件无论来源如何，始终具有 SHA1。

### 传输

Backblaze 建议你同时进行大量传输以获得最大速度。在我的 SSD 笔记本电脑测试中，最佳设置约为 `--transfers 32`，不过更高的数值可能会带来轻微的速度提升。你的最佳数值可能因硬件、文件大小以及你希望对计算机的负载程度等因素而异。不过 `--transfers 4` 的默认值对 Backblaze B2 来说肯定太低了。

请注意，上传大文件（默认大于 200 MiB）默认将使用 96 MiB 的 RAM 缓冲区。任何时刻最多可以有 `--transfers` 个此类缓冲区同时使用，因此这设定了内存使用的上限。

### 版本

B2 的默认设置是保留文件的旧版本。这意味着当 rclone 上传文件的新版本时，它会创建一个[新版本](https://www.backblaze.com/docs/cloud-storage-file-versions)。同样，当你删除文件时，旧版本将被标记为隐藏，但仍然可用。

B2 是否保留文件的旧版本可以在每个存储桶的基础上调整，使用 B2 控制面板上的"生命周期设置"，或在创建存储桶时使用 [--b2-lifecycle](#b2-lifecycle) 标志，或在创建后使用 [rclone backend lifecycle](#lifecycle) 命令。

你可以使用 `--b2-hard-delete` 标志选择"硬删除"文件，这将在删除时永久移除文件而不是隐藏它们。

文件的旧版本（如果可用）可以通过 `--b2-versions` 标志查看。这些版本可以使用 `delete` 按需删除。

还可以使用 `--b2-version-at` 标志查看存储桶在某个时间点的状态。这将显示该时间的文件版本，显示此后被删除的文件，并隐藏此后创建的文件。

如果你想删除所有旧版本和未完成的大文件上传，可以使用 `rclone cleanup remote:bucket` 命令，该命令将删除所有旧版本的文件，保留当前版本不变。你也可以提供路径，只删除该路径下的旧版本，例如 `rclone cleanup remote:bucket/path/to/stuff`。

请注意，`cleanup` 将移除存储桶中超过一天的未完成上传文件。如果你想要对过期时间有更多控制，可以运行 `rclone backend cleanup b2:bucket -o max-age=1h` 来移除所有超过一小时的未完成大文件上传，保留旧版本不变。

如果你想删除所有旧版本，保留当前文件和未完成的大文件不变，可以使用 [`rclone backend cleanup-hidden remote:bucket`](#cleanup-hidden) 命令。你也可以提供路径，只删除该路径下的旧版本，例如 `rclone backend cleanup-hidden remote:bucket/path/to/stuff`。

当你对存储桶执行 `purge` 时，当前版本和旧版本都将被删除，然后存储桶将被删除。

但是 `delete` 会使文件的当前版本变为隐藏的旧版本。

以下是一个会话示例，显示了列出和检索旧版本，然后对旧版本执行 `cleanup`。

使用 `--b2-versions` 标志显示当前版本和所有版本。

```console
$ rclone -q ls b2:cleanup-test
        9 one.txt

$ rclone -q --b2-versions ls b2:cleanup-test
        9 one.txt
        8 one-v2016-07-04-141032-000.txt
       16 one-v2016-07-04-141003-000.txt
       15 one-v2016-07-02-155621-000.txt
```

检索旧版本

```console
$ rclone -q --b2-versions copy b2:cleanup-test/one-v2016-07-04-141003-000.txt /tmp

$ ls -l /tmp/one-v2016-07-04-141003-000.txt
-rw-rw-r-- 1 ncw ncw 16 Jul  2 17:46 /tmp/one-v2016-07-04-141003-000.txt
```

清理所有旧版本并确认已删除。

```console
$ rclone -q cleanup b2:cleanup-test

$ rclone -q ls b2:cleanup-test
        9 one.txt

$ rclone -q --b2-versions ls b2:cleanup-test
        9 one.txt
```

#### 版本命名注意事项

使用 `--b2-versions` 标志时，rclone 依赖文件名来判断对象是否为版本。版本名称通过在文件名和扩展名之间插入时间戳来创建。

```console
        9 file.txt
        8 file-v2023-07-17-161032-000.txt
       16 file-v2023-06-15-141003-000.txt
```

如果存在与版本名称相同的真实文件，则 `--b2-versions` 的行为可能不可预测。

### 数据使用

了解不同场景下向服务器发送了多少请求是很有用的。

所有复制命令发送以下 4 个请求：

```text
/b2api/v4/b2_authorize_account
/b2api/v1/b2_create_bucket
/b2api/v1/b2_list_buckets
/b2api/v1/b2_list_file_names
```

`b2_list_file_names` 请求将针对远程路径中的每 1000 个文件发送一次，提供列出文件的校验和和修改时间。从 1.33 版本起，问题 [#818](https://github.com/rclone/rclone/issues/818) 导致 B2 与 Crypt 一起使用时会发送额外请求。当复制操作不需要上传任何文件时，不会再发送更多请求。

上传不需要分块的文件，每个文件上传发送 2 个请求：

```text
/b2api/v1/b2_get_upload_url
/b2api/v1/b2_upload_file/
```

上传需要分块的文件，将发送 2 个请求（分别用于开始和完成上传）以及每个分块 2 个请求：

```text
/b2api/v1/b2_start_large_file
/b2api/v1/b2_get_upload_part_url
/b2api/v1/b2_upload_part/
/b2api/v1/b2_finish_large_file
```

#### 版本

版本可以通过 `--b2-versions` 标志查看。设置后，rclone 将显示并操作文件的旧版本。例如

不使用 `--b2-versions` 列出

```console
$ rclone -q ls b2:cleanup-test
        9 one.txt
```

使用时

```console
$ rclone -q --b2-versions ls b2:cleanup-test
        9 one.txt
        8 one-v2016-07-04-141032-000.txt
       16 one-v2016-07-04-141003-000.txt
       15 one-v2016-07-02-155621-000.txt
```

表明当前版本不变，但可以看到旧版本。这些版本名称后附加了上传到服务器时的 UTC 日期时间，精确到毫秒。

注意，使用 `--b2-versions` 时不允许任何文件写入操作，因此你不能上传或删除文件。

### B2 与 rclone link

Rclone 支持为私有 B2 存储桶生成文件共享链接。可以是针对某个文件的，例如：

```console
./rclone link B2:bucket/path/to/file.txt
https://f002.backblazeb2.com/file/bucket/path/to/file.txt?Authorization=xxxxxxxx

```

或者如果对目录运行，你将得到：

```console
./rclone link B2:bucket/path
https://f002.backblazeb2.com/file/bucket/path?Authorization=xxxxxxxx
```

然后你可以在该目录下的任何文件路径上使用授权令牌（URL 中 `?Authorization=` 及之后的部分）。例如：

```text
https://f002.backblazeb2.com/file/bucket/path/to/file1?Authorization=xxxxxxxx
https://f002.backblazeb2.com/file/bucket/path/file2?Authorization=xxxxxxxx
https://f002.backblazeb2.com/file/bucket/path/folder/file3?Authorization=xxxxxxxx

```

属性：

- Config:      version_at
- Env Var:     RCLONE_B2_VERSION_AT
- Type:        Time
- Default:     off

#### --b2-upload-cutoff

切换到分块上传的阈值。

大于此大小的文件将以 "--b2-chunk-size" 的大小分块上传。

此值不应大于 4.657 GiB（== 5 GB）。

属性：

- Config:      upload_cutoff
- Env Var:     RCLONE_B2_UPLOAD_CUTOFF
- Type:        SizeSuffix
- Default:     200Mi

#### --b2-copy-cutoff

切换到多部分复制的阈值。

任何需要服务端复制且大于此大小的文件，将按此大小的分块进行复制。

最小值为 0，最大值为 4.6 GiB。

属性：

- Config:      copy_cutoff
- Env Var:     RCLONE_B2_COPY_CUTOFF
- Type:        SizeSuffix
- Default:     4Gi

#### --b2-chunk-size

上传分块大小。

上传大文件时，按此大小对文件进行分块。

必须能装入内存。这些分块在内存中缓冲，同时可能最多有 "--transfers" 个分块在进行中。

5,000,000 字节是最小大小。

属性：

- Config:      chunk_size
- Env Var:     RCLONE_B2_CHUNK_SIZE
- Type:        SizeSuffix
- Default:     96Mi

#### --b2-upload-concurrency

多部分上传的并发数。

这是同一文件中同时上传的分块数。

注意，分块存储在内存中，同时可能最多有 "--transfers" * "--b2-upload-concurrency" 个分块存储在内存中。

属性：

- Config:      upload_concurrency
- Env Var:     RCLONE_B2_UPLOAD_CONCURRENCY
- Type:        int
- Default:     4

#### --b2-disable-checksum

对大文件（> 上传阈值）禁用校验和。

通常 rclone 会在上传前计算输入的 SHA1 校验和，以便将其添加到对象的元数据中。这对数据完整性检查很有好处，但可能导致大文件上传前出现长时间延迟。

属性：

- Config:      disable_checksum
- Env Var:     RCLONE_B2_DISABLE_CHECKSUM
- Type:        bool
- Default:     false

#### --b2-download-url

自定义下载端点。

通常设置为 Cloudflare CDN URL，因为 Backblaze 为通过 Cloudflare 网络下载的数据提供免费出口流量。Rclone 通过发送 "Authorization" 请求头来处理私有存储桶。如果自定义端点重写了认证请求，例如在 Cloudflare Workers 中，则需要正确处理此请求头。如果你想使用 Backblaze 提供的端点，请留空。

此处提供的 URL 应包含协议，不应有尾部斜杠，也不应指定 /file/bucket 子路径，因为 rclone 将以 "{download_url}/file/{bucket_name}/{path}" 的格式请求文件。

示例：

属性：

- Config:      download_url
- Env Var:     RCLONE_B2_DOWNLOAD_URL
- Type:        string
- Required:    false

#### --b2-lifecycle

设置创建存储桶时已删除文件的保留天数。

创建存储桶时，此参数用于为整个存储桶创建生命周期规则。

如果 lifecycle 为 0（默认值），则不创建生命周期规则，因此应用 B2 的默认行为。即在删除和覆盖时创建文件版本并无限期保留。

此参数的最小值为 1 天。

你也可以在配置中启用 hard_delete，这意味着删除操作不会产生版本，但覆盖操作仍会产生版本。

参见：[rclone backend lifecycle](#lifecycle) 用于存储桶创建后设置生命周期。


属性：

- Config:      lifecycle
- Env Var:     RCLONE_B2_LIFECYCLE
- Type:        int
- Default:     0

### lifecycle

读取或设置存储桶的生命周期。

```console
rclone backend lifecycle remote: [options] [<arguments>+]
```

此命令可用于读取或设置存储桶的生命周期。

显示当前生命周期规则：

```console
rclone backend lifecycle b2:bucket
```

这将输出类似以下内容，显示生命周期规则。

```json
[
    {
        "daysFromHidingToDeleting": 1,
        "daysFromUploadingToHiding": null,
        "daysFromStartingToCancelingUnfinishedLargeFiles": null,
        "fileNamePrefix": ""
    }
]
```

如果没有生命周期规则（默认），则只返回 `[]`。

重置当前生命周期规则：

```console
rclone backend lifecycle b2:bucket -o daysFromHidingToDeleting=30
rclone backend lifecycle b2:bucket -o daysFromUploadingToHiding=5 -o daysFromHidingToDeleting=1
```

这将运行，然后如上打印新的生命周期规则。

Rclone 只允许你为整个存储桶设置生命周期，fileNamePrefix = ""。

你无法使用 B2 禁用版本控制。最好的方法是将 daysFromHidingToDeleting 设为 1 天。你也可以在配置中启用 hard_delete，这意味着删除操作不会产生版本，但覆盖操作仍会产生版本。

```console
rclone backend lifecycle b2:bucket -o daysFromHidingToDeleting=1
```

参见：<https://www.backblaze.com/docs/cloud-storage-lifecycle-rules>

选项：

- "daysFromHidingToDeleting"：文件隐藏这么多天后将被删除。0 表示关闭。
- "daysFromStartingToCancelingUnfinishedLargeFiles"：这么多天后取消任何未完成的大文件版本。
- "daysFromUploadingToHiding"：上传文件这么多天后文件被隐藏。

### cleanup

移除未完成的大文件上传。

```console
rclone backend cleanup remote: [options] [<arguments>+]
```

此命令移除超过 max-age 的未完成大文件上传，默认为 24 小时。

注意，你可以使用 --interactive/-i 或 --dry-run 与此命令来查看它会执行什么操作。

```console
rclone backend cleanup b2:bucket/path/to/object
rclone backend cleanup -o max-age=7w b2:bucket/path/to/object
```

持续时间的解析方式与 rclone 其他部分一致，2h、7d、7w 等。

选项：

- "max-age"：要删除的上传的最大存活时间。

### cleanup-hidden

移除文件的旧版本。

```console
rclone backend cleanup-hidden remote: [options] [<arguments>+]
```

此命令移除文件的所有旧隐藏版本。

注意，你可以使用 --interactive/-i 或 --dry-run 与此命令来查看它会执行什么操作。

```console
rclone backend cleanup-hidden b2:bucket/path/to/dir
```

## 限制

B2 后端不支持 `rclone about`。不具有此功能的后端无法确定 rclone 挂载的可用空间，也无法将策略 `mfs`（最多可用空间）用作 rclone union 远程存储的成员。

参见[不支持 rclone about 的后端列表](https://rclone.org/overview/#optional-features)和 [rclone about](https://rclone.org/commands/rclone_about/)。
