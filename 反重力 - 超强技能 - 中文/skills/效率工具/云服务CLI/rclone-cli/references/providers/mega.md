# Mega

[Mega](https://mega.nz/) 是一项云存储和文件托管服务，
以其安全特性而著称——所有文件在上传前都会在本地进行加密。
这可以防止任何人在不知晓加密密钥的情况下访问文件
（包括 Mega 的员工）。

这是 rclone 的 Mega 后端，使用相同的客户端加密
支持 Mega 的文件传输功能。

**注意** [MEGA S4 对象存储](/s3#mega) 是一个兼容 S3 的对象存储，
它同样可与 rclone 配合使用，推荐用于新建项目。

路径以 `remote:path` 的形式指定。

路径可以根据需要嵌套任意深度，例如 `remote:directory/subdirectory`。


## 配置

下面示例演示如何创建一个名为 `remote` 的远端。首先运行：

```console
rclone config
```

该命令将引导你完成交互式配置流程：

```text
No remotes found, make a new one?
n) New remote
s) Set configuration password
q) Quit config
n/s/q> n
name> remote
Type of storage to configure.
Choose a number from below, or type in your own value
[snip]
XX / Mega
   \ "mega"
[snip]
Storage> mega
User name
user> you@example.com
Password.
y) Yes type in my own password
g) Generate random password
n) No leave this optional password blank
y/g/n> y
Enter the password:
password:
Confirm the password:
password:
Remote config
Configuration complete.
Options:
- type: mega
- user: you@example.com
- pass: *** ENCRYPTED ***
Keep this "remote" remote?
y) Yes this is OK
e) Edit this remote
d) Delete this remote
y/e/d> y
```

**注意：** 加密密钥必须在通过浏览器进行常规登录之后生成，
否则在 `rclone` 中使用这些凭据将失败。

完成配置后，你可以像下面这样使用 `rclone`（将 `remote` 替换为你为远端设置的名称）：

列出 Mega 顶层目录：

```console
rclone lsd remote:
```

列出 Mega 中的所有文件：

```console
rclone ls remote:
```

将本地目录复制到名为 backup 的 Mega 目录中：

```console
rclone copy /home/source remote:backup
```

### 修改时间和哈希

Mega 目前不支持修改时间或哈希。

### 受限的文件名字符

| 字符 | 取值 | 替换字符 |
| --------- |:-----:|:-----------:|
| NUL       | 0x00  | ␀           |
| /         | 0x2F  | ／          |

无效的 UTF-8 字节也会被 [替换](/overview/#invalid-utf8)，
因为它们无法用于 JSON 字符串。

### 重复文件

Mega 允许两个文件具有完全相同的名称和路径（与普通文件系统不同）。

重复文件会导致同步问题，你将在日志中看到关于重复文件的提示信息。

可使用 `rclone dedupe` 来修复重复文件。

### 登录失败

#### 对象未找到

如果你是第一次连接自己的 Mega 远端来测试访问与同步，
可能会收到类似下面的错误：

```text
Failed to create file system for "my-mega-remote:":
couldn't login: Object (typically, node or user) not found
```

[rclone 论坛](https://forum.rclone.org/search?q=mega) 中通常推荐
的诊断步骤从 **MEGAcmd** 工具开始。请注意，这里指的是位于
<https://github.com/meganz/MEGAcmd> 的官方 C++ 命令行工具，
而非已不再维护的 t3rm1n4l/megacmd 中的 Go 语言实现版本。

按照说明安装 MEGAcmd，并按其推荐方式尝试访问你的远端。
你可以借此判断能否通过 MEGAcmd 登录，
获取有助于排查问题的诊断信息，并在论坛中搜索或与他人协作。

```text
MEGA CMD> login me@example.com
Password:
Fetching nodes ...
Loading transfers from local cache
Login complete as me@example.com
me@example.com:/$
```

请注意，有用户反映密码中包含特殊字符时会遇到问题。
如果你无法使用 rclone 登录，但 MEGAcmd 可以正常登录，
那么可以尝试将密码临时改为纯字母数字字符，看是否能解决问题。

#### 重复命令导致访问被阻止

Mega 远端在"重度使用"下似乎会被阻止（拒绝登录）。
目前尚未摸清具体的封禁规则，但看起来与短时间内
连续执行多次 rclone 命令有关。

例如，连续执行 90 次以下命令 `rclone link
remote:file` 会导致远端被"封禁"。这种情况在需要获取
含数百个文件的目录的公开链接时并不罕见。大约一周后，
远端通常会恢复正常接受 rclone 登录。

你可以通过 `rclone mount` 挂载该远端来缓解这个问题。
挂载时登录，卸载时注销，仅此而已。你也可以运行 `rclone rcd`，
然后通过 `rclone rc` 经由 API 执行命令，以避免每次都登录。

Rclone 目前不会主动关闭 Mega 会话（你可以在网页界面中看到它们），
不过关闭这些会话并不能解决问题。

如果你将 rclone 命令间隔 3 秒执行，则可以避免远端被封禁。
我们尚未确定具体的封禁规则，所以也许可以连续执行 80 次而不等待，
然后等待 3 秒再继续，这样就能避开封禁……

请注意，以上结论是通过反复试验得出的，未必是一成不变的。

其他工具似乎不会触发这种封禁效果，因为它们采用
不同的工作方式（基于状态，使用 sessionID 而非登录），
这与当前无状态的 rclone 方式不兼容。

请注意，一旦被封禁，借助其他工具（如 megacmd）也未必能规避：
在被封禁的远端上相继观察到以下 megacmd 登录耗时：
7 分钟、20 分钟、30 分钟、30 分钟、30 分钟。
不过网页访问似乎不受影响。

围绕超时、限速、重试和 tpslimits 等方面的规避方案仍在调查之中，
如果你发现了相关信息，欢迎到论坛发帖分享。

因此，如果 rclone 一向运行良好，但突然无法登录，
而且你确认用户名和密码都没问题，那么很可能是远端
被暂时封禁了。

### 标准选项

以下是 mega (Mega) 特有的标准选项。

#### --mega-user

用户名。

属性：

- Config:      user
- Env Var:     RCLONE_MEGA_USER
- Type:        string
- Required:    true

#### --mega-pass

密码。

**提示** 此项的输入必须经过混淆处理——参见 [rclone obscure](/commands/rclone_obscure/)。

属性：

- Config:      pass
- Env Var:     RCLONE_MEGA_PASS
- Type:        string
- Required:    true

#### --mega-2fa

MEGA 账户的 2FA 验证码（如果账户已设置双因素认证）。

属性：

- Config:      2fa
- Env Var:     RCLONE_MEGA_2FA
- Type:        string
- Required:    false

### 进阶选项

以下是 mega (Mega) 特有的进阶选项。

#### --mega-session-id

会话标识（仅限内部使用）。

属性：

- Config:      session_id
- Env Var:     RCLONE_MEGA_SESSION_ID
- Type:        string
- Required:    false

#### --mega-master-key

主密钥（仅限内部使用）。

属性：

- Config:      master_key
- Env Var:     RCLONE_MEGA_MASTER_KEY
- Type:        string
- Required:    false

#### --mega-debug

输出更多 Mega 调试信息。

如果设置了此标志（同时使用 -vv），mega 后端会输出更详细的调试信息。

属性：

- Config:      debug
- Env Var:     RCLONE_MEGA_DEBUG
- Type:        bool
- Default:     false

#### --mega-hard-delete

永久删除文件，而不是放入回收站。

默认情况下，mega 后端会将所有删除操作放入回收站，
而不是永久删除。如果你指定此选项，rclone 将改为永久删除对象。

属性：

- Config:      hard_delete
- Env Var:     RCLONE_MEGA_HARD_DELETE
- Type:        bool
- Default:     false

#### --mega-use-https

使用 HTTPS 进行传输。

MEGA 默认使用明文 HTTP 连接。
部分 ISP 会对 HTTP 连接进行限速，这会导致传输变得非常慢。
启用此选项会强制 MEGA 对所有传输使用 HTTPS。
由于所有数据本来就已经加密，因此通常不需要使用 HTTPS。
启用 HTTPS 会增加 CPU 使用率并带来额外的网络开销。

属性：

- Config:      use_https
- Env Var:     RCLONE_MEGA_USE_HTTPS
- Type:        bool
- Default:     false

#### --mega-encoding

后端的编码方式。

更多信息参见 [概述中的编码章节](/overview/#encoding)。

属性：

- Config:      encoding
- Env Var:     RCLONE_MEGA_ENCODING
- Type:        Encoding
- Default:     Slash,InvalidUtf8,Dot

#### --mega-description

远端的描述信息。

属性：

- Config:      description
- Env Var:     RCLONE_MEGA_DESCRIPTION
- Type:        string
- Required:    false

### 进程 `killed`

在大文件较多的账户或其他情况下，执行 list/sync 命令时内存占用可能显著增加。
当运行在云服务提供商（如 AWS 的 EC2）上时，请检查实例类型是否具备
足够的内存/CPU 来执行这些命令。可借助资源监控工具在命令发送之后进行观察。
可参考 [这个 issue](https://forum.rclone.org/t/rclone-with-mega-appears-to-work-only-in-some-accounts/40233/4)。

## 限制

本后端使用 [go-mega go 库](https://github.com/t3rm1n4l/go-mega)，
它是一个实现 Mega API 的开源 Go 库。除了 [mega C++ SDK](https://github.com/meganz/sdk)
源码之外，似乎没有任何关于 mega 协议的文档，因此这个库中可能仍存在不少错误。

Mega 允许存在重复文件，这可能会使 rclone 产生混淆。
