# Track 管理实现手册

此文件包含技能引用的详细模式、清单和代码示例。

# Track 管理

创建、管理和完成 Conductor Track 的指南——通过规格说明、计划和实现阶段来组织功能、缺陷和重构的逻辑工作单元。

## 适用场景

- 创建新的功能、缺陷或重构 Track
- 编写或审查 spec.md 文件
- 创建或更新 plan.md 文件
- 管理 Track 从创建到完成的生命周期
- 理解 Track 状态标记和约定
- 操作 tracks.md 注册表
- 解读或更新 Track 元数据

## Track 概念

Track 是封装完整工作单元的逻辑工作单元。每个 Track 包含：

- 唯一标识符
- 定义需求的规格说明
- 将工作分解为任务的分阶段计划
- 跟踪状态和进度的元数据

Track 为工作提供语义化组织，支持：

- 清晰的范围边界
- 进度跟踪
- Git 感知操作（按 Track 回滚）
- 团队协调

## Track 类型

### feature

新功能或能力。适用于：

- 新的用户面向功能
- 新的 API 端点
- 新的集成
- 重大增强

### bug

缺陷修复。适用于：

- 错误行为
- 错误条件
- 性能退化
- 安全漏洞

### chore

维护和清理。适用于：

- 依赖更新
- 配置变更
- 文档更新
- 清理任务

### refactor

代码改进但不改变行为。适用于：

- 代码重构
- 模式采用
- 技术债务减少
- 性能优化（相同行为，更好性能）

## Track ID 格式

Track ID 遵循模式：`{shortname}_{YYYYMMDD}`

- **shortname**：2-4 个词的 kebab-case 描述（例如 `user-auth`、`api-rate-limit`）
- **YYYYMMDD**：ISO 格式的创建日期

示例：

- `user-auth_20250115`
- `fix-login-error_20250115`
- `upgrade-deps_20250115`
- `refactor-api-client_20250115`

## Track 生命周期

### 1. 创建（newTrack）

**定义需求**

1. 通过交互式问答收集需求
2. 确定验收标准
3. 确定范围边界
4. 识别依赖项

**生成规格说明**

1. 创建包含结构化需求的 `spec.md`
2. 记录功能和非功能需求
3. 定义验收标准
4. 列出依赖项和约束

**生成计划**

1. 创建包含分阶段任务分解的 `plan.md`
2. 将任务组织为逻辑阶段
3. 在阶段后添加验证任务
4. 估算工作量和复杂度

**注册 Track**

1. 向 `tracks.md` 注册表添加条目
2. 创建 Track 目录结构
3. 生成 `metadata.json`
4. 创建 Track `index.md`

### 2. 实现

**执行任务**

1. 从计划中选择下一个待处理任务
2. 将任务标记为进行中
3. 按工作流实现（TDD）
4. 标记任务完成并附上提交 SHA

**更新状态**

1. 更新 plan.md 中的任务标记
2. 记录提交 SHA 以便追溯
3. 更新阶段进度
4. 更新 tracks.md 中的 Track 状态

**验证进度**

1. 完成验证任务
2. 等待检查点审批
3. 记录检查点提交

### 3. 完成

**同步文档**

1. 如有新功能则更新 product.md
2. 如有依赖变更则更新 tech-stack.md
3. 验证所有验收标准已满足

**归档或删除**

1. 在 tracks.md 中将 Track 标记为已完成
2. 记录完成日期
3. 归档或保留 Track 目录

## 规格说明（spec.md）结构

```markdown
# {Track 标题}

## 概述

简要描述此 Track 要完成什么以及为什么。

## 功能需求

### FR-1：{需求名称}

功能需求的描述。

- 验收：如何验证此需求已满足

### FR-2：{需求名称}

...

## 非功能需求

### NFR-1：{需求名称}

非功能需求的描述（性能、安全等）

- 目标：具体的可测量目标
- 验证：如何测试

## 验收标准

- [ ] 标准 1：具体的、可测试的条件
- [ ] 标准 2：具体的、可测试的条件
- [ ] 标准 3：具体的、可测试的条件

## 范围

### 范围内

- 明确包含的项目
- 要实现的功能
- 要修改的组件

### 范围外

- 明确排除的项目
- 未来考虑
- 相关但独立的工作

## 依赖项

### 内部

- 此 Track 依赖的其他 Track 或组件
- 所需的上下文工件

### 外部

- 第三方服务或 API
- 外部依赖项

## 风险和缓解措施

| 风险             | 影响          | 缓解措施          |
| ---------------- | --------------- | ------------------- |
| 风险描述 | 高/中/低 | 缓解策略 |

## 待解决问题

- [ ] 需要解决的问题
- [x] 已解决的问题 - 答案
```

## 状态标记约定

在 plan.md 中使用一致的标记：

| 标记 | 含义     | 用途                       |
| ------ | ----------- | --------------------------- |
| `[ ]`  | 待处理     | 任务未开始            |
| `[~]`  | 进行中 | 当前正在处理      |
| `[x]`  | 完成    | 任务已完成（附 SHA） |
| `[-]`  | 跳过     | 有意不做      |
| `[!]`  | 阻塞     | 等待依赖       |

示例：

```markdown
- [x] **任务 1.1**：设置数据库 schema `abc1234`
- [~] **任务 1.2**：实现用户模型
- [ ] **任务 1.3**：添加验证逻辑
- [!] **任务 1.4**：集成认证服务（阻塞：等待 API 密钥）
- [-] **任务 1.5**：遗留迁移（跳过：不需要）
```

## Track 操作

### 创建 Track

1. 运行 `/conductor:new-track`
2. 回答交互式问题
3. 审查生成的 spec.md
4. 审查生成的 plan.md
5. 确认 Track 创建

### 开始实现

1. 读取 spec.md 和 plan.md
2. 验证上下文工件是最新的
3. 将第一个任务标记为 `[~]`
4. 开始 TDD 工作流

### 完成阶段

1. 确保所有阶段任务为 `[x]`
2. 完成验证任务
3. 等待检查点审批
4. 记录检查点 SHA
5. 进入下一阶段

### 完成 Track

1. 验证所有阶段已完成
2. 验证所有验收标准已满足
3. 如需要则更新 product.md
4. 在 tracks.md 中将 Track 标记为已完成
5. 更新 metadata.json

### 回滚 Track

1. 运行 `/conductor:revert`
2. 选择要回滚的 Track
3. 选择粒度（Track/阶段/任务）
4. 确认回滚操作
5. 更新状态标记

## 处理 Track 依赖

### 识别依赖

在 Track 创建期间，识别：

- **硬依赖**：必须在此 Track 开始前完成
- **软依赖**：可以并行进行但可能影响集成
- **外部依赖**：第三方服务、API 或团队决策

### 记录依赖

在 spec.md 中，列出依赖并包含：

- 依赖类型（硬/软/外部）
- 当前状态（可用/待处理/阻塞）
- 解决路径（需要发生什么）

### 管理阻塞的 Track

当 Track 被阻塞时：

1. 用 `[!]` 和原因标记阻塞的任务
2. 更新 tracks.md 状态
3. 在 metadata.json 中记录阻塞项
4. 如需要则考虑创建依赖 Track

## Track 规模指南

### 适当规模的 Track

目标 Track 应：

- 在 1-5 天工作内完成
- 有 2-4 个阶段
- 总共包含 8-20 个任务
- 交付一个连贯的、可测试的单元

### 过大

Track 过大的迹象：

- 超过 5 个阶段
- 超过 25 个任务
- 多个不相关的功能
- 预计耗时 > 1 周

解决方案：拆分为多个具有清晰边界的 Track。

### 过小

Track 过小的迹象：

- 单个阶段只有 1-2 个任务
- 不需要有意义的验证
- 可以是另一个 Track 的子任务
- 工作量少于几小时

解决方案：与相关工作合并或作为现有 Track 的一部分处理。

## 规格说明质量清单

在最终确定 spec.md 之前，验证：

### 需求质量

- [ ] 每个需求都有清晰的验收标准
- [ ] 需求是可测试的
- [ ] 需求是独立的（可以单独验证）
- [ ] 没有模糊语言（"应该很快" → "响应 < 200ms"）

### 范围清晰度

- [ ] 范围内项目是具体的
- [ ] 范围外项目防止范围蔓延
- [ ] 边界对实现者是清晰的

### 依赖已识别

- [ ] 所有内部依赖已列出
- [ ] 外部依赖有负责人/联系人
- [ ] 依赖状态是最新的

### 风险已处理

- [ ] 主要风险已识别
- [ ] 影响评估是现实的
- [ ] 缓解措施是可操作的

## 计划质量清单

在开始实现之前，验证 plan.md：

### 任务质量

- [ ] 任务是原子的（一个逻辑操作）
- [ ] 任务是独立可验证的
- [ ] 任务描述是清晰的
- [ ] 子任务提供有用的细节

### 阶段组织

- [ ] 阶段将相关任务分组
- [ ] 每个阶段交付可测试的内容
- [ ] 每个阶段后有验证任务
- [ ] 阶段在逻辑上相互构建

### 完整性

- [ ] 所有规格需求都有对应的任务
- [ ] 包含文档任务
- [ ] 包含测试任务
- [ ] 包含集成任务

## 常见 Track 模式

### 功能 Track 模式

```
阶段 1：基础
- 数据模型
- 数据库迁移
- 基本 API 结构

阶段 2：核心逻辑
- 业务逻辑实现
- 输入验证
- 错误处理

阶段 3：集成
- UI 集成
- API 文档
- 端到端测试
```

### 缺陷修复 Track 模式

```
阶段 1：复现
- 编写捕获缺陷的失败测试
- 记录复现步骤

阶段 2：修复
- 实现修复
- 验证测试通过
- 检查回归

阶段 3：验证
- 手动验证
- 如需要则更新文档
```

### 重构 Track 模式

```
阶段 1：准备
- 添加特性测试
- 记录当前行为

阶段 2：重构
- 增量应用变更
- 全程保持测试通过

阶段 3：清理
- 删除死代码
- 更新文档
```

## 最佳实践

1. **一个 Track，一个关注点**：保持 Track 专注于单个逻辑变更
2. **小阶段**：将工作分解为最多 3-5 个任务的阶段
3. **阶段后验证**：始终包含验证任务
4. **立即更新标记**：工作时标记任务状态
5. **记录 SHA**：始终为已完成任务记录提交 SHA
6. **计划前审查规格**：确保规格完整后再创建计划
7. **链接依赖**：明确注明 Track 依赖
8. **归档而非删除**：保留已完成 Track 以供参考
9. **适当规模**：保持 Track 在 1-5 天工作量之间
10. **清晰的验收标准**：每个需求必须是可测试的
