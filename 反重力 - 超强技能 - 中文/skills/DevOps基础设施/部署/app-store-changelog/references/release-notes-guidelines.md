# App Store 发布说明指南

## 目标
- 生成面向用户的发布说明，描述自上次标签以来的可见变更。
- 包含所有影响用户的变更；省略纯内部或仅重构的工作。
- 保持语言通俗、简短、以收益为导向。

## 输出格式
- 大多数版本优先 5 到 10 条要点。
- 如需按主题分组：新增、改进、修复。
- 每条要点应为一句话，以动词开头。
- 避免内部代号、工单 ID 或文件路径。

## 过滤规则
- 包含：新功能、UI 变更、行为变更、用户会注意到的 bug 修复、有可见影响的性能改进。
- 排除：重构、依赖升级、CI 变更、开发者工具、内部日志、分析变更（除非影响用户隐私或行为）。
- 如果变更含糊不清，请求澄清，或仅当变更对用户可见时才描述为小改进。

## 语言指导
- 将技术术语转化为面向用户的描述。
- 避免使用"API"、"refactor"、"nil"、"crash log"或"dependency"等词汇。
- 优先使用"改进"、"新增"、"修复"、"更新"或动作动词如"搜索"、"上传"、"同步"。
- 保持时态一致：使用"新增了"、"改进了"、"修复了"。

## 示例
- "新增了从个人资料菜单切换账户的功能。"
- "改进了慢速连接下的时间线加载速度。"
- "修复了媒体附件无法全屏打开的问题。"

## QA 检查清单
- 每条要点关联到范围内的真实变更。
- 没有描述同一变更的重复要点。
- 没有内部术语或文件路径。
- 最终列表符合目标商店的 App Store 文本限制（如已提供）。
