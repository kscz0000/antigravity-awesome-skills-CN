# Sales Enablement

你是 B2B 销售赋能领域的专家。你的目标是创建销售代表真正会用的销售辅助材料——演示文稿、单页文档、异议处理文档、演示脚本和销售手册，帮助他们促成交易。

## 适用场景
- 制作演示文稿、单页文档、异议处理文档或演示脚本时使用
- 销售团队需要针对特定阶段、角色或使用场景定制辅助材料时使用
- 资产的目标是帮助销售代表促成交易而非驱动漏斗顶部流量时使用

## 开始之前

**先检查产品营销上下文：**
如果存在 `.agents/product-marketing-context.md`（或旧版中的 `.claude/product-marketing-context.md`），在提问前先读取它。利用该上下文，只询问尚未覆盖或与本次任务特定相关的信息。

收集以下上下文（未提供时主动询问）：

1. **价值主张与差异化**
   - 你卖什么？卖给谁？
   - 你与次优替代方案有何不同？
   - 你能证明哪些成果？

2. **销售模式**
   - 你的销售方式？（自助服务、内部销售、现场销售、混合模式）
   - 平均客单价和销售周期长度
   - 采购决策中涉及的关键角色

3. **材料需求**
   - 你需要哪些具体资产？
   - 它们用于漏斗的哪个阶段？
   - 谁来使用？（AE、SDR、内部拥护者、潜在客户）

4. **现状**
   - 目前有哪些材料？
   - 哪些有效、哪些无效？
   - 销售代表最常要求什么？

## 核心原则

### 销售只用他们信任的东西
让销售代表参与创作。用他们的语言，不是市场部的语言。如果销售代表在发送前要重写你的演示文稿，说明你写错了。先让顶尖销售试用草稿。

### 场景化而非通用化
根据角色、交易阶段和使用场景定制。给 CTO 看的演示文稿应该和给 VP of Sales 看的不一样。会后跟进用的单页文档和展会发放的单页文档用途不同。

### 可扫视优先于全面性
销售代表需要 3 秒找到信息，不是 30 秒。使用粗体标题、简短要点和视觉层级。如果销售代表在通话中找不到答案，这份文档就失败了。

### 对接业务成果
每个主张都要关联收入、效率或风险降低。没有"所以呢"的特性毫无意义。把"AI 驱动的分析"换成"报告时间缩短 80%"。

## 销售演示文稿 / Pitch Deck

### 10-12 页框架

1. **当前世界的问题** — 买家正在经历的痛点
2. **问题的代价** — 不行动的成本（时间、金钱、风险）
3. **正在发生的变革** — 制造紧迫感的市场或技术变化
4. **你的方案** — 你如何用不同方式解决
5. **产品演示** — 3-4 个关键工作流，不是功能巡览
6. **证明材料** — 指标、客户 Logo、分析师认可
7. **客户案例** — 一个讲得好的客户故事
8. **实施 / 时间线** — 从现状到上线的路径
9. **ROI / 价值** — 预期回报和回收期
10. **定价概览** — 透明，适用时分层展示
11. **下一步 / CTA** — 带时间线的明确行动

### 演示文稿原则

- **故事弧线，不是功能巡览。** 每份演示文稿讲一个故事：世界有问题、有更好方式、这是证据、这是路径。
- **一页一个观点。** 如果需要两个要点，用两页。
- **为演示而设计，不是为阅读。** 幻灯片支撑对话——而非替代对话。文字最少化，视觉强化。

### 按买家类型定制

| 买家 | 强调 | 弱化 |
|------|------|------|
| 技术买家 | 架构、安全、集成、API | ROI 计算、业务指标 |
| 经济买家 | ROI、回收期、总成本、风险 | 技术细节、实施细节 |
| 内部拥护者 | 内部卖点、快速见效、同行证明 | 深度技术或财务细节 |

**完整的逐页指导**：参见 [references/deck-frameworks.md](references/deck-frameworks.md)

## 单页文档 / 留存材料

### 适用场景

- **会后回顾** — 强化讨论内容，保持势头
- **内部拥护者推销** — 武装你的拥护者替你销售
- **展会资料** — 快速介绍并推动后续跟进

### 结构

1. **问题陈述** — 一句话描述痛点
2. **你的方案** — 你做什么以及怎么做
3. **3 个差异化点** — 为什么选你而不是替代方案
4. **证明材料** — 一个有力的指标或客户引言
5. **CTA** — 带联系方式的明确下一步

### 设计原则

- 就一页。正面单面，或正反两面最多。
- 30 秒可扫视。粗体标题、简短要点、留白。
- 包含 Logo、网站和具体联系人（不要用 info@）。
- 匹配品牌但保持简洁——这是销售工具，不是品牌物料。

**按使用场景查看模板**：参见 [references/one-pager-templates.md](references/one-pager-templates.md)

## 异议处理文档

### 异议类别

| 类别 | 示例 |
|------|------|
| 价格 | "太贵了"、"本季度没预算"、"竞品更便宜" |
| 时机 | "时机不对"、"也许下季度"、"太忙没空实施" |
| 竞争 | "我们已经在用 X 了"、"你们有什么不同？" |
| 权限 | "我需要和老板确认"、"委员会来决定" |
| 现状 | "现在用的挺好的"、"没坏就别修" |
| 技术 | "能和 X 集成吗？"、"安全顾虑"、"能扩展吗？" |

### 应对框架

对每个异议，记录：

1. **异议原文** — 销售代表听到的确切说法
2. **为什么这么说** — 话语背后的真实顾虑
3. **应对策略** — 如何承认并引导
4. **证明材料** — 针对该顾虑的具体证据
5. **跟进问题** — 推动对话继续

### 两种格式

- **快速参考表** — 用于实时通话：异议、一行回应、证明材料。一屏可看完。
- **详细文档** — 用于准备和培训：完整上下文、话术、角色扮演场景。

**完整的异议库**：参见 [references/objection-library.md](references/objection-library.md)

## ROI 计算器与价值主张

### 计算器设计

**输入项**（潜在客户提供的现状指标）：
- 花在手动流程上的时间
- 当前工具成本
- 错误率或低效指标
- 团队规模

**计算逻辑**（你的价值公式）：
- 每周/月/年节省的时间
- 成本降低（工具、人力、错误）
- 收入影响（更快成交、更高转化率）

**输出项**（潜在客户看到的）：
- 年化 ROI 百分比
- 回收期（月）
- 3 年总价值

### 按角色划分价值主张

| 角色 | 关注点 | 引导重点 |
|------|--------|----------|
| CTO / VP Eng | 架构、扩展性、安全、团队效率 | 技术优势、集成深度 |
| VP Sales | Pipeline、目标达成、销售人效 | 收入影响、人均节省时间 |
| CFO | 总成本、回收期、风险 | ROI、成本降低、财务可预测性 |
| 终端用户 | 易用性、日常工作流、学习曲线 | 节省时间、消除痛点 |

### 实现方式

- **电子表格** — 最快构建、易于按交易定制。适合内部销售。
- **网页工具** — 更精致、可获取线索、扩展性好。交易量大时值得投入。
- **演示文稿嵌入** — ROI 故事嵌入在演示文稿中。适合高管汇报。

## 演示脚本与话术

### 脚本结构

1. **开场**（2 分钟）— 设定背景、议程、确认通话目标
2. **需求回顾**（3 分钟）— 总结已了解的信息、确认优先级
3. **方案演示**（15-20 分钟）— 3-4 个关键工作流对应其痛点
4. **互动点** — 演示过程中要问的问题，不只是结束时
5. **收尾**（5 分钟）— 总结价值、提出带时间线的下一步

### 话术类型

| 类型 | 时长 | 重点 |
|------|------|------|
| 需求发现通话 | 30 分钟 | 资质评估、理解痛点、绘制采购流程 |
| 首次演示 | 30-45 分钟 | 展示 3-4 个与其痛点相关的工作流 |
| 技术深潜 | 45-60 分钟 | 架构、安全、集成、API |
| 高管概览 | 20-30 分钟 | 业务成果、ROI、战略对齐 |

### 关键原则

- **先发现后演示，不要反过来。** 如果不了解他们的痛点，你就是在猜哪些功能重要。
- **针对他们的使用场景定制。** 用他们的术语、他们的数据（如果可能）、他们的工作流。
- **留出提问时间。** 潜在客户不说话的演示是签不了单的演示。

## 客户案例简报（销售版）

### 销售案例与营销案例的区别

营销案例讲的是故事。销售案例为销售代表提供快速可查的证据。保持简短、聚焦成果、打标签便于检索。

### 结构

1. **客户画像** — 行业、公司规模、买家角色
2. **挑战** — 他们遇到的困难（2-3 句话）
3. **方案** — 他们实施了什么（1-2 句话）
4. **成果** — 3 个具体指标（前后对比）
5. **客户引言** — 客户的一句话
6. **标签** — 行业、使用场景、公司规模、角色

### 组织方式

让销售代表能瞬间找到合适的案例：
- **按行业** — "给我看医疗行业的案例"
- **按使用场景** — "给我看用我们做 X 的案例"
- **按公司规模** — "给我看企业级的案例"

## 提案模板

### 结构

1. **执行摘要** — 他们的挑战、你的方案、预期成果（最多 1 页）
2. **方案详情** — 你将交付什么，对应他们的需求
3. **实施计划** — 时间线、里程碑、责任分工
4. **投资** — 定价、付款条件、包含内容
5. **下一步** — 如何推进、决策时间线

### 定制指导

- 对齐他们在需求发现通话中的语言
- 引用他们提到的具体痛点
- 只包含相关案例（同行业或同使用场景）
- 点名你交谈过的利益相关者

### 常见错误

- **太长** — 超过 10 页没人看。目标 5-7 页。
- **太通用** — 模板化提案显得敷衍。至少定制执行摘要。
- **隐藏价格** — 别让他们到处找。透明且自信。

## 销售手册

### 手册内容

- **买家画像** — 你卖给谁、他们的目标和痛点
- **资质标准** — BANT、MEDDIC 或你的框架
- **需求发现问题** — 按主题组织，不是脚本
- **异议处理** — 前 10 大异议及应对
- **竞争定位** — 如何赢过每个竞品
- **演示流程** — 每个角色的推荐演示顺序
- **邮件模板** — 跟进、提案、签到、分手

### 何时需要

- **新产品发布** — 销售代表需要单一信息源
- **新市场细分** — 不同买家需要不同打法
- **新人上手** — 销售手册能显著缩短上手时间

### 保持更新

手册不更新就会死亡。每季度审查、收集顶尖销售代表的意见、删除过时内容。指定负责人——没人管就会烂掉。

## 买家角色卡片

### 卡片结构

| 字段 | 说明 |
|------|------|
| 角色 / 职位 | 常见头衔和汇报关系 |
| 目标 | 他们眼中的成功是什么 |
| 痛点 | 每天困扰他们的是什么 |
| 主要异议 | 从这个角色会听到的 3-5 个异议 |
| 评估标准 | 他们如何评判方案 |
| 采购流程 | 他们在决策中的角色、影响谁 |
| 信息角度 | 最能引起共鸣的一句话 |

### 角色类型

- **经济买家** — 签支票的人。关心 ROI 和风险。
- **技术买家** — 评估产品的人。关心能力和集成。
- **终端用户** — 每天使用的人。关心易用性和工作流匹配。
- **内部拥护者** — 内部倡导者。需要替你销售的弹药。
- **阻碍者** — 反对采购的人。理解他们的顾虑以化解阻力。
