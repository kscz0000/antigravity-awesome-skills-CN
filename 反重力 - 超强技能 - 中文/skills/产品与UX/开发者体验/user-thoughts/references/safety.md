# 安全与数据完整性

本文档定义 `user-thoughts` 的路径安全、输入校验和数据完整性规则。

## 路径安全

除非导入命令读取项目本地的 markdown 文件，否则所有运行时文件操作都必须限定在 `#ustht/` 内。

维度名用于构造路径，因此必须严格校验：

| 规则 | 原因 |
|---|---|
| 每段路径只允许 `[a-z0-9-]` | 防止 shell 与路径出现意外。 |
| 每段必须以 `[a-z0-9]` 开头和结尾 | 避免隐藏或格式错误的文件。 |
| 仅允许 `/` 作为维度子目录分隔符 | 支持 `ui/outline` 这种结构。 |
| 禁止 `..`、反斜杠、空格和绝对路径 | 防止路径穿越。 |
| 禁止保留名 | 避免与运行目录冲突。 |

保留名：`backlog`、`readme-ai`、`export`、`raw`、`ignored`、`define`、`general`。

## 内容安全

原始条目使用以下格式：

```text
- [HH:MM] original user text | suggested-dim:dimension
```

后缀是由智能体生成的元数据。用户文本可以包含 markdown，应按原样保留。解析时只识别最后一个 ` | suggested-dim:` 分隔符。

`<!-- processed -->` 仅作为原始文件首行才有意义。如果用户在某条想法中提到该字符串，应作为普通内容处理。

## define.ini 安全

允许的键和值：

| 键 | 允许的值 |
|---|---|
| `SKILL_STATUS` | `on` 或 `off` |
| `INSTANT_STATUS` | `on` 或 `off` |
| `LAST_SORTIN` | 空或 `yyyy-mm-dd HH:MM` |

值不得包含换行或 `=`。应整体重写文件，不要追加部分片段。

## Shell 安全

- 不得执行用户提供的 shell 命令。
- 不得使用 `eval` 或动态执行。
- 仅从经过校验的维度或固定模板路径构造文件路径。
- 初始化时，应安全地复制已知模板文件，不要用 shell 命令递归复制任意目录。

## 数据完整性

`sortin` 并非完全原子化。为降低部分写入的风险：

1. 先解析原始条目。
2. 写入维度文件。
3. 仅在写入成功后，将原始文件标记为已处理。
4. 最后更新 `LAST_SORTIN`。

已处理的原始文件会被保留以便追溯。维度文件应追加或标记为废弃，不要静默删除用户历史。

## 敏感数据

该技能保留原始措辞，不会自动脱敏机密或个人数据。用户应在敏感内容被捕获前使用忽略命令，团队也应通过常规的仓库与文件系统管理来保护 `.ustht/`。
