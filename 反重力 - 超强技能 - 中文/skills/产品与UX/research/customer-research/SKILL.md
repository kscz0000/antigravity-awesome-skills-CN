# 客户研究
## 适用场景

涉及开展、分析或综合客户研究时使用本技能。涉及"客户研究"、"ICP 研究"、"与客户沟通"、"分析转录文本"、"客户访谈"、"问卷分析"、"工单分析"、"客户之声"、"VOC"、"构建画像"等场景时触发。


你是一名资深的客户研究员。目标是帮助你挖掘客户的真实想法、感受、说法与痛点——让定位、产品、文案都根植于现实，而非凭空假设。

## 开始之前

**先检查产品营销上下文：**
若 `.agents/product-marketing.md` 存在（或 `.claude/product-marketing.md`，以及旧版本中的 `product-marketing-context.md` 文件名），先读取再提问。借助上下文跳过已回答过的问题。

## 两种研究模式

### 模式 1：分析已有素材
你手头已有原始研究素材（访谈转录、问卷、评论、工单）。任务是抽取信号。

### 模式 2：主动搜集
需要从在线渠道（Reddit、G2、论坛、社区、评测网站）获取情报。任务是要知道去哪里查、抽取什么。

大多数项目两种模式兼用。先明确属于哪种模式再继续。

---

## 模式 1：分析已有研究素材

### 素材类型

**客户访谈 / 销售通话转录**
- 抽取：痛点、触发事件、期望结果、所用语言、异议、考虑过的替代方案
- 留意：他们决定寻找解决方案的那一刻、之前尝试过什么、对他们而言成功是什么样

**问卷结果**
- 在下结论前按客户层级、用途或使用时长对回答做分群
- 标注：开放式答案与多选答案分别说了什么（两者常常冲突）
- 识别：包含最多有用信号的那 20% 回答

**客户支持对话**
- 挖掘：反复出现的投诉、困惑点、功能请求，以及"我希望它能……"式表述
- 分析前先对工单分类——不要把所有工单当作等量信号
- 把 Bug、误解、缺失功能、预期错配区分开

**赢/输访谈与流失客户记录**
- 赢单：是什么促成了决定？是什么差点让他们选了竞争对手？
- 输单与流失：是价格、功能、契合度、时机，还是其他原因？
- 按原因分群——不要把不同流失原因混在一起取均值

**NPS 回复**
- 被动者与贬损者比推荐者更能提供改进信号
- 把分数与原话配对看——一条带具体抱怨的 9 分，比一条没任何评论的 10 分更有价值

### 抽取框架

对每个素材，抽取：

1. **待办任务（Jobs to Be Done）**——客户想达成什么结果？
   - 功能性任务：任务本身
   - 情感性任务：他们想有什么感觉
   - 社交性任务：他们想被如何看待

2. **痛点**——当前情境中什么是令人挫败、失效或不充分的？
   - 优先考虑主动提及且带有情绪化语言的痛点

3. **触发事件**——什么变化让他们开始寻找解决方案？
   - 常见触发：团队扩张、新人入职、未达目标、尴尬事件、竞争对手的动作

4. **期望结果**——用他们自己的话来说，成功是什么样子？
   - 记录原话而非转述

5. **语言与词汇**——客户使用的原词原句
   - 这是文案的金矿。"我们被电子表格淹没了" > "手工流程效率低下"

6. **考虑过的替代方案**——他们还看过或试过什么？
   - 包括什么都不做、雇人或自建

### 综合步骤

完成单素材抽取后：

1. **按主题聚类**——跨素材把相似的痛点、结果、触发事件归组
2. **频次 + 强度打分**——主题出现频率有多高，情感强度有多大？
3. **按客户画像分群**——按公司规模、角色、用途或使用时长，模式是否不同？
4. **找出"金句"**——5-10 条最能代表每个主题的原话
5. **标注矛盾**——客户说的是一回事，做的却是另一回事的地方

### 研究质量护栏

每条洞察在呈现前都贴上置信度标签：

| 置信度 | 标准 |
|------------|----------|
| **高** | 主题在 3+ 个独立来源中出现；被主动提及；不同分群间一致 |
| **中** | 主题在 2 个来源中出现，或仅在提示后提及，或局限于单一分群 |
| **低** | 单一来源；可能是离群值；需要验证 |

**时效窗口**：更重加权过去 12 个月的来源。市场在变——3 年前的转录可能反映的是不同的产品和买家。

**样本偏差检查**：
- 在线评论者偏向重度用户和有强烈意见的人
- 支持工单偏向问题，而非价值
- Reddit 偏向技术人群与怀疑论者，而非主流买家
- 在对"所有客户"下结论时把这些因素算进去

**最小可行样本**：每个分群至少 5 个独立数据点，才能构建画像或得出文案结论。

## 模式 2：数字水源地研究

在线社区是客户不加滤镜发声的地方。目标是找到关于问题空间真实、未受审查的语言。

### 渠道选择

依据你的 ICP 类型选择来源——然后阅读 `references/source-guides.md` 获取详细的打法、搜索算子与平台抽取技巧。

| ICP 类型 | 主要来源 |
|----------|----------------|
| B2B SaaS / 技术买家 | Reddit（角色相关子版块）、G2/Capterra、Hacker News、LinkedIn、Indie Hackers、SparkToro |
| SMB / 创业者 | Reddit（r/entrepreneur、r/smallbusiness）、Indie Hackers、Product Hunt、Facebook Groups、SparkToro |
| 开发者 / DevOps | r/devops、r/programming、Hacker News、Stack Overflow、Discord 服务器 |
| B2C / 消费者 | 应用商店评论（1-3 星）、Reddit 兴趣/生活方式子版块、YouTube 评论、TikTok/Instagram 评论 |
| 企业级 | LinkedIn、行业分析师报告、G2 Enterprise 筛选、招聘启事、SparkToro |

**快速决策指南：**
- 有产品类别？→ 先看 G2/Capterra 评论（自己的 + 竞争对手的）
- 需要知道目标受众在哪儿活跃？→ SparkToro（揭示播客、YouTube、subreddit、网站、社交账号）
- 需要原生态语言？→ Reddit 与 YouTube 评论
- 需要触发事件？→ LinkedIn 帖子、招聘启事、Hacker News "Ask HN" 串
- 需要竞争情报？→ 竞品 G2 4 星评论；Product Hunt 讨论；SparkToro 竞品受众分析

### 每个来源的抽取内容

对每条找到的内容：

| 字段 | 抽取内容 |
|-------|----------------|
| 来源 | 平台、帖子 URL、日期 |
| 原话引用 | 一字不改——不要转述 |
| 上下文 | 这条评论由什么引发？ |
| 情感倾向 | 正面 / 负面 / 中立 / 挫败 |
| 主题标签 | 痛点 / 触发 / 结果 / 替代 / 语言 |
| 客户画像信号 | 角色、公司规模、行业线索（来自帖子本身） |

### 研究综合模板

从多个来源搜集完毕后，综合为：

```
## 核心主题（按频次 × 强度排序）

### 主题 1：[名称]
**摘要**：[1-2 句]
**频次**：在 X/Y 个来源中出现
**强度**：高 / 中 / 低（依据所用情感化语言）
**代表性引语**：
- "[原话]"——[来源, 日期]
- "[原话]"——[来源, 日期]
**影响**：这对文案 / 产品 / 定位意味着什么

### 主题 2：…
```

## 画像生成

画像应该基于研究而非凭空捏造。在同一分群积累到至少 5-10 个数据点（访谈、评论、社区帖子）之前，不要创建画像。

### 画像结构

```
## [画像名] — [角色/职位]

**基本资料**
- 职位范围：[如"市场经理至市场副总裁"]
- 公司规模：[如"50–500 人，A–C 轮 SaaS"]
- 行业：[若聚焦]
- 汇报对象：[谁]
- 团队管理规模：[若相关]

**首要待办任务**
[一句话：他们在岗位上想达成什么结果？]

**触发事件**
什么让他们开始寻找像你这样的解决方案？
- [触发 1]
- [触发 2]

**核心痛点**
1. [痛点——若可能用他们原话]
2. [痛点]
3. [痛点]

**期望结果**
- [成功在他们眼里是什么样]
- [他们如何衡量]
- [这让他们在老板/团队面前看起来如何]

**异议与顾虑**
- [什么让他们在购买或迁移时犹豫]

**他们考虑的替代方案**
- [竞品、自建、什么都不做、雇人]

**关键用语**
研究中提取的、客户实际使用的词句：
- "[短语]"
- "[短语]"

**如何触达他们**
- 渠道：[他们活跃的地方]
- 消费的内容：[形式、话题]
- 信任的影响者/社区：[若已知，给出具体名字]
```

### 画像反模式

- **不要起可爱的名字**（"营销 Mary"），除非你的团队觉得有用——常常是干扰
- **不要跨分群取均值**——代表所有人的画像等于谁也没代表
- **不要凭空编造细节**——某项若没有数据，宁可留空也别填
- **每季度回顾一次**——画像会随市场和产品演化而失效

## 交付物格式

根据用户需要，可提供：

1. **研究综合报告**——主题、引语、模式与影响
2. **VOC 引语库**——按主题归类的原话，用于文案
3. **画像文档**——基于研究构建的 1-3 个画像
4. **待办任务地图**——按分群列出的功能性、情感性、社交性任务
5. **竞争情报摘要**——客户在竞品和你之间如何说
6. **研究空白分析**——还不知道什么以及如何找到

在生成输出前先问用户需要哪一项（或几项）交付物。

---

## 动手前要问的问题

若上下文不清晰：

1. **目标是什么？**改进文案？构建画像？找产品缺口？理解流失？
2. **已经有什么？**（转录、问卷、工单、G2 评论、什么都没有）
3. **目标分群是谁？**（全部客户、特定层级、流失用户、未成交潜客）
4. **产品是什么？**（若不在产品营销上下文文件中）
5. **要交付什么？**（综合报告、画像、引语库、竞争情报）

不要一次问完五个——先问 #1 和 #2，再按需追问。

- 仅在任务与上游来源和本地项目上下文明确匹配时使用本技能。
- 在应用变更前，验证命令、生成代码、依赖、凭据与外部服务行为。
- 不要把示例当作环境特定测试、安全审查或用户对破坏性/高成本操作的审批的替代品。
