# 深化

在已知依赖的前提下，如何安全地深化一组浅层模块。假定你已掌握 [SKILL.md](SKILL.md) 中的术语 —— **module（模块）**、**interface（接口）**、**seam（接缝）**、**adapter（适配器）**。

## 依赖类别

评估一个候选模块是否值得深化时，先对其依赖进行分类。类别决定了深化后的模块如何跨接缝进行测试。

### 1. 进程内（In-process）

纯计算、内存状态、无 I/O。任何时候都可以深化 —— 直接合并相关模块，并通过新接口直接测试。无需适配器。

### 2. 本地可替代（Local-substitutable）

依赖存在本地测试替代物（PGLite 替代 Postgres、内存文件系统等）。只要存在替代物就可以深化。深化后的模块在测试套件中配合该替代物运行进行测试。接缝是内部的；在模块的外部接口处不需要端口。

### 3. 远程但自有的（Ports & Adapters）

跨网络边界运行的你自己的服务（微服务、内部 API）。在接缝处定义一个 **port（端口，即接口）**。深度模块承载核心逻辑；传输层作为 **adapter（适配器）** 注入。测试使用内存适配器，生产环境使用 HTTP/gRPC/队列适配器。

推荐做法形态：*"在接缝处定义一个 port，为生产环境实现一个 HTTP 适配器，为测试实现一个内存适配器，这样即使跨越网络部署，逻辑仍集中在一个深度模块中。"*

### 4. 真正的外部依赖（Mock）

你无法控制的第三方服务（Stripe、Twilio 等）。深化后的模块将外部依赖作为注入的端口引入；测试时提供一个 mock 适配器。

## 接缝纪律

- **一个适配器只是假想接缝。两个适配器才是真正的接缝。** 除非至少能合理地支撑两个适配器（通常是生产 + 测试），否则不要引入 port。只有一个适配器的接缝只是无谓的间接层。
- **内部接缝与外部接缝。** 深度模块可以同时拥有内部接缝（私有于其实现，仅由其自身测试使用）以及位于其接口处的外部接缝。不要仅因为测试使用了内部接缝，就把它暴露到外部接口上。

## 测试策略：替换，而非叠加

- 浅层模块上的旧单元测试，一旦深度模块接口处的测试建立起来，就变成了废物 —— 删除它们。
- 在深化后模块的接口处编写新测试。**接口就是测试面（test surface）**。
- 测试应通过接口断言可观察的结果，而不是内部状态。
- 测试应能经受住内部重构而不失效 —— 它们描述行为，而非实现。如果一个测试在实现变更时不得不跟着改动，那它已经越过了接口去窥探实现。
