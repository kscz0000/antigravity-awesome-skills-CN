# 设计两次（Design It Twice）

当用户希望为已选定的深化候选探索多种替代接口时，使用此并行子智能体模式。基于 Ousterhout 的 "Design It Twice" 原则 —— 你的第一个想法几乎从来都不是最优解。

使用 [SKILL.md](SKILL.md) 中的术语 —— **module（模块）**、**interface（接口）**、**seam（接缝）**、**adapter（适配器）**、**leverage（杠杆）**。

## 流程

### 1. 框定问题空间

在派生子智能体之前，先为所选候选写一份面向用户的问题空间说明：

- 任何新接口都必须满足的限制条件
- 它将依赖哪些依赖，以及这些依赖分别属于哪一类别（参见 [DEEPENING.md](DEEPENING.md)）
- 一段粗略的示例性代码草图，用于让限制条件变得具体 —— 不是提案，只是把抽象的限制落地的方式

把这份说明展示给用户，然后立即进入第 2 步。子智能体并行工作时，用户可以边读边思考。

### 2. 派生子智能体

使用 Agent 工具并行派生出 3 个或更多子智能体。每个都必须为深化后的模块提出一个**截然不同的**接口。

为每个子智能体准备一份独立的技术简报（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别、接缝背后是什么），并赋予不同的设计约束。简报独立于第 1 步中面向用户的问题空间说明。为每个智能体分配一个不同的设计约束：

- 智能体 1："最小化接口 —— 入口点最多 1–3 个。让每个入口点的杠杆最大化。"
- 智能体 2："最大化灵活性 —— 支持尽可能多的用例和扩展点。"
- 智能体 3："针对最常见的调用者进行优化 —— 让默认场景足够简单。"
- 智能体 4（如适用）："围绕 ports & adapters，为跨接缝的依赖而设计。"

在简报中同时包含 [SKILL.md](SKILL.md) 的术语和 CONTEXT.md 的领域术语，确保每个子智能体都使用与架构语言和项目领域语言一致的命名。

每个子智能体输出：

1. 接口（类型、方法、参数 —— 以及不变式、顺序、错误模式）
2. 用法示例，展示调用者如何使用
3. 接缝背后隐藏的实现细节
4. 依赖策略与适配器（参见 [DEEPENING.md](DEEPENING.md)）
5. 取舍 —— 在哪里杠杆高，在哪里杠杆薄

### 3. 展示与对比

依次展示每个设计，让用户逐一消化，然后以散文形式对比。从以下几个维度进行对照：**深度（depth，接口处的杠杆）**、**局部性（locality，变更集中在哪里）**、**接缝位置（seam placement）**。

对比之后，给出你自己的推荐：你认为哪个设计最强，以及原因。如果不同设计中的某些元素可以很好地组合，就提出一个混合方案。要有自己的判断 —— 用户要的是一份立场鲜明的解读，而不是一份菜单。
