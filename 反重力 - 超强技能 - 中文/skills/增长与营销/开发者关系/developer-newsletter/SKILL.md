# 开发者 Newsletter
## 适用场景

当用户需要为开发者受众创建、撰写或改进 newsletter 时使用本技能。触发词包括"newsletter"、"邮件营销"、"开发者邮件"、"每周精选"、"dev newsletter"、"邮件订阅"、"newsletter 增长"、"邮件列表"。


本技能帮助你构建和撰写开发者真正愿意打开、阅读并期待收到的 newsletter。涵盖内容策略、写作、增长与送达率。

---

## 开始之前

**先加载你的受众上下文。** 读取 `.agents/developer-audience-context.md` 来了解：

- 你的读者是谁（角色、资历、技术栈）
- 什么内容能打动他们（痛点、兴趣）
- 他们还在哪些渠道消费内容（避免重复造轮子）
- 语气风格（口语化程度、技术浓度）

如果上下文文件不存在，先运行 `developer-audience-context` 技能。

## Newsletter 策略

### 明确 Newsletter 类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **产品更新** | 更新日志、新功能、技巧 | Vercel 的更新邮件 |
| **精选链接** | 全网最佳内容 | TLDR、Bytes |
| **原创内容** | 自己的文章、教程 | Cassidy Williams |
| **社区摘要** | 你所在社区的动态 | 开发者社区周报 |
| **教学系列** | 围绕一个主题持续教学 | 课程式 newsletter |

**最佳实践**：选定一个主要类型。可以混搭其他类型，但要有清晰的定位。

### 频率矩阵

| 频率 | 最适合 | 风险 |
|------|--------|------|
| **每日** | 精选链接、新闻 | 读者疲劳，难以持续 |
| **每周** | 大多数 newsletter | 大多数情况的甜蜜点 |
| **双周** | 重原创内容 | 容易失去势头 |
| **每月** | 产品更新、摘要 | 读者容易忘记你 |

**开发者偏好**：每周是最佳频率。开发者都很忙，会严格管理收件箱。

## 内容组合框架

### 70-20-10 法则

| 比例 | 内容类型 | 目的 |
|------|----------|------|
| **70%** | 价值内容 | 教学、传递信息、帮忙解决问题 |
| **20%** | 产品内容 | 更新、新功能、使用教程 |
| **10%** | 推广内容 | CTA、诉求、销售 |

### 内容类别

建立一个轮换清单：

| 类别 | 示例 |
|------|------|
| **教程** | "如何实现 X" |
| **新闻解读** | "Y 公司的公告对你意味着什么" |
| **工具/库盘点** | "处理 Z 的 5 个库" |
| **代码片段** | "小技巧：更好的错误处理" |
| **社区精选** | "本周 Discord 精华" |
| **行业观点** | "为什么我觉得 X 被过度吹捧" |
| **幕后故事** | "我们是如何构建功能 Y 的" |
| **Q&A** | "你问，我们答" |

## 撰写开发者邮件

### 标题框架

对开发者有效的方法：

| 模式 | 示例 | 为什么有效 |
|------|------|-----------|
| **具体收益** | "把构建时间砍掉 40%" | 明确价值 |
| **技术好奇心** | "那个没人用的 JavaScript 特性" | 触发好奇心 |
| **直接公告** | "v2.0 已发布：async/await 支持" | 清晰、有新闻价值 |
| **数字 + 主题** | "资深开发者常用的 7 个 TypeScript 技巧" | 易扫读、具体 |
| **提问** | "你还在用回调吗？" | 打破模式 |
| **突发新闻** | "React 19 来了：你需要知道的事" | 时效性强、有紧迫感 |

无效的方法：

| 避免 | 原因 |
|------|------|
| 全大写 | 像垃圾邮件信号 |
| "快速提问" | 显套路 |
| 滥用 emoji | 像营销邮件 |
| "你不会相信..." | 标题党疲劳 |
| 没有标题 | 直接毙掉 |

### 预览文本

紧跟在标题后面的预览文字。一定要用。

| 标题 | 预览文本 |
|------|----------|
| "v2.0 已发布" | "附：需要注意的几个破坏性变更" |
| "本周 Node.js 速览" | "fetch() 风波、npm 安全事件，外加一个有趣的 CLI" |

### 邮件结构

```
[简短个人开场 - 1-2 句话]

[主要内容分节，标题清晰]

[必要时插入代码片段]

[快捷链接区]

[带有个性色彩的署名收尾]
```

### 邮件中的代码

代码在邮件里的呈现很棘手。几种方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **行内代码**（反引号） | 通用 | 无高亮 |
| **纯文本块** | 稳定 | 不美观 |
| **代码截图** | 漂亮 | 无法复制，无障碍问题 |
| **"在浏览器中查看"链接** | 完整格式 | 多一步操作 |
| **带样式的 HTML 表格** | 格式尚可 | 复杂、容易出问题 |

**建议**：代码尽量短。小片段用行内代码，完整示例链到外面。

```html
<pre style="background-color: #1e1e1e; color: #d4d4d4; padding: 16px; border-radius: 4px; font-family: 'Fira Code', monospace; font-size: 14px; overflow-x: auto;">
const result = await fetch('/api/data');
</pre>
```

## 标题测试

### A/B 测试框架

每次只测一个变量：

| 变量 | 版本 A | 版本 B |
|------|--------|--------|
| **长度** | "TypeScript 5.0 新特性" | "7 个 TypeScript 5.0 新特性，彻底改变你的写法" |
| **具体程度** | "新功能" | "异步导入、装饰器，以及另外 5 项" |
| **格式** | 陈述句 | 疑问句 |
| **个性化** | 通用 | "[姓名]，你的每周精选" |
| **Emoji** | 不用 | 一个相关 emoji |

### 标题自查清单

发送前过一遍：

- [ ] 50 字符以内（手机预览能完整显示）
- [ ] 没有垃圾邮件触发词（免费、立即行动、限时）
- [ ] 具体，不模糊
- [ ] 与正文一致（不要挂羊头卖狗肉）
- [ ] 你自己会点开吗？

## 增长策略

### 自然增长

| 策略 | 落地方式 |
|------|----------|
| **博客页脚 CTA** | "把这类文章送到你收件箱"，配 inline 订阅表单 |
| **内容升级** | "下载完整清单"换邮箱 |
| **退出意图弹窗** | 离开时弹出（适度使用） |
| **Twitter/社交提及** | "我每周在 newsletter 里写这些" |
| **文档页 CTA** | 在文档页脚放订阅框 |
| **开源 README** | 在项目 README 中放 newsletter 链接 |
| **技术大会演讲** | "订阅即可领取演讲 PPT + 额外内容" |

### 推荐奖励

| 奖励档位 | 奖励示例 |
|----------|----------|
| **1 个推荐** | 在 newsletter 中点名致谢 |
| **5 个推荐** | 独家内容 / 抢先体验 |
| **10 个推荐** | 周边（贴纸、T 恤） |
| **25 个推荐** | 1:1 通话 / 高级会员 |

### 交叉推广

与互补的 newsletter 互换曝光：

| 你的 Newsletter | 适合的合作对象 |
|-----------------|----------------|
| 专注 React | TypeScript、Node.js、前端 newsletter |
| DevOps | 云、Kubernetes、基础设施 newsletter |
| AI/ML | Python、数据科学 newsletter |

交换的是提及，不是完整广告。

## 避开垃圾邮件过滤器

### 技术配置

| 项目 | 操作 |
|------|------|
| **SPF** | 添加 DNS 记录授权发送方 |
| **DKIM** | 对邮件进行加密签名 |
| **DMARC** | 设定身份验证失败的处理策略 |
| **自有域名** | 从 `news@yourcompany.com` 发，不要用个人邮箱 |
| **预热** | 先小批量发送，逐步增大 |

### 内容卫生

| 应该 | 不应 |
|------|------|
| 附带纯文本版本 | 只有 HTML |
| 合理的图文比例 | 全是图片没有文字 |
| 醒目的退订链接 | 退订链接隐藏或难找 |
| 发送节奏稳定 | 毫无规律的随时发送 |
| 干净的列表 | 含大量退信、不活跃、购买的地址 |

### 危险词列表

在标题和正文中避免：

| 类别 | 应避免的词 |
|------|-----------|
| **紧迫感** | 立即行动、限时、即将过期 |
| **免费相关** | 免费、不花钱、无需承诺 |
| **金钱相关** | $$、现金、赚取、投资 |
| **夸张** | 惊人、不可思议、前所未有 |
| **经典垃圾词** | 点此、中奖、恭喜 |

## 指标与基准

### 关键指标

| 指标 | 开发者 Newsletter 基准 |
|------|------------------------|
| **打开率** | 30-50%（高于 B2C） |
| **点击率** | 5-15% |
| **退订率** | 每次发送 <0.5% |
| **垃圾投诉率** | <0.1% |
| **列表增长率** | 每月 5-10% |

### 跟踪哪些指标

| 指标 | 能告诉你什么 |
|------|--------------|
| **按标题的打开率** | 标题效果 |
| **按链接的点击率** | 内容共鸣度 |
| **回复率** | 互动深度 |
| **发送后退订率** | 内容匹配度 |
| **转发率** | 可分享性 |
| **增长来源** | 最佳获客渠道 |

---

```markdown
## [主要章节 1]

[2-3 段有价值的内容]

\`\`\`javascript

## [主要章节 2]

[内容]

## 快捷链接

- [链接 1]：一句话描述

- [链接 3]：一句话描述

[从 Discord/Twitter/GitHub 挑选一条内容]

---

```

## 局限性

- 仅当任务明确匹配其上游来源和本地项目上下文时使用本技能。
- 在执行变更前，校验命令、生成的代码、依赖、凭据以及外部服务的行为。
- 不要把示例当作环境专属测试、安全审查或用户对破坏性/高成本操作的批准的替代品。
