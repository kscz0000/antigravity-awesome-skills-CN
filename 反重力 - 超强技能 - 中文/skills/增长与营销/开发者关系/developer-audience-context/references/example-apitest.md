## 问题与痛点

### 功能性问题
- 现有测试设置既慢又不稳定
- 测试框架的学习曲线陡峭
- 模拟 API 和数据库的过程令人痛苦
- 测试抓不到真正重要的 bug
- CI 跑得太久，因为大量集成测试

### 情感性痛苦
- 本可由测试发现的生产 bug 让人尴尬
- 测试维护开销带来的挫败感
- 没有信心上线带来的焦虑
- 因时间紧迫而跳过测试的愧疚感

### 触发时刻
- 刚刚因测试本可避免的 bug 而上线
- 新同事问"测试在哪里？"
- CI 跑 20 分钟以上
- 想重构代码，结果把一切搞坏了

**把我们带给开发者的头号挫败感**：
> "我想测试我的 API，但不想花一周去学一个新框架、搞一堆 fixture。就让我写测试时，跟我已经写的代码长得一样。"

## 当前的替代方案

| 替代方案 | 开发者为何选择 | 哪里让人不满 | 促使切换的触发点 |
|-------------|-------------------|-------------------|-------------------|
| **Jest + Supertest** | 已经在用 Jest，Supertest 简单 | 慢、配置啰嗦、TypeScript 支持差 | 速度问题、TypeScript 报错 |
| **Postman/Insomnia** | 可视化、上手快 | 测试脱离代码库、版本控制困难 | 需要在 CI 中跑测试、团队扩张 |
| **Playwright/Cypress** | 强大、DX 好 | 对 API 测试来说过重、慢 | 只需要 API 测试，不需要 E2E |
| **完全自建** | 完全可控 | 维护负担、重复造轮子 | 团队扩张、需要统一规范 |
| **不作为** | 不投入时间 | 生产出 bug、没有信心 | 重大故障、新合规要求 |

## 关键差异化

| 类型 | 我们的主张 | 证据 |
|------|-----------|-------|
| **技术层面** | 比 Jest+Supertest 快 10 倍 | 基准测试：500 个测试 2 秒 vs 20 秒 |
| **DX** | TypeScript 零配置开箱即用 | 仅 `npm install`，无需配置 |
| **生态系统** | 兼容你现有的测试运行器 | Vitest、Jest、Node test runner |
| **价值观** | 测试应该像你的代码 | 没有 DSL，只有 TypeScript |
