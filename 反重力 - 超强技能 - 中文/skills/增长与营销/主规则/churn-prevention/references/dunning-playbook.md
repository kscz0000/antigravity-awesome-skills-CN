# 催收手册

恢复支付失败和减少被动流失的完整指南。

---

## 催收为何重要

- 支付失败导致 30-50% 的订阅流失
- 大多数支付失败可通过正确策略恢复
- 订阅业务每年估计因被动流失损失 1290 亿美元
- 有效催收可恢复 50-60% 的支付失败

---

## 催收时间线

```
第 -30 到 -7 天：预催收（预防失败）
第 0 天：        支付失败 → 智能重试 #1 + 邮件 #1
第 1-3 天：      智能重试 #2 + 邮件 #2
第 3-5 天：      智能重试 #3
第 5-7 天：      智能重试 #4 + 邮件 #3
第 7-10 天：     最终重试 + 邮件 #4（最终警告）
第 10-14 天：    宽限期结束 → 账户暂停/取消
第 14+ 天：      挽回序列开始
```

## 预催收：在失败发生前预防

### 卡片过期管理

| 时机 | 行动 |
|------|------|
| 过期前 30 天 | 邮件："您尾号 4242 的卡片下月过期" |
| 过期前 15 天 | 邮件："更新支付方式以避免中断" |
| 过期前 7 天 | 邮件："您的卡片 7 天后过期 — 立即更新" |
| 过期前 3 天 | 应用内横幅："支付方式即将过期" |

**邮件模板 — 卡片即将过期：**
```
主题：您尾号 4242 的卡片即将过期

您好 [姓名]，

您 [产品] 订阅的卡片将于 [日期] 过期。

立即更新支付方式以避免任何中断：

[更新支付方式 →]

只需不到 30 秒。

— [产品] 团队
```

### 卡片更新服务

主要卡网络提供自动卡片更新计划：

| 服务 | 网络 | 功能 |
|------|------|------|
| Visa Account Updater (VAU) | Visa | 自动更新存储的卡号和过期日期 |
| Mastercard Automatic Billing Updater (ABU) | Mastercard | 同上适用于 Mastercard |
| Amex Cardrefresher | American Express | 同上适用于 Amex |

**影响：** 减少 30-50% 因过期/更换卡片导致的硬拒绝。

**如何启用：**
- **Stripe**：自动 — 默认启用
- **Chargebee**：通过网关设置启用
- **Recurly**：内置，默认启用
- **Braintree**：联系处理器启用

### 备用支付方式

提示添加第二支付方式：
- 注册时："添加备用支付方式"（转化低）
- 首次成功支付后："用备用卡片保护您的账户"（时机更好）
- 支付失败恢复后："添加备用以防止未来中断"（最佳时机 — 他们感受到了痛苦）

### 预计费通知

对于年付或高价值订阅：
- 续费前 7 天发邮件，含金额和日期
- 包含更新支付方式的链接
- 展示续费包含什么
- 某些自动续费法规要求

## 智能重试策略

### 拒绝类型分类

| 代码 | 类型 | 含义 | 重试？ |
|------|------|------|--------|
| `insufficient_funds` | 软拒绝 | 临时余额不足 | 是 — 2-3 天后重试 |
| `card_declined`（通用） | 软拒绝 | 各种临时原因 | 是 — 重试 3-4 次 |
| `processing_error` | 软拒绝 | 网关/网络问题 | 是 — 24 小时内重试 |
| `expired_card` | 硬拒绝 | 卡片已过期 | 否 — 请求新卡 |
| `stolen_card` | 硬拒绝 | 卡片报告被盗 | 否 — 请求新卡 |
| `do_not_honor` | 软/硬拒绝 | 银行拒绝（模糊） | 再试一次，然后请求新卡 |
| `authentication_required` | 认证 | 需要 SCA/3DS | 引导客户认证 |

### 按服务商的重试计划

**Stripe（Smart Retries — 推荐）：**
- 在 Stripe Dashboard → Billing → Settings 中启用"Smart Retries"
- Stripe 的 ML 模型基于数十亿交易选择最佳重试时机
- 通常 3-4 周内 4-8 次重试尝试
- 比固定计划重试多恢复约 15%

**手动重试计划（如无智能重试）：**

| 重试 | 时机 | 最佳日期/时间 |
|------|------|--------------|
| 1 | 第 1 天（失败后 24 小时） | 上午，与原支付相同星期几 |
| 2 | 第 3 天 | 尝试不同时间段 |
| 3 | 第 5 天 | 典型发薪日后（1 号、15 号） |
| 4 | 第 7 天 | 下一个工作日上午 |
| 5（最终） | 第 10 天 | 宽限期结束前最后尝试 |

**重试时机洞察：**
- 在原支付成功的每月同日重试
- 在常见发薪日后重试（每月 1 号和 15 号）
- 避免周末重试（批准率较低）
- 上午重试（当地时间 8-10 点）效果略好

## 宽限期管理

### 宽限期期间发生什么

| 设置 | 建议 |
|------|------|
| 时长 | 最终重试后 7-14 天 |
| 访问权限 | 降级（只读）或完全访问 |
| 可见性 | 应用内横幅："支付逾期 — 更新以继续" |
| 重试 | 宽限期内继续后台重试 |
| 沟通 | 催收邮件继续 |

### 访问降级选项

**选项 A：宽限期完全访问（B2B 推荐）**
- 摩擦更低，客户感觉受尊重
- 恢复率更高（他们仍看到价值）
- 风险：部分客户利用宽限期

**选项 B：只读访问（B2C 推荐）**
- 可查看但不能创建/编辑
- 制造紧迫感但无数据丢失恐惧
- 清晰信息："更新支付以恢复完全访问"

**选项 C：立即锁定（不推荐）**
- 激进，损害关系
- 恢复率较低
- 仅适合极低价套餐

### 宽限期后

| 时机 | 行动 |
|------|------|
| 宽限期结束 | 暂停账户（非删除） |
| 暂停后第 1 天 | "您的账户已暂停"邮件 |
| 暂停后第 7 天 | "您的数据还在这里"提醒 |
| 暂停后第 30 天 | 挽回尝试配新优惠 |
| 暂停后第 60 天 | 最终挽回 |
| 暂停后第 90 天 | 数据删除警告（如适用） |

## 服务商特定设置

### Stripe

**启用 Smart Retries：**
1. Dashboard → Settings → Billing → Subscriptions and emails
2. 在重试规则下启用"Smart Retries"
3. 在 Dashboard → Settings → Emails 设置支付失败邮件

**自定义重试规则（如不使用 Smart Retries）：**
```
重试 1：失败后 3 天
重试 2：失败后 5 天
重试 3：失败后 7 天
最终：   最后一次重试后标记订阅为未支付
```

**需处理的 Webhook 事件：**
- `invoice.payment_failed` — 触发催收
- `invoice.paid` — 取消催收，恢复访问
- `customer.subscription.updated` — 状态变更
- `customer.subscription.deleted` — 最终取消

### Chargebee

**内置催收：**
1. Settings → Configure Chargebee → Retry Settings
2. 配置重试尝试和间隔
3. Settings → Configure Chargebee → Email Notifications → Dunning

**催收选项：**
- 可配置计划自动重试
- 内置催收邮件（可自定义模板）
- 每个套餐的宽限期配置

### Paddle

**托管催收：**
- Paddle 自动处理重试和催收
- 自定义有限（Paddle 管理关系）
- Webhook：`subscription.payment_failed`、`subscription.cancelled`
- 适合放手式方法

### Recurly

**Revenue Recovery：**
1. Configuration → Dunning Management
2. 每个套餐设置重试计划
3. 配置宽限期和最终操作（暂停 vs 取消）

**高级功能：**
- 机器学习重试优化
- 每个套餐的催收计划
- 内置 Account Updater

## 应用内催收

不要仅依赖邮件。在应用内展示支付失败：

### 横幅模式
```
┌──────────────────────────────────────────────────────┐
│ ⚠ 您的 ¥29 支付失败。更新卡片以避免失去访问权限。    │
│ [更新支付方式 →]  [关闭]                             │
└──────────────────────────────────────────────────────┘
```

**规则：**
- 催收期间每次页面加载都展示
- 允许关闭（但下次会话再展示）
- 直接链接到支付更新（最少点击）
- 不阻止产品 — 让他们继续使用

```
┌─────────────────────────────────────┐
│                                     │

│                                     │

│  和 [Y] 位团队成员的访问。          │
│                                     │
│  [更新支付方式]                     │

│                                     │
```

## 衡量催收效果

### 关键指标

| 指标 | 计算方法 | 目标 |
|------|---------|------|
| 恢复率 | 恢复支付数 / 总失败数 | 50-60% |
| 按拒绝类型的恢复率 | 每种类型恢复数 / 失败数 | 软拒绝：70%+，硬拒绝：40%+ |
| 恢复时间 | 从失败到成功支付的天数 | <5 天 |
| 预催收预防率 | 预防失败数 / 预期失败数 | 20-30% |
| 催收邮件打开率 | 每封邮件打开数 / 发送数 | 60%+ |
| 催收邮件点击率 | 每封邮件点击数 / 打开数 | 30%+ |
| 恢复收入（月度） | 恢复支付金额总和 | 追踪趋势 |
| 被动流失损失收入 | 失败 + 未恢复金额总和 | 追踪趋势 |

### 基准

**按公司阶段：**

| 阶段 | 典型被动流失 | 优化后目标 |
|------|-------------|-----------|
| 早期（< $1M ARR） | MRR 的 3-5%/月 | 1-2% |
| 成长期（$1-10M ARR） | MRR 的 2-4%/月 | 0.5-1.5% |
| 规模化（$10M+ ARR） | MRR 的 1-3%/月 | 0.3-0.8% |

### ROI 计算

```
月度支付失败 MRR：        ¥10,000
当前恢复率：              30%（恢复 ¥3,000）
目标恢复率：              60%（恢复 ¥6,000）
月度改进：                ¥3,000/月
年度改进：                ¥36,000/年
催收优化成本：            约 ¥200-500/月（工具）
ROI：                     6-15 倍
```
