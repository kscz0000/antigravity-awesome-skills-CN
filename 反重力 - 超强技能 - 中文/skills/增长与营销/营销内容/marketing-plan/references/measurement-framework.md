# 度量框架 — KPI、北极星指标、评审节奏

每个计划都需要一个度量部分，告诉团队如何判断计划是否奏效。本文档是第 13 节度量子部分的来源。

**相关文档：**
- `growth-patterns.md` — 3-3-2-2-2 VC 增长路径（第 1–2 年 3×，从 $1M ARR 起第 3–7 年 2×）以及公司处于 SaaS 增长的哪个阶段（$0–10K / $10K–100K / $100K–1M+）
- `budget-planning.md` — CAC 计算（综合加本，而非仅付费）和预测现实检查（$100M ARR 以下的预测是有根据的猜测，不是精确预测）

## 北极星原则

北极星是一个指标，在最高层面捕捉商业模式论点。它应该：
- 可以从漏斗 + 变现模型推导
- 变动足够缓慢，可作为战略指南针（不被每周噪音左右）
- 与其他指标正确权衡 — 改善北极星通常应改善业务

不要默认选择"ARR"或"MRR"单独作为北极星。那些是结果，不是方向。选择一个能捕捉商业模式的指标。

## 按商业模式的北极星模式

### B2B SaaS（订阅制）
- **净收入留存率（NRR）** — 保持现有客户 + 扩展收入在焦点中
- 替代方案："Logo 留存率 × 扩展 ARR"
- 原因：仅看 ARR 会隐藏流失 / 让毛增增长掩盖产品匹配问题

### D2C 消费者应用（订阅制）
- **综合 LTV / 综合 CAC** — 在付费层扩展时保持单位经济效益诚实
- 替代方案："第 35 天付费用户 × LTV"
- 原因：月度订阅指标波动大；群组 × LTV 可平滑波动

### 硬件 + 软件混合（如 Quietude）
- **跨硬件 + 软件的综合 LTV / 综合 CAC** — 捕捉楔子论点
- 替代方案："硬件购买者到订阅者转化率 × 综合利润率"
- 原因：硬件收入不是免费的（有制造成本）；如果硬件资助获客，订阅收入的获取成本就不高

### 市场平台（双边）
- **流动性比率 × 抽成率** — 捕捉双方 + 变现
- 替代方案："月度交易用户 × 抽成率 × 重复频率"
- 原因：仅看 GMV 不能捕捉市场平台是否正在成为习惯

### 开发者工具 / 开源
- **周活跃开发者 × 付费转化率** — 捕捉采用和变现
- 替代方案："周活跃组织 × 每组织席位数 × ARPU"

### 内容 / 媒体业务
- **日活跃读者 / 听众 × 每会话广告收入** — 捕捉触达和变现
- 替代方案："订阅者数 × 留存率 × ARPU"

### 电商（DTC，非订阅制）
- **复购率 × AOV × 频率** — 捕捉叠加在客户质量上的变现
- 替代方案："客户 LTV / CAC × 回本周期"

## 按 AARRR 阶段的领先指标

在北极星之后，每个计划需要按 AARRR 阶段的领先指标。这些比北极星变化更快，可触发调查。

### 获客领先指标
- 有机访问量/月，总量 + 按支柱（SEO 健康）
- App Store / Play Store 访问到安装率（ASO 健康）
- 创始人主导社交渠道增长 → 邮件订阅者转化（LinkedIn / X / Substack 漏斗）
- 活动到应用转化率（活动 ROI）
- 大使归因访问量（推荐漏斗）
- 按渠道的付费 CAC（当付费渠道运行时）

### 激活领先指标
- 第 1 天 / 第 7 天 / 第 35 天 → 付费转化率
- 引导流程完成率
- 首次关键操作完成（注册后激活事件）
- App Store 转化率（安装 → 试用 → 付费）
- 试用 → 付费转化率

### 留存领先指标
- 第 30 天 / 第 60 天 / 第 90 天留存
- 月度流失率（毛 + 净）
- 生命周期邮件参与度（按流程的打开 / 点击 / 退订率）
- 硬件 → 应用激活率（混合业务）
- 赢回 / 重新激活率

### 推荐领先指标
- 大使归因新增订阅（通过 Dub 或类似工具）
- 价值时刻后分享率（分享用户占比）
- 双边推荐完成率
- Guides 项目推荐（上线时）
- NPS 评分（如已调研）

### 变现领先指标
- 按群组的 ARPU
- 年度计划采用率 %
- 按来源的群组 LTV
- 计划分布变化
- 眼罩 / 硬件附加率（混合业务）
- 扩展收入（B2B）

## 评审节奏

计划应指定三种节奏：

### 周度（运营同步）
- **参与人：** fCMO ↔ 创始人（通常是 CEO）
- **时长：** 30 分钟
- **格式：** AARRR 记分板（当前 vs 上周领先指标数据）+ 本周交付 + 阻碍
- **输出：** 行动项、已做决策

### 月度（指标评审）
- **参与人：** fCMO + 创始人 + 扩展团队（CXO、产品负责人、设计师如适用）
- **时长：** 60–90 分钟
- **格式：** 全面指标评审 + 与季度 KPI 目标对比 + 定性发现 + 想法库重新排序
- **输出：** 可能的计划调整、招聘决策

### 季度（计划再校准）
- **参与人：** fCMO + 创始人 + 关键顾问
- **时长：** 2–3 小时
- **格式：** 对照 90 天和 12 个月成果的全面计划评审、渠道级分析、融资阶段转换检查、下个 90 天再校准
- **输出：** 更新后的计划（可能是 v2 / v3 文档迭代）

## KPI 目标设定

对于第 10 节中的每个季度，计划必须包含 3–5 个具体的 KPI 目标。这些应该是：
- **具体的** — 不是"改善留存"，而是"第 30 天留存从 22% → 30%"
- **可度量的** — 从已对接的数据源获取
- **有挑战但可实现的** — 基于漏斗状态 + 历史模式
- **触发决策的** — 如果未达成，意味着什么？（调整策略、砍掉某个渠道等）

### 按季度的 KPI 目标模式

**Q1（基础季度）：**
- 主要是*基础修复*指标 — 修补漏点。"耳机门转化下降逆转。""第 1 天 → 付费 +25–50%。"
- 一些*基础建设*指标 — 铺设轨道。"4 个 SEO 支柱确立。""App Store 重写上线。"
- 避免大胆的增长目标 — 基础设施还没就位

**Q2（验证季度）：**
- 主要是*验证*指标 — 我们构建的东西有效吗？"综合付费 CAC < $X。""有机流量 1,500–3,500/月。"
- 一些*群组*指标 — 新群组表现更好吗？"Q2 群组 vs Q1 的第 7 天留存。"

**Q3（扩展季度）：**
- 主要是*扩展*指标 — 能推多远？"付费扩展到 $20–30K/月且 CAC 稳定。""首个 B2B 安装参考案例上线。"
- 一些*能力*指标 — 有什么新东西上线？"首次 Guides 试点启动。"

**Q4（复合季度）：**
- 主要是*复合*指标 — 飞轮在转吗？"50%+ 新订阅来自非付费渠道。""大使驱动 15–25% 新订阅。"
- 一些*叙事*指标 — Series A 的故事能自圆其说吗？"综合 LTV/CAC > 3。"

## 以 VC 增长路径为锚点

对于已过 $1M ARR 的 VC 支持客户，将 12 个月和多年目标锚定到 **3-3-2-2-2 规则**（第 1 和第 2 年 3×，然后第 3 到第 7 年 2×）。达成此目标很少见；大多数公司做不到。锚定它迫使计划要么匹配并展示如何做到，要么明确辩护为何选择较慢的轨迹。完整表格和上下文见 `growth-patterns.md`。

对于非 VC 支持的公司（自举、创始人自资、利润导向），3-3-2-2-2 不适用。改用线性模式目标（"每月增加 $X MRR"）或阶梯函数目标（"企业版上线后收入跳跃 $Y"）。

## 预测现实检查

计划推导出预算和年度目标。它不产出精确到美元的 12 个月逐月预测。

**除非公司是上市公司，否则所有预测都是有根据的猜测。** 没有低于 $100M ARR 的初创公司能持续命中逐月预测。季度评审是计划调整的时机 — 不是将偏差视为失败的时候。

计划诚实承诺的内容：
- 年度目标是可辩护的方向性指引
- 预算是使目标合理的资源承诺
- 90 天路线图（第 9 节）是当前可执行的
- 逐月预测是说明性的，不是承诺

过度工程化预测的创始人最终每月都在解释偏差而非执行。计划应抵制这一点 — 命名年度目标、季度 KPI 和终止条件。不要承诺月份。

完整上下文见 `budget-planning.md`。

## 终止条件

对于每个渠道或举措，计划应指定何时停止。计划中常缺少终止条件，它强制纪律。

示例：
- "如果付费渠道在有意义的花费下 30 天后 CAC > 2× 目标，暂停。"
- "如果引导变体 3 在 4 周后没有显示统计上显著的提升（或有方向性提升 + 一致的定性信号），切换到变体 1。"
- "如果生命周期流程 4 在 6 周后打开率 < 12%，重做主题行 + 受众分群。"

## 护栏指标

某些指标设置硬护栏（不能低于阈值）。适用于在激进增长期间保护品牌或单位经济效益。

示例：
- "品牌语调投诉率 > 客户反馈的 1% 触发内容审查。"
- "付费 CAC > $X 连续两个月暂停付费扩展并审计。"
- "App Store 评分降至 4.5 以下触发产品审查。"

## 数据源映射

计划应指出每个指标的来源。这使其可审计。

| 指标 | 来源 |
|---|---|
| 有机流量 | GA4 / Ahrefs |
| App Store 转化 | App Store Connect |
| 漏斗转化（第 N 天 → 付费） | 内部分析（Mixpanel / Amplitude）或 App Store Connect 群组导出 |
| 留存 | Customer.io 分群 + 产品分析 |
| MRR / ARR | Stripe（如已对接则通过 MCP） |
| 计划分布 | Stripe |
| 生命周期邮件指标 | Customer.io |
| 大使归因 | Dub.co |
| 硬件 → 应用激活 | Shopify + App Store + 内部关联 |
| NPS | 调研工具（Customer.io / Typeform / SurveyMonkey） |

## 当数据未对接时

如果某个指标当前无法度量，在第 13 节的待定决策中标记。示例：

> "硬件 → 应用激活率目前在 App Store 仪表板中不可见。需要 Shopify ↔ App Store Connect 关联。Q1 工作项。"

有不可度量目标的计划是无法验证的计划。明确暴露埋点工作。

## 报告节奏 + 自动化

在可能的情况下，自动生成指标评审，而非每次手动构建。Stripe MCP + GA4 MCP + Customer.io MCP 可以拉取大部分所需数据。

对于 Tier 1 客户，一封简单的每周指标邮件（Markdown 表格，通过技能 + MCP 生成）成本为零并创造纪律。

对于 Tier 2+ 客户，考虑实时仪表盘（Hex、Metabase、Looker 或内部工具）。
