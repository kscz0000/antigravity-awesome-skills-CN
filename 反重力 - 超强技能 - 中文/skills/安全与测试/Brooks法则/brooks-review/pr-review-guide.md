# PR 审查指南 — 模式 1

**目的：** 分析代码 diff 或特定文件中可直接观察到的腐化风险。每条发现都必须遵循铁律：症状 → 根源 → 后果 → 修复。

---

## 开始之前

**自动生成的文件：** 若 diff 包含自动生成的文件（protobuf 桩、OpenAPI 客户端、ORM 迁移、锁文件、压缩后的打包文件），请整体跳过这些文件。生成代码反映的是工具选型而非开发者的决策。请在报告中注明跳过了哪些文件及原因。

**范围校准：** 在开始前根据 PR 体量调整分析深度。

| PR 体量 | 做法 |
|---------|------|
| < 50 行 | 仅聚焦步骤 1–3；若 import 有变则执行步骤 6a；若新增或重命名了类、方法、变量，则执行步骤 6b |
| 50–300 行 | 完整流程，全部步骤 |
| > 300 行 | 完整流程；在"范围"一行注明本次为抽样审查，优先覆盖高风险区域而非逐文件审查 |

针对超过 500 行的 PR：在"摘要"中标记——体量如此本身就是变更传播的信号。一次过无法审完的改动，意味着职责纠缠不清。

---

## 分析流程

按顺序完成以下七个步骤。不得跳过。

### 步骤 1：理解范围

阅读 diff 或文件并回答：
- 本次改动的既定目的是什么？
- 修改了哪些文件？
- 若 PR 修改了 10 个以上互不相关的文件，立即标记——
  🟡 警告：变更传播（一份 PR 改动许多互不相关的内容，是职责纠缠的征兆）。

### 步骤 2：扫描变更传播

*优先扫描此步——它是 diff 中最显见的风险。*

查找：
- 本次改动是否触及与既定目的并无概念关联的模块中的文件？
- 任一被修改的类在本次 diff 中是否因多个业务理由而变化？
- 任一方法是否使用了来自另一个类的数据，多于使用自身的数据？

若 diff 中未出现超出功能所需的跨模块改动 → 跳过，无需记录发现。

### 步骤 3：扫描认知过载

查找：
- 是否有新增或修改的函数超过 20 行？
- 新增或修改代码中是否有超过 3 层的嵌套？
- 是否有任一新函数签名参数超过 4 个？
- 新代码中是否存在魔法数字或未解释的常量？
- 新变量或函数名是否需要阅读实现才能理解？
- 是否存在火车链式调用（3 个及以上方法串联）？

### 步骤 4：扫描知识重复

查找：
- 本次改动是否引入了代码库中已存在的逻辑？
- 本次改动是否为已命名的概念引入了一个新名称？
- 本次改动是否在层级结构中新增了一个与其他模块有并行结构的类？

### 步骤 5：扫描意外复杂度

查找：
- 本次改动是否新增了仅有一个具体用途的抽象？
- 本次改动是否新增了仅封装另一个类或全权委托的类？
- 本次改动是否新增了无当前需求的配置选项或扩展点？

### 步骤 6a：扫描依赖错乱

- 任一新 import 是否造成高层模块对低层模块的依赖？
  （例如：领域服务现在导入了数据库驱动或 HTTP 客户端）
- 任一新 import 是否引入了模块间的循环？
- 任一新接口是否迫使调用方依赖其不使用的方法？

若没有新增 import 也无结构性变更 → 跳过，无需记录发现。

### 步骤 6b：扫描领域模型扭曲

- 新增的类或变量名是否与业务对同一概念使用的语言一致？
- 是否有新增类只持有数据而无行为（纯数据袋），而本应有行为？
- 是否有新增方法把应属领域的逻辑放到了服务层或工具层？

## 严重程度校准

应用 `../_shared/common.md` 的铁律格式。`../_shared/decay-risks.md` 中的每一项风险都有带数值阈值的严重程度指南——以该指南为主参考。当发现处在两个等级边界时，按此规则取舍：
- 🔴 严重 —— 当前正在破坏开发速度或制造生产风险
- 🟡 警告 —— 若不处理，经过后续几个特性后会演变成严重
- 🟢 建议 —— 顺路修即可，不紧急

当存在多条发现时，优先列出"严重"。若发现超过 5 条，请在"发现"章节末尾追加一行"建议修复顺序"。

---

## 步骤 7：快速测试检查

*此步最后执行。仅看三个信号——并非完整的模式 4 审查。*

若 diff 仅包含自动生成文件、配置或文档，且无生产逻辑变更 → 整体跳过步骤 7。

**信号 1：是否存在针对变更行为的测试？**

- diff 是否修改了生产代码？
- 相应的测试文件变更是否包含在 diff 中？
- 若新增公共行为却未新增测试：
  → 🟡 警告：覆盖幻觉——新行为未受测试覆盖
  → 来源：Feathers ——《Working Effectively with Legacy Code》第 1 章
- 若改动为纯重构且已有测试覆盖行为 → 不记录发现。

**信号 2：快速嗅探 Mock 滥用**

仅当 diff 包含测试文件变更时检查。

- 新增或修改的测试中，mock 设置代码是否明显比测试逻辑还长？
- 主要断言是否为 `expect(mock).toHaveBeenCalledWith(...)` 且没有行为验证？
- diff 是否给生产类新增了仅在测试文件中被调用的方法？

若以上任一为真：
  → 🟡 警告：Mock 滥用——测试复杂度超过行为复杂度
  → 来源：Osherove ——《The Art of Unit Testing》mock 使用指南

**信号 3：快速嗅探测试可读性差**

仅当 diff 包含测试文件变更时检查。

- 新测试的名称是否同时表达场景与预期结果？
  （模式：`methodName_scenario_expectedResult` 或等同写法）
- 是否有新增测试包含多条断言且均无消息字符串？

若测试名称模糊或断言缺少消息：
  → 🟢 建议：测试可读性差——从测试名或断言看不出意图
  → 来源：Meszaros ——《xUnit Test Patterns》，Assertion Roulette（第 224 页）

**输出规则：**

若三项信号均干净 → 不输出测试相关发现，直接进入报告。

若存在发现 → 按标准铁律格式加入"发现"章节。
将风险标记为对应的测试腐化名称（如"覆盖幻觉"、"Mock 滥用"、"测试可读性差"）。

> **注意：** 步骤 7 是快速检查，并非完整的测试审计。若发现系统性的测试问题，请在"摘要"中注明："建议运行 `/brooks-lint:brooks-test` 进行完整的测试质量诊断。"
