# 测试质量审查指南 — 模式 4

**目的：** 通过六类测试维度衰退风险，诊断测试套件的健康状况。
每项发现都必须遵循铁律：症状 → 来源 → 后果 → 修复。

---

## 开始之前：绘制测试套件全景图

在扫描任何风险之前，先梳理当前测试套件的结构：

```
Unit tests:        X files, ~N tests
Integration tests: X files, ~N tests
E2E tests:         X files, ~N tests
Ratio:             Unit X%  :  Integration X%  :  E2E X%
Coverage areas:    [modules with tests] vs [modules without tests]
```

如果无法直接访问测试文件，则向用户**只提一个问题**——选择最相关的那一条：
1. "哪个模块最难测试，或者测试覆盖最少？"
2. "当你做一次改动时，多常会出现无关的测试也跟着失败？"
3. "代码库里是否有某部分，因为没有测试，团队一直回避去碰它？"

得到一个答案后即可继续。**不要再追加问题。**

---

## 分析流程

按以下五个步骤顺序推进。

### 步骤 1：扫描测试晦涩度

*优先扫描此项——它是最显眼的风险，也是判断测试套件是否具备可维护性的关键指标。*

查找要点：
- 随机抽取 5–10 个测试名：每个测试名是否能在不打开测试体的前提下，把"主体 + 场景 + 预期结果"三要素表达清楚？
- 是否存在这样的测试：一旦失败，无法从报错信息中判断是哪个行为被破坏（多重断言、缺少说明字符串）？
- 是否有测试依赖外部状态（文件、数据库行、环境变量、共享的可变 fixture），而这种依赖在测试体中不可见？
- 是否存在一个臃肿的 setUp 或 beforeEach，让每个测试无论实际是否需要都得继承？

如果所有测试名都清晰、且 setup 都最小化 → 不出具发现。

### 步骤 2a：扫描测试脆弱性

*脆弱测试在未改变可观察行为的重构中崩溃——它们测的是实现细节，而不是契约。*

查找要点：
- 询问用户（或查看 git 历史）：近期是否有重构引发过测试失败，但并未改变行为？
- 是否存在名称中包含 "and" 的测试方法，或者断言了 3 个及以上互不相关行为（急切测试 / Eager Test）？
- 断言是否规定了 mock 调用顺序或无关紧要的精确参数值，而这些与可观察行为无关？
- 测试是否与私有方法或内部状态直接耦合？

如果是系统性问题（文件里大多数测试在重命名时都会崩）→ 🔴 严重。
如果是局部问题（1–2 个脆弱测试）→ 🟢 建议。

### 步骤 2b：扫描 Mock 滥用

*Mock 滥用会产生"无论真实行为是否正确，测试都能通过"的用例。
与脆弱性分开扫描——过度 mock 往往是脆弱性的成因，但它本身是另一个独立问题，值得单独列出。*

**对 3–5 个测试一次性同时抽样，进行步骤 2a 和 2b 的检查**——在同一次阅读中同时核对脆弱性信号和 mock 与断言的比例，再分别撰写发现（如两者皆存在）。

查找要点：
- 在所抽样测试中，mock 设置代码是否比断言逻辑更长？
- 主要断言是否为 `expect(mock).toHaveBeenCalledWith(...)` 这类形式，而不是对输出、状态或可观察事件的断言？
- 生产类中是否存在仅在测试文件中被调用的方法（测试诱导的设计损伤）？
- 是否有单个测试创建了超过 3 个 mock 对象？

如果 mock 设置与断言的比例超过 3:1 → 🟡 警告。
如果生产方法仅为了测试可访问而存在 → 🔴 严重（架构正被测试套件扭曲）。

### 步骤 3：扫描测试重复

查找要点：
- 相同的 setup 块（用相同方式初始化相同变量）是否在 5 个及以上测试文件中重复出现，且缺少共享的辅助函数？
- 是否存在多个测试，输入相同、断言也相同，且没有差异化（懒测试 / Lazy Test）？
- 同一个业务场景是否在单元、集成、E2E 三层都被覆盖，而各层之间没有区分测什么？

如果是系统性问题（10 处及以上）→ 严重。
如果是局部问题（3–5 处）→ 警告。

### 步骤 4：扫描覆盖率幻觉与架构不匹配

查找覆盖率幻觉：
- 挑出最近被修改过的核心模块：它的错误处理分支以及 null/边界输入是否被测试覆盖？
- 是否存在那些没有测试文件覆盖的遗留区域（旧函数、附近找不到测试），但仍在被持续修改？
- 测试断言的是副作用（数据库写入、事件触发、状态流转），还是只断言返回值？

**特征化测试（Characterization Test）检查：** 如果遗留代码在被修改之前没有现成测试，团队需要在动手**之前**补齐特征化测试——而不是改完之后再补。识别该模式，若缺失则标注。

特征化测试用于锁定"当前行为"（无论对错），防止后续改动悄悄回归。模板：
```
test("characterize: [module].[method] given [input], returns [current output]") {
  // Call the code under test with realistic inputs
  // Assert on whatever it currently returns — even if you suspect the output is wrong
  // Add a comment: "This captures current behavior, not necessarily correct behavior"
}
```
来源：Feathers —《修改代码的艺术》第 13 章：特征化测试

查找架构不匹配：
- 对照一开始的套件全景图：单元 / 集成 / E2E 的比例是否接近 70% / 20% / 10%？
- 高风险模块的测试密度是否高于那些边边角角的工具类？

**测试套件性能：** 测试套件运行慢属于一等可维护性风险——它会打破快速反馈循环，导致开发者跳过本地测试运行。
- 若已知全量套件运行时间 > 10 分钟 → 🟡 警告
- 若全量套件运行时间 > 30 分钟，或根本未知 → 🔴 严重（套件耗时未知意味着没人定期跑）
- 若本可作为单元测试的用例却写成集成测试，则属于性能不匹配：每一例被错分类的测试都会带来数秒的额外等待

来源：Meszaros — xUnit Test Patterns，慢测试（第 253 页）

### 步骤 5：应用铁律，输出报告

针对每项发现，应用 `../_shared/common.md` 中的铁律格式。

使用标准报告模板。模式：Test Quality Review。
在 `## Findings` 标题之前，立即以代码块形式插入测试套件全景图，标注为 "Test Suite Map"。
