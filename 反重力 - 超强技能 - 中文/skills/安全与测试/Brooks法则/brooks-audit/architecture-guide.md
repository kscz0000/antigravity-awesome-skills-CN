# 架构审计指南 — 模式 2

**目的：** 分析系统的模块与依赖结构，识别在架构层面运作的腐化风险。每一条发现都必须遵循"铁律"：症状 → 出处 → 后果 → 处方。

**Monorepo 提示：** 把每个可独立部署的服务或库视作顶层模块。在服务之间绘制依赖关系，而非其内部包之间。康威定律的检查也落在服务所有权层面。在单个服务内部，则按标准的模块级分析进行。

---

## 分析流程

按顺序完成以下六个步骤。

### 步骤 0：收集代码库上下文

在动手绘制之前，先确认你能看到什么。

**如果用户提供了完整目录树或粘贴了相关文件内容：** 跳过下面的主动读取，直接进入步骤 1。

**否则，使用下列工具主动读取项目：**

1. **顶层结构** —— glob 前两层以识别模块边界：
   ```
   Glob: **/*(depth 2, directories only)
   ```
2. **入口点** —— 读取包清单或主配置文件（例如 `package.json`、`go.mod`、`pom.xml`、`Cargo.toml`、`pyproject.toml`），以确认语言、框架和声明的依赖。
3. **依赖边** —— 用 grep 抓取 import 语句，发现模块间调用。每种语言各跑一次；为避免 token 过多，最多匹配前 200 条：
   ```
   Grep: "^\s*(import|from|require\(|use )" across *.ts|*.py|*.go|*.rs|*.java
   ```
4. **大型模块** —— 对于任何文件数大于 10 的顶层目录，读取匹配 `index.*`、`main.*` 或 `__init__.*` 的文件，以理解其声明的职责。

**当你能够同时回答下列三问时即可停止：**
- 顶层模块有哪些（名称与数量）？
- 哪些模块从哪些模块导入？
- 哪个模块的扇入（fan-in）或扇出（fan-out）最大？

如果项目的顶层文件数大于 100 或嵌套深度大于 4，请在报告中注明哪些区域是抽样检查、哪些是推断得出，并在报告范围行中标记。

### 步骤 1：绘制模块依赖图（Mermaid）

在评估任何风险之前，先把依赖关系绘制成 Mermaid 图。使用以下格式：

````mermaid
graph TD
  subgraph UI
    WebApp
    MobileApp
  end

  subgraph Domain
    AuthService
    OrderService
    PaymentService
  end

  subgraph Infrastructure
    Database
    MessageQueue
  end

  WebApp --> AuthService
  WebApp --> OrderService
  MobileApp --> AuthService
  MobileApp --> OrderService
  OrderService --> PaymentService
  OrderService --> Database
  OrderService --> MessageQueue
  PaymentService --> Database
  AuthService -.->|circular| OrderService

  classDef critical fill:#ff6b6b,stroke:#c92a2a,color:#fff
  classDef warning fill:#ffd43b,stroke:#e67700
  classDef clean fill:#51cf66,stroke:#2b8a3e,color:#fff

  class PaymentService critical
  class OrderService warning
  class Database,MessageQueue,AuthService,WebApp,MobileApp clean
````

先画出图的结构——节点、子图、边——不要带任何 `classDef` 或 `class` 行。在完成第 2–4 步的风险扫描之前，你都没办法分配颜色。

**在完成第 4 步之后**，回到这张图，基于发现结果添加 `classDef` 与 `class` 行。上例展示的就是带颜色的最终输出。

规则：
1. **节点** —— 使用顶层目录或服务作为节点，而非单个文件
2. **分组** —— 每个架构层或顶层目录一个 `subgraph`（例如 UI、Domain、Infrastructure）
3. **边** —— 实线箭头（`-->`）从依赖方指向被依赖方；循环依赖使用带标签的虚线箭头（`-.->|circular|`）。如果没有循环依赖，则仅使用实线箭头
4. **节点上限** —— 图中节点数控制在约 50 个以内；必要时把低风险的叶子模块折叠到其父节点
5. **扇出** —— 对于扇出大于 5 的节点，使用描述性标签：`HighFanOutModule["ModuleName (fan-out: 7)"]`
6. **颜色** —— 在完成第 2–4 步之后再应用 `classDef` 颜色：`critical`（红色 `#ff6b6b`）用于包含"严重"发现的节点，`warning`（黄色 `#ffd43b`）用于"警告"发现的节点，`clean`（绿色 `#51cf66`）用于没有发现或仅有"建议"的节点。如果完全没有发现，则所有节点都归为 `clean`
7. **方向** —— 默认使用 `graph TD`（自上而下）；只有当架构明显是左到右的流水线时，才使用 `graph LR`

### 步骤 2：扫描依赖混乱

*这是架构层面影响最大的风险——请优先扫描。*

留意：
- 循环依赖（上图中任何 `-.->|circular|` 边）
- 箭头向上流动（高层领域依赖底层基础设施）
- 稳定、被广泛依赖的模块却从频繁变动的模块导入
- 扇出大于 5 的模块
- 缺乏清晰的分层规则（对"谁依赖谁"没有一个一致的答案）

### 步骤 3：扫描领域模型扭曲

留意：
- 模块名是否与业务领域词汇一致？
- 是否存在一个名为 "services" 的层，把全部业务逻辑都装了进去，而领域对象却只是纯粹的数据结构？
- 是否存在跨越限界上下文边界的模块（例如，把计费逻辑放到用户模块中）？
- 是否有防腐层，让外部系统与领域对接？

### 步骤 4：扫描剩余的四类风险

依次检查每一项：

**知识重复：**
- 是否存在多个模块各自独立实现同一概念？
- 同一个领域概念是否在不同模块里以不同名字出现？

**意外复杂度：**
- 架构中是否整层都没有创造价值？
- 是否存在职责无法用一句话说清的模块？

**变更扩散：**
- 哪些模块是"爆炸半径热点"？（改动这里会牵连许多其他模块）
- 依赖图能否解释为何某些功能开发得慢？

**认知过载：**
- 仅凭模块名，能否用一句话说清每个模块的职责？
- 新加入的开发者是否知道一个新功能应该加到哪个模块？

### 步骤 5：可测试性接缝评估

*接缝*是架构中可以替换行为而无需修改源代码的位置——通常是接口、配置点或依赖注入边界。接缝密度是可测试性与可演化性的代理指标。

扫描以下情形：
- **基础设施边界处没有接缝**：能否在不修改被测模块的前提下，把真实的数据库、文件系统或 HTTP 客户端换成测试替身？如果不能，那么架构就在用集成测试代替本可胜任的单元测试。
- **接缝坍塌**：曾经可以独立测试的模块，其接缝被抹掉了（例如，依赖注入点被直接的构造函数实例化取代，或者被注入的协作者被全局单例替换）。
- **遗留区域缺少接缝**：模块没有明显的注入点或接口边界——任何改动都要动整条调用栈才能替换行为。

如果所有模块在基础设施边界处都有清晰接缝 → 不输出发现。

如果接缝缺失或坍塌：以 🟡 警告标记，并在处方中指明具体模块以及需要恢复或引入的注入点。

出处：Feathers——《Working Effectively with Legacy Code》第 4 章：接缝模型

### 步骤 6：康威定律检查

完成六类风险扫描后，评估架构与团队结构之间的关系：

- 模块或服务的结构是否反映了团队结构？
  （康威定律："组织设计出的系统，正是其沟通结构的镜像"）
- 如果是：这是有意的设计还是偶然的耦合？
- 错配导致每个特性都要跨团队协调 → 🔴 严重。
- 错配理论上存在但尚未造成痛点 → 🟡 警告。
- 如果团队结构未知，请注明上下文缺失并跳过本检查。

**校准示例：**
- 🔴 严重：Payments 模块归 Team A 所有，却包含归 Team B 所有的鉴权逻辑——每次改 Payments 都要和 Team B 开同步会
- 🟡 警告：两个独立的团队分别拥有 `utils/` 与 `helpers/` 目录，做的事情相似——理论上很痛，但暂未引发发布协调问题
- 不算发现：单一团队拥有的 monorepo 内部含多个逻辑模块——康威定律的错配需要存在*独立的团队*才有意义
