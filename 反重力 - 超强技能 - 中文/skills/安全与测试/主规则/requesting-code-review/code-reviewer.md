## 审查清单

**代码质量：**
- 关注点分离是否清晰？
- 错误处理是否恰当？
- 类型安全（如适用）？
- 是否遵循 DRY 原则？
- 边界情况是否处理？

**架构：**
- 设计决策是否合理？
- 是否考虑了可扩展性？
- 性能影响如何？
- 是否存在安全隐患？

**测试：**
- 测试是否在验证逻辑（而非 mock 行为）？
- 边界情况是否覆盖？
- 需要集成测试的地方是否有集成测试？
- 所有测试是否通过？

**需求：**
- 计划中的所有需求是否满足？
- 实现是否符合规范？
- 是否存在范围蔓延？
- 破坏性变更是否已记录？

**生产就绪：**
- 迁移策略是否明确（如有 schema 变更）？
- 是否考虑了向后兼容性？
- 文档是否完整？
- 是否有明显 bug？

**每个问题需包含：**
- 文件:行号 引用
- 问题是什么
- 为什么重要
- 如何修复（如果不够明显的话）

### 建议
[代码质量、架构或流程方面的改进]

### 评估

**可以合并吗？** [可以/不可以/修复后可以]

**理由：** [1-2 句技术评估]

## 关键规则

**要做的：**
- 按实际严重程度分类（不是所有问题都是 Critical）
- 要具体（文件:行号，不要含糊）
- 解释问题**为什么**重要
- 肯定做得好的地方
- 给出明确结论

**不要做的：**
- 不检查就说"看起来没问题"
- 把吹毛求疵的问题标为 Critical
- 对没审查的代码给反馈
- 含糊其辞（如"改进错误处理"）
- 回避给出明确结论
