# 事件响应 Playbook

> 常见安全事件的扩展 Playbook。
> 每个遵循 5 阶段：遏制、评估、修复、预防、记录。
> 与 `007 incident` 配合使用或在响应任何安全事件时使用。

---

## Playbook 1：数据泄露

**严重性：** 严重
**响应时间：** 立即（< 15 分钟开始遏制）

### 阶段 1：遏制
- [ ] 识别泄露来源（被攻陷凭证、漏洞、内部人员）
- [ ] 立即撤销被攻陷凭证（API 密钥、Token、密码）
- [ ] 将受影响系统从网络隔离
- [ ] 如可识别，阻止攻击者的 IP/访问路径
- [ ] 保留取证证据（暂不要擦除或重启受影响系统）

### 阶段 2：评估
- [ ] 确定暴露了什么数据（PII、财务、凭证、业务数据）
- [ ] 确定范围：影响了多少用户/记录
- [ ] 识别攻击时间线（何时开始、何时被发现）
- [ ] 审查访问日志追踪攻击者行为
- [ ] 评估数据是否被窃取或仅被访问

### 阶段 3：修复
- [ ] 修补遭攻击者利用的漏洞
- [ ] 强制所有受影响用户重置密码
- [ ] 轮换所有可能被攻陷的机密（API 密钥、数据库密码、证书）
- [ ] 清理恶意软件/后门（如已安装）
- [ ] 如数据被篡改，从干净备份恢复

### 阶段 4：预防
- [ ] 实施泄露期间识别的缺失访问控制
- [ ] 为使用的攻击模式添加监控
- [ ] 为暴露的数据存储启用静态加密
- [ ] 实施 DLP（数据防泄漏）规则
- [ ] 审查并限制访问权限（最小权限）

### 阶段 5：记录
- [ ] 完成带时间戳的事件时间线
- [ ] 根本原因分析（RCA）
- [ ] 所有受影响系统和数据的列表
- [ ] 采取的行动及执行人
- [ ] 监管通知（LGPD：72 小时，GDPR：72 小时）
- [ ] 如 PII 被暴露，通知用户
- [ ] 经验教训和流程改进

### 沟通模板
```
主题：[严重] 安全事件 - 检测到数据泄露

状态：{时间戳} 起活跃事件
严重性：严重
事件 ID：INC-{YYYY}-{NNN}

摘要：
检测到影响 {范围} 的数据泄露。泄露涉及来自 {源系统} 的 {数据类型}。

当前行动：
- 被攻陷访问已撤销
- 受影响系统已隔离
- 调查进行中

受影响数据：
- 类型：{PII / 财务 / 凭证 / 业务}
- 记录：{约数}
- 用户：{约数}

下一步：
- {预计时间} 完成取证分析
- {截止时间} 监管通知
- {截止时间} 用户沟通

联系人：{事件指挥官} {联系方式}
```

## Playbook 2：DDoS / DoS

**严重性：** 高
**响应时间：** < 5 分钟开始缓解

### 阶段 1：遏制
- [ ] 确认是攻击（而非合法流量激增）
- [ ] 激活 CDN/WAF DDoS 保护（Cloudflare Under Attack Mode、AWS Shield 等）
- [ ] 启用速率限制紧急模式（激进阈值）
- [ ] 在边缘阻止明显的攻击源 IP/范围
- [ ] 如可能扩展基础设施（自动伸缩组）
- [ ] 如攻击来自特定区域，启用地理封锁

### 阶段 2：评估
- [ ] 识别攻击类型（容量型、协议型、应用层）
- [ ] 识别攻击源模式（IP 范围、User-Agent、请求模式）
- [ ] 衡量对服务可用性和用户体验的影响
- [ ] 检查 DDoS 是否是其他攻击的掩护（数据泄露等）
- [ ] 审查资源使用（CPU、内存、带宽、连接）

### 阶段 3：修复
- [ ] 根据攻击模式实施针对性阻止规则
- [ ] 优化应用以处理增加的负载（缓存、静态响应）
- [ ] 如需要，联系 ISP/托管提供商进行上游过滤
- [ ] 将关键服务移至额外保护层后
- [ ] 随着攻击消退逐步放松紧急保护

### 阶段 4：预防
- [ ] 实施带适当阈值的永久速率限制
- [ ] 为所有公共端点部署带 DDoS 保护的 CDN
- [ ] 设置带成本限制的自动伸缩
- [ ] 为常见 DDoS 模式创建 Runbook
- [ ] 为敏感端点实施基于挑战的保护（CAPTCHA）

### 阶段 5：记录
- [ ] 攻击时间线、峰值流量、持续时间
- [ ] 攻击类型和来源特征
- [ ] 服务影响（停机、性能下降、受影响用户）
- [ ] 缓解行动及效果
- [ ] 成本影响（基础设施、收入损失）
- [ ] 提高弹性的建议

## Playbook 3：勒索软件

**严重性：** 严重
**响应时间：** 立即（< 10 分钟隔离）

### 阶段 1：遏制
- [ ] 立即将受影响系统从网络断开（拔网线、禁用 WiFi）
- [ ] 不要关机（保留内存中的取证证据）
- [ ] 识别零号病人（首个已感染的系统）
- [ ] 阻止横向移动（禁用 SMB、网段间的 RDP）
- [ ] 隔离备份系统防止被加密
- [ ] 提醒所有员工断开可疑系统

### 阶段 2：评估
- [ ] 识别勒索软件变种（检查勒索信、文件扩展名）
- [ ] 确定范围：哪些系统和数据被加密
- [ ] 检查备份是否完整且未被攻陷
- [ ] 评估数据在加密前是否被窃取（双重勒索）
- [ ] 检查解密工具（NoMoreRansom.org）
- [ ] 确定入口点（钓鱼邮件、RDP 暴力破解、漏洞软件）

### 阶段 3：修复
- [ ] 如有干净备份：擦除并从备份恢复
- [ ] 如无备份：评估解密选项（公开工具、谈判作为最后手段）
- [ ] 修补遭攻击者利用的漏洞
- [ ] 移除所有持久化机制（计划任务、注册表键、服务）
- [ ] 重连前扫描所有系统查找残留恶意软件
- [ ] 更改所有密码（先改域管理员，然后所有用户）

### 阶段 4：预防
- [ ] 实施网络分段
- [ ] 在所有系统部署 EDR（端点检测与响应）
- [ ] 禁用 SMB v1，限制 RDP 访问
- [ ] 实施 3-2-1 备份策略（3 份副本、2 种介质、1 份异地）
- [ ] 离线或不可变备份存储
- [ ] 定期备份恢复测试
- [ ] 员工钓鱼意识培训

### 阶段 5：记录
- [ ] 完整攻击时间线
- [ ] 入口点和传播方式
- [ ] 数据影响（加密、窃取、丢失）
- [ ] 恢复方法和恢复时间
- [ ] 财务影响（赎金要求、停机成本、恢复成本）
- [ ] 执法报告（推荐）

## Playbook 4：供应链攻陷

**严重性：** 严重
**响应时间：** < 30 分钟评估，< 2 小时遏制

### 阶段 1：遏制
- [ ] 识别受感染的依赖/包/供应商
- [ ] 立即固定到最后已知良好版本
- [ ] 阻止受影响系统到未知 IP 的出站连接
- [ ] 审计所有使用受感染组件的系统
- [ ] 在评估完成前暂停所有部署
- [ ] 检查受感染代码是否在生产环境执行

### 阶段 2：评估
- [ ] 确定恶意代码做什么（数据窃取、后门、挖矿）
- [ ] 识别受影响版本和攻陷时间线
- [ ] 检查包管理器公告（npm、PyPI、Maven 安全公告）
- [ ] 审查首次引入受感染版本时的构建日志
- [ ] 扫描所有使用受感染依赖构建的制品
- [ ] 检查机密/凭证是否暴露给恶意代码

### 阶段 3：修复
- [ ] 更新到修补版本或移除依赖
- [ ] 轮换所有可能被访问的机密
- [ ] 从干净源重新构建并重新部署所有受影响服务
- [ ] 扫描所有系统查找后门或持久化机制
- [ ] 审计构建流水线查找额外攻陷

### 阶段 4：预防
- [ ] 使用锁定文件固定依赖
- [ ] 启用完整性检查（校验和、签名）
- [ ] 设置自动化漏洞扫描（Dependabot、Snyk、pip-audit）
- [ ] 使用带批准包的私有包注册表
- [ ] 实施 SBOM（软件物料清单）
- [ ] 依赖更新代码审查
- [ ] 监控依赖的抢注攻击

### 阶段 5：记录
- [ ] 受感染组件、版本和时间线
- [ ] 影响评估（受影响系统、暴露数据）
- [ ] 检测方法（如何发现）
- [ ] 修复行动和验证
- [ ] 实施的供应链安全改进

## Playbook 5：内部威胁

**严重性：** 高到严重
**响应时间：** < 1 小时（平衡速度与谨慎）

### 阶段 1：遏制
- [ ] 暂不提醒可疑的内部人员
- [ ] 在技术行动前让 HR 和法务介入
- [ ] 增加对可疑账户的监控（审计日志）
- [ ] 在不引起怀疑的情况下限制对最敏感系统的访问
- [ ] 保留所有证据（日志、邮件、文件访问记录）
- [ ] 安全备份证据副本

### 阶段 2：评估
- [ ] 审查访问日志查找异常模式（非工作时间访问、批量下载）
- [ ] 检查未授权数据传输（USB、邮件、云存储）
- [ ] 审查代码更改查找后门或未授权修改
- [ ] 评估内部人员有权访问的数据/系统
- [ ] 确定威胁是恶意的还是疏忽
- [ ] 如有必要，请数字取证介入

### 阶段 3：修复
- [ ] 与 HR/法务协调采取适当行动
- [ ] 采取行动时立即撤销所有访问
- [ ] 更改内部人员可访问的共享凭证
- [ ] 审查并撤销内部人员创建的任何 API 密钥/Token
- [ ] 审计内部人员最近 N 个月的代码更改
- [ ] 检查计划任务、Cron 作业或定时炸弹

### 阶段 4：预防
- [ ] 实施 DLP（数据防泄漏）工具
- [ ] 在整个组织强制执行最小权限访问
- [ ] 定期访问审查（至少每季度）
- [ ] 实施用户行为分析（UBA）
- [ ] 带全面访问撤销的离职检查清单
- [ ] 敏感访问岗位的背景调查

### 阶段 5：记录
- [ ] 完整的内部人员行为时间线
- [ ] 访问或攻陷的数据/系统
- [ ] 收集的证据和证据链
- [ ] 采取的 HR/法务行动
- [ ] 实施的访问控制改进

## Playbook 6：撞库攻击

**严重性：** 高
**响应时间：** < 30 分钟开始缓解

### 阶段 1：遏制
- [ ] 检测攻击（登录失败激增、同一 IP 多账户）
- [ ] 在登录端点启用激进速率限制
- [ ] 在 WAF/CDN 层阻止攻击 IP 范围
- [ ] 在登录表单启用 CAPTCHA
- [ ] 临时锁定多次失败的账户

### 阶段 2：评估
- [ ] 确定有多少账户被成功攻陷
- [ ] 识别凭证列表来源（检查 haveibeenpwned.com）
- [ ] 审查受感染账户是否有未授权行为
- [ ] 检查攻击者是否访问了敏感数据或做了更改
- [ ] 评估财务影响（欺诈交易、数据访问）

### 阶段 3：修复
- [ ] 强制所有受感染账户重置密码
- [ ] 通知受影响用户使用唯一密码
- [ ] 撤销任何未授权行为（交易、设置更改）
- [ ] 阻止已知被攻陷的凭证对
- [ ] 使受影响账户的所有活跃会话失效

### 阶段 4：预防
- [ ] 实施 MFA（多因素认证），推送到所有用户
- [ ] 部署撞库检测（速率 + 模式分析）
- [ ] 注册/更改时对照泄露数据库检查密码
- [ ] 在登录失败时实施渐进延迟
- [ ] 设备指纹和异常检测
- [ ] 认证端点的机器人检测

### 阶段 5：记录
- [ ] 攻击时间线、量和成功率
- [ ] 受感染账户数量和影响
- [ ] 源 IP 分析
- [ ] 检测方法和检测时间
- [ ] 发送的用户沟通
- [ ] 认证安全改进

## Playbook 7：API 滥用

**严重性：** 中到高
**响应时间：** < 1 小时

### 阶段 1：遏制
- [ ] 识别滥用客户端（API 密钥、IP、用户账户）
- [ ] 专门对滥用客户端进行速率限制或节流
- [ ] 如数据爬取：阻止客户端并返回通用错误
- [ ] 如金融滥用：冻结账户待审查
- [ ] 保留请求日志用于分析

### 阶段 2：评估
- [ ] 确定滥用类型（爬取、暴力破解、欺诈、免费层滥用）
- [ ] 量化影响（成本、暴露数据、服务降级）
- [ ] 审查滥用是利用合法 API 还是漏洞
- [ ] 检查 ToS 违规
- [ ] 确定是自动化（机器人）还是手动

### 阶段 3：修复
- [ ] 撤销滥用客户端的 API 密钥
- [ ] 阻止滥用模式（特定端点、请求签名）
- [ ] 如基于漏洞：修补漏洞
- [ ] 如爬取：实施反机器人措施
- [ ] 如欺诈：撤销欺诈交易，报告法务

### 阶段 4：预防
- [ ] 实施带适当层级的按客户端速率限制
- [ ] 添加请求成本跟踪（昂贵端点的加权速率限制）
- [ ] 部署机器人检测（指纹、行为分析）
- [ ] 实施 API 使用配额和计费
- [ ] 添加 API 使用模式异常检测
- [ ] 审查 API 设计的滥用向量（分页、过滤、批量端点）

### 阶段 5：记录
- [ ] 滥用类型、方法和时间线
- [ ] 影响（财务、数据、服务）
- [ ] 客户端识别和证据
- [ ] 采取的行动（阻止、撤销）
- [ ] 实施的 API 安全改进

## 通用沟通模板

对任何事件类型使用此模板：

```
主题：[{严重性}] 安全事件 - {简短描述}

状态：{活跃 / 已遏制 / 已解决}
严重性：{严重 / 高 / 中 / 低}
事件 ID：INC-{YYYY}-{NNN}
检测时间：{时间戳}
事件指挥官：{姓名}

摘要：
{2-3 句话描述发生了什么、受影响内容和当前状态。}

影响：
- 系统：{受影响系统}
- 数据：{受影响数据类型，约数范围}
- 用户：{受影响用户数}
- 业务：{业务影响描述}

当前状态：
- 阶段：{遏制 / 评估 / 修复 / 预防 / 记录}

- 进行中行动：{列表}

下次更新：{下次状态更新的时间戳}
联系人：{事件指挥官联系方式}
```

## 严重性分类参考

| 严重性 | 示例 | 响应时间 | 升级 |
|--------|------|----------|------|
| **严重** | 数据泄露、勒索软件、活跃利用 | < 15 分钟 | 立即：CEO、CTO、法务 |
| **高** | DDoS、撞库、供应链攻陷 | < 30 分钟 | 1 小时内：CTO、工程负责人 |
| **中** | API 滥用、单账户攻陷、非关键漏洞利用 | < 2 小时 | 4 小时内：工程负责人 |
| **低** | 攻击尝试失败、发现轻微配置错误 | < 24 小时 | 下个工作日：团队负责人 |
