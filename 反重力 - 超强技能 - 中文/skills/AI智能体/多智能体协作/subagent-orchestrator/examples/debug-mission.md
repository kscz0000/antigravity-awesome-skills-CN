# 示例：构建故障的修复任务

## 场景
用户："我的 Vercel 部署失败了。构建错误：Cannot find module '@/components/ProposalResult'"

这是一个定向修复——最少的智能体，最少的配额。

---

## 任务简报

```
MISSION BRIEF
─────────────────────────────────────────
Goal: 修复导致 Vercel 构建失败的导入错误。不要触碰其他任何东西。
Total Agents: 1
Quota Strategy: 仅 FLASH
Expected Token Cost: LOW

AGENTS:
[1] ID: repair-001
    Role: 修复智能体
    Scope: 仅包含错误导入的文件 + 缺失文件（如果需要创建）
    Model: Gemini Flash
    Input: 确切的错误消息、导入的文件路径
    Output: 修复的导入或创建的缺失文件——仅此而已
    Depends on: none
─────────────────────────────────────────
```

---

## 修复协议

```
步骤 1：读取确切的错误——不要假设上下文
  Error: "Cannot find module '@/components/ProposalResult'"

步骤 2：检查文件是否存在
  → 如果存在：修复导入路径（大小写问题、错误的别名等）
  → 如果缺失：创建最小可用文件（即使为空但导出正确）

步骤 3：心理验证修复能编译
  → 检查所有导入同一模块的其他文件
  → 确认 tsconfig.json 路径别名正确

步骤 4：报告修改了什么和为什么——各一句话
```

---

## 修复任务中不要做什么

- 不要重新阅读整个代码库来"了解上下文"
- 不要顺手重构旁边的文件
- 不要"为了安全"切换到更强大的模型
- 不要打开浏览器智能体查看 Vercel 面板
- 不要为一个导入错误生成额外的智能体

---

## 配额日志

| 事件 | 影响 |
|-------|--------|
| 1 个 Flash 智能体 | LOW |
| 2–3 个文件读取 | LOW |
| 1–2 个文件写入 | LOW |
| **预估总量** | **< 5% 冲刺** |

> 修复任务几乎永远不应超过 10% 冲刺配额。
> 如果修复变得复杂，停下来——将其分解为新的完整任务。
