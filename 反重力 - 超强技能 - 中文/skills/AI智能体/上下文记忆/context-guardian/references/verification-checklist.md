# 验证与冗余检查清单

## 核心原则

在复杂技术项目中，丢失任何一个细节都可能导致数小时的返工。本检查清单保证压缩前的提取完整且一致。**逐项执行——不可跳过任何一项**。

## 完整性验证

### 文件

```
□ 本会话创建的每个文件都已列出，并附带绝对路径
□ 每个被修改的文件都明确标注了修改的性质
□ 没有遗漏任何文件（核对 Write/Edit/Read 的工具调用）
□ 路径使用与操作系统匹配的分隔符（Windows 用 \）
```

### 决策

```
□ 每项技术决策都有文档化的"为什么"
□ 被放弃的替代方案已记录
□ 没有决策与其他已列出的决策相矛盾
□ 推翻先前决策的决策被明确标注为回退
```

### Bug 与修复

```
□ 每个 Bug 包含：症状、根因、修复方案、受影响文件
□ 修复已经过验证（运行测试、输出已确认）
□ 不存在"部分修复"的 Bug——若未完成，列入待办
□ 安全漏洞已分类标注（SQLi、XSS、token 泄露等）
```

### 任务

```
□ 已完成任务有完成证据（输出、测试、验证）
□ 待办任务有优先级（P0/P1/P2）和依赖关系
□ 没有任何任务"进行中"但没有定义下一步
□ 被阻塞的任务有文档化的阻塞原因
```

### 数字与指标

```
□ 不同章节中提到的数字保持一致
  （例如："18/18 查询"在进度和测试处出现相同数字）
□ 计数器已更新（不要使用上一次迭代的旧数字）
□ 文件大小使用统一的单位
```

### 代码

```
□ 关键代码片段已保留（不依赖记忆）
□ 行号正确（对照当前文件核对）
□ 函数/变量名称准确（无拼写错误）
```

## 交叉一致性验证

完成提取后，进行以下交叉核对：

1. **文件 ↔ 决策**：每次文件修改是否对应一项决策？
2. **Bug ↔ 修复 ↔ 文件**：每个 Bug 是否有修复方案和受影响文件？
3. **任务 ↔ 进度**：已完成任务是否与报告的进度一致？
4. **依赖 ↔ 代码**：关键依赖是否已在代码中体现？

## 三重冗余验证

对于 P0 信息（致命丢失），确认出现在全部 3 个层级：

| P0 信息 | 快照 | MEMORY.md | Context-Agent |
|---------|------|-----------|---------------|
| 决策 X  | □    | □         | □             |
| 修复 Y  | □    | □         | □             |
| 任务 Z  | □    | □         | □             |

若任何 P0 信息少于 2 个层级，先补全再继续。

## 红旗警告（停止并重新提取）

若发现以下任一情况，提取未完成：

- "我觉得我们做了什么关于 X 的事，但记不清具体……"
- 不同章节之间的数字不一致
- 提到文件但未给出完整路径
- 决策没有说明理由（"我们决定用 X"但没有"因为 Y"）
- Bug 修复没有修复方案描述
- 任务"已完成"但没有证据

## 验证后

所有验证通过后：

1. 生成带时间戳的快照
2. 用超紧凑的 P0 信息更新 MEMORY.md
3. 执行 context-agent save
4. 将过渡简报作为最后一条消息写入

过渡简报是最关键的部分——它位于压缩后上下文的顶部，是下一位 Claude 首先看到的内容。
