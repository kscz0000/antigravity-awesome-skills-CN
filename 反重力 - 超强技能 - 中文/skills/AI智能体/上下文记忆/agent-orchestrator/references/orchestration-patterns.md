# 多技能编排模式

协调多个技能完成复杂工作流的详细指南。

---

## 1. 顺序管道

一个技能的输出作为下一个技能的输入。

### 何时使用
- 混合"生产者"技能（data-extraction、government-data、analytics）和"消费者"技能（messaging、social-media、content-management）
- 任务有明确的阶段：收集 -> 处理 -> 交付

### 流程
```
user_query -> Skill A (生产者) -> 数据 -> Skill B (消费者) -> 结果
```

### 具体示例
**请求：** "收集SP拍卖师价格并通过WhatsApp发送"
```
1. junta-leiloeiros: 执行SP爬虫，导出数据
2. whatsapp-cloud-api: 将数据格式化为消息并发送
```

### 上下文规则
- 每个步骤的输出必须作为上下文传递给下一步
- 常见的传递格式：JSON、Markdown表格、摘要文本
- 如果某个步骤失败，中断管道并向用户报告

---

## 2. 并行执行

技能独立处理请求的不同方面。

### 何时使用
- 所有技能角色相同（全部为生产者或全部为消费者）
- 任务的各个方面相互独立
- 技能之间没有数据依赖

### 流程
```
              ┌─> Skill A ─> output A ─┐
user_query ──>├─> Skill B ─> output B ─├──> 聚合结果
              └─> Skill C ─> output C ─┘
```

### 具体示例
**请求：** "在Instagram发布促销信息并通过WhatsApp发送"
```
1. (并行) instagram: 创建并发布促销帖子
1. (并行) whatsapp-cloud-api: 发送促销消息
-> 聚合：报告两个发布的状态
```

### 上下文规则
- 每个技能接收完整的原始查询
- 输出聚合为统一的响应
- 如果某个技能失败，其他技能继续正常执行
- 单独报告每个技能的成功/失败

---

## 3. 主技能 + 支援

一个主技能主导；其他提供辅助数据。

### 何时使用
- 一个技能的相关性得分远高于其他（>= 2倍）
- 主要任务明确，但可以从额外数据中受益
- 支援技能是可选的/"锦上添花"

### 流程
```
user_query -> Skill A (主) ──────────────> 结果
                  ↑
              Skill B (支援) ─> 额外数据
```

### 具体示例
**请求：** "配置WhatsApp聊天机器人以回复拍卖师数据"
```
1. (主) whatsapp-cloud-api: 配置webhook和聊天机器人逻辑
2. (支援) junta-leiloeiros: 提供端点/数据供聊天机器人查询
```

### 上下文规则
- 主技能主导工作流
- 支援技能按需调用
- 如果支援技能失败，主技能应继续（优雅降级）

---

## 错误处理

### 通用规则
1. **单个技能失败**：向用户报告哪个技能失败及原因
2. **管道失败**：中断并显示已完成的步骤
3. **并行部分失败**：继续其他任务，报告失败
4. **技能不完整**：在使用前警告技能状态不完整

### 回退
- 如果某个技能失败，检查是否有其他技能具有类似能力
- 如果没有替代方案，不使用该技能运行并通知用户

---

## 上下文序列化

技能间传递数据的标准格式：

```json
{
  "source_skill": "web-scraper",
  "target_skill": "whatsapp-cloud-api",
  "data_type": "table",
  "data": [
    {"nome": "Joao Silva", "uf": "SP", "registro": "12345"},
    {"nome": "Maria Santos", "uf": "RJ", "registro": "67890"}
  ],
  "metadata": {
    "total_items": 2,
    "collected_at": "2026-02-25T12:00:00",
    "query": "leiloeiros de SP e RJ"
  }
}
```
