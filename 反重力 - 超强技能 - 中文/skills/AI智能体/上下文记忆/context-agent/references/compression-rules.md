# 压缩与归档规则

## 归档时机

满足以下任一条件即为归档候选：
- 距当前会话超过 20 个会话
- 可通过 `config.py` 中的 `ARCHIVE_AFTER_SESSIONS` 配置

## 归档保留内容

保留章节（持久信息）：
- **话题**：始终保留
- **决策**：始终保留（构成知识库）
- **待办任务**：若仍未完成则保留
- **发现**：始终保留
- **已解决的错误**：始终保留（避免重复劳动）

移除章节（临时信息）：
- **指标**：token、计数器（瞬时数据）
- **修改的文件**：长期看不必要的细粒度细节
- **技术债**：通常早已解决

## 归档整合

当 `archive/` 累积 5+ 个独立会话时，将其整合为
单个 `ARCHIVE_YYYY.md`，采用超紧凑格式：

```markdown
# 归档整合 — 2026

### 会话 001 — 2026-01-15
  - 架构相关决策
  - 数据库相关决策

### 会话 002 — 2026-01-20
  - API 相关决策
```

整合时仅保留标题与决策。

## ACTIVE_CONTEXT.md 维护

为保持 150 行上限：

1. **已完成任务**：保存新会话时自动移除
2. **旧决策**：30 天后裁剪（可配置）
3. **近期会话**：仅保留最近 5 个
4. **已解决的阻塞项**：不再被提及时移除
5. **约定**：永久保留（极少变更）

## 漂移检测

系统会校验 `ACTIVE_CONTEXT.md` 与 `MEMORY.md` 是否同步。
若存在分歧（手动编辑、损坏），`maintain` 会自动修复。

## 自动维护流程

```
maintain
  ├── 检查旧会话 → 归档
  ├── 必要时整合归档
  ├── 检测 ACTIVE_CONTEXT ↔ MEMORY.md 漂移 → 同步
  └── 重建 FTS5 搜索索引
```
