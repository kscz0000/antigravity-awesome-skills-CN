# 准备或发布循环

当用户要求分享、提交或发布循环到 Loop Library 时使用此工作流。准备内容与执行外部提交是不同的操作。

## 准备候选项

1. 解析确切的循环及其作者或贡献者归属。不要编造所有权、日期、来源链接、结果或声明。
2. 运行 `SKILL.md` 中设计循环的预检。要求真实的反馈周期、可复现验证、有界动作、明确的停止和需要的批准边界。
3. 按结果、触发器、验证、关键词和相关流程搜索在线目录。当强匹配已存在时优先贡献适配或改进。如果目录不可用，在提交前停止，因为无法检查重复。
4. 准备当前官方提交平台要求的字段，或在 Loop Library 仓库中工作时准备当前模式和示例记录。可用时使用仓库工具验证。保持公开提示紧凑；将可选的长格式指引放在支持的辅助字段中，而非膨胀提示。
5. 展示确切的提议记录、目的地、归属和请求的状态。清楚区分公开建议、仅所有者草稿和公开发布。对于公开建议，还要展示官方平台要求的确切当前所有权和许可声明。

## 要求批准并回读

- 未经明确批准预览，不要发送建议、保存所有者草稿或公开发布。
- 永远不要从通用提交批准设置公开建议的权限或声明字段。要求用户明确确认预览中展示的确切当前所有权和许可条款。
- 仅使用范围内已有的官方提交或认证所有者工具。永远不要暴露所有者凭证或绕过验证。
- 授权所有者操作默认为草稿，除非用户明确批准公开发布。保存草稿的批准不等于批准发布。
- 对于公开建议，将官方平台的成功接受响应视为回执。不要编造标识符或声称仅所有者建议已保存、审核、起草或发布。
- 对于所有者草稿，回读其标识符和草稿状态。对于公开发布，在报告成功前验证在线详情页和目录条目。将验证、授权或必需回读失败报告为受阻。

## 返回预览或回执

批准前，返回：

```markdown
## Loopy 发布预览

目的地：[官方平台]
状态：建议 | 草稿 | 公开
重复检查：清晰 | 可能重叠 | 受阻
声明：[确切当前所有权/许可条款，或不适用]
候选项：[确切记录或简洁的逐字段预览]
```

批准的公开建议后，返回接受响应并仅声明建议已被接收。所有者操作后，返回结果标识符、状态和必需的回读证据。不要将准备好的预览或已接受的建议称为"已发布"。
