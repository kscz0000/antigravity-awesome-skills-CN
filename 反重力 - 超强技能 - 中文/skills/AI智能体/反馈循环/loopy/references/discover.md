# 循环发现

当用户要求从代码库、编码线程历史或两者挖掘应成为循环的工作时使用此工作流。

## 检查证据

1. 从请求中确认最小可发现范围。当当前仓库明确在范围内时检查它。仅对用户授权的编码线程使用可用的线程列表、搜索和阅读工具。如果线程历史不可用，继续使用代码库证据并披露该限制。
2. 在代码中，检查揭示重复工作的操作路径：脚本、CI 和部署配置、维护命令、测试、贡献者指引、issue 模板、运维手册和重复的生命周期模式。仅凭外观相似的函数是重构信号，不是循环的证据。
3. 在线程中，识别已完成的动作及其结果。即使措辞不同也按语义等价工作分组。计算不同出现次数，而非对同一出现的重复讨论。记录紧凑来源标识如线程标题或标识符及所执行的动作；不要复制密钥或不必要的私有内容。
4. 当可行时，对照仓库或运行时验证线程声明。线程历史可能过时、不完整或有误。

## 资格审定和排名候选项

重复的任务不自动成为好的循环。要求候选项遵循 `SKILL.md` 中的反馈周期和验证规则，而非仅仅在代码或线程历史中多次出现。

候选项仅当以下全部存在或可从范围证据推导时才具备循环形态：

- 一个要观察的反复出现的事件或状态；
- 可响应新反馈而改变的下一个动作；
- 动作是否有帮助的可观察检查；
- 有界范围和适当的成功、无操作、受阻、需审批或无进展停止。

要求至少两次不同出现才能将线程推导的任务描述为重复。没有运行历史的代码库模式可报告为潜在循环，但不能报告为已证明的重复。拒绝一次性迁移、直线检查清单、模糊目标和另一次循环不会收到新证据的任务。

按重复证据、时间或失败成本、可用反馈质量、可逆性和安全权限排名合格候选项。不要编造频率、节省的工作量、负责人、调度、指标或权限。优先最小高价值循环而非捆绑无关工作的宽泛循环。

## 转换最佳候选项

1. 使用候选项的结果、触发器、动作和验证术语搜索在线目录。适配强匹配的已发布循环而非重复它。如果目录不可用，继续使用明确未发布的设计并披露无法检查重复。
2. 如果多个候选项强度相似或在权限上实质不同，展示简短排名名单并询问用户转换哪一个。否则直接转换最强候选项。
3. 从证据推导触发器、新观察、有界动作、可复现验证、记录和终止行为。应用 `SKILL.md` 中的每条设计规则；不要因为重复性记录良好就降低标准。仅当缺失决策会实质性地改变安全或成功时问一个简短问题。
4. 运行 `SKILL.md` 中强制的设计循环预检。在交付前修复实质性缺陷，不扩展权限或编造缺失细节。
5. 使用 `SKILL.md` 中的标准交付格式返回紧凑证据和循环。将其标记为未发布设计或适配。如果没有候选项合格，报告干净无操作并解释缺失的反馈或重复证据；不要制造循环。
