# 资源边界规范

本规范定义信息在技能包内部的归属位置。

## 原则

保持主技能足够小，以便清晰路由与执行。一旦内容不再有助于路由或分支选择，就立即从 `SKILL.md` 中移出。

不要为想象中的未来需求添加结构。只有当文件夹、脚本、评测或治理文件能减少当前的歧义、执行负担、路由风险或维护风险时，才应纳入技能包。

## 上下文预算层级

使用最轻、仍能适配技能包的预算。

- `scaffold`：`700` 初始加载 token
- `production`：`1000` 初始加载 token
- `library`：`1300` 初始加载 token
- `governed`：`1300` 初始加载 token

若 `manifest.json` 设置了 `context_budget_tier`，该层级会覆盖由生命周期或成熟度元数据推导出的默认预算。这允许高治理要求的技能保持比单纯依据生命周期标签所暗示的更严格的初始加载预算。

## 放置规则

### 以下内容应放入 `SKILL.md`：

- 触发面的一部分
- 核心执行骨架的一部分
- 输出契约的一部分
- 分支选择或安全默认值所必需

### 以下内容应放入 `references/`：

- 领域指引
- 较长的示例
- 策略材料
- 人类或智能体按需阅读的 schema 或模板

### 以下内容应放入 `scripts/`：

- 确定性的
- 重复性的
- 若以散文重写则易出错
- 相比作为指令，作为代码更易于验证

### 以下内容应放入 `evals/`：

- 该技能被复用到足够程度，路由错误会产生影响
- 容易与邻近技能混淆
- 质量声明应可复现

### 以下内容应放入 `assets/`：

- 技能包包含输出工件、示例或不应膨胀提示上下文的静态文件

## 反模式

避免以下做法：

- 将长篇策略文本直接存放在 `SKILL.md` 中
- 添加 `references/` 但其中没有真正被使用的文件
- 为仍最适合用散文表达的逻辑添加 `scripts/`
- 为一次性或可丢弃的技能添加 `evals/`
- 默认创建每个文件夹，即使为空
- 保留既未被 `SKILL.md` 引用、也未声明为工厂组件的文件夹
- 在真实变体出现之前就添加宽泛的配置开关
- 在没有复用压力的情况下添加治理或报告，让脚手架显得成熟

## 启发式

### `SKILL.md`

- 应保持聚焦
- 不应成为完整的知识库
- 应提及任何对执行有实质影响的可选目录

### `references/`

- 应物尽其用
- 通常应在 `SKILL.md` 中命名并可被发现

### `scripts/`

- 仅当确实存在确定性逻辑或格式化逻辑时才应存在
- 执行需要时应在 `SKILL.md` 中显式引用

### `evals/`

- 当需要为路由或质量声明提供依据时应存在
- 对于可丢弃的个人草稿应跳过

## 未使用资源检测

`resource_boundary_check.py` 在非空的可选目录看起来只是装饰时发出警告：

- 目录存在且包含文件
- 主 `SKILL.md` 未引用该目录
- 且该目录未在 `manifest.json` 的工厂组件中声明

这能防止技能包看起来比实际更复杂。

## 质量密度

检查器还会报告 `quality_density`，这是关于在初始加载预算中打包了多少治理与质量证据的局部信号。

它综合考虑：

- 治理分数
- 是否存在评测
- 是否存在报告
- 引用与脚本
- 接口与清单元数据
- 失败或测试证据

密度越高，意味着技能包在保持精简的同时仍能证明质量。

## 质量意图

最好的技能不是文件最多的那个。最好的技能是仍能令常规任务可靠、可复用、可审计的最小技能包。

关于将资源增长与真实用户目标挂钩的作者与审查者规则，请参阅 [创作规范](authoring-discipline.md)。
