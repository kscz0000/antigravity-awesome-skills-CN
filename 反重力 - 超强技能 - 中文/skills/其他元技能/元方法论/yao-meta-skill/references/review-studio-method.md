# Review Studio 2.0 方法

Review Studio 是技能包的发布面审计界面。它不替代各份详细报告，而是把多份报告聚合成一个面向审核者的决策页。

## 目的

- 在技能包进入深化阶段前，展示发布阻塞项与告警。
- 将每一道关卡反向链接到具体的证据产物。
- 为每个阻塞项与告警生成审查动作，并给出源文件修复位置与验证命令。
- 通过豁免台账，让人工告警接受行为可审计。
- 通过关卡、源/报告路径及可选行号绑定的批注台账，让审核者意见可审计。
- 保持审核流程纵向排列：先摘要，再关卡，最后支撑细节。
- 避免把输出质量、运行时、信任、画像、运营循环等问题分散到不同页面。

## 必过关卡

1. Intent Canvas：意图置信度，以及未解决的输入/输出/排除项缺口。
2. Trigger Lab：路由记分卡、误路由、含糊案例，以及近邻安全。
3. Output Lab：装上技能前后的差值、执行模式、耗时/凭据证据、案例数、文件支撑案例、近邻案例、边界案例、盲测 A/B 审查包证据，以及审核者裁定状态。
4. Context Budget：初始加载、预算层级、告警与质量密度。
5. Runtime Matrix：目标合规通过/未通过，以及降级说明。
6. Trust Report：密钥扫描、脚本面、依赖钉版本、网络/交互标志以及包哈希。
7. Permission Gates：审核者批准的能力范围、理由、有效期、证据以及目标强制执行说明。
8. Runtime Permission Probes：打包后的适配器权限契约、原生强制标志、元数据回退说明以及残余风险。
9. Skill Atlas：路由冲突、过期技能、所有者缺口以及无路由机会。
10. Operations Loop：本地优先仅元数据采用、错失触发、错误输出、脚本错误以及审查漂移信号。
11. Review Waivers：人工风险审批、有效覆盖、过期审批、无效记录以及过期策略。
12. Registry Audit：包元数据、安装证据、兼容性条目以及归档/源校验和。
13. Release Notes：晋升状态、迁移说明、已知缺口以及下一步行动。

## 关卡语义

- `pass`：证据齐全，关卡满足。
- `warn`：审查可继续，但该项问题必须在发布前被看见。
- `block`：在修复前不得声称已具备生产、库、受治理或公开就绪状态。

对于库与受治理技能，Output Lab 至少需要 5 个案例，并覆盖文件支撑、近邻与边界场景。

生产、库与受治理审查还应展示一份盲测 A/B 审查包。当记分卡证据存在但缺少盲测包时，Review Studio 关卡可发出 `warn`，因为该包能证明其主张，但还无法降低审核者偏差。

当 `reports/output_execution_runs.json` 存在时，Review Studio 应展示变体运行数、命令执行运行数、模型执行运行数、已记录夹具数、观察耗时的运行数以及凭据估算的运行数。已记录夹具是有效的可复现性证据，但绝不能被描述为模型执行的输出证据。

当 `reports/output_review_adjudication.json` 存在时，Review Studio 应展示已裁定对与待裁定对。待裁定审核者决策可作为一种显式状态存在，但绝不能被计为已达成一致或人工审查证据。对于生产、库与受治理包，在审核者决策被记录或告警在豁免台账中被显式接受之前，待裁定审核者决策应使 Output Lab 保持 `warn`。无效裁定记录应阻塞发布，因为它们会让盲测审查审计失去信任。

Operations Loop 绝不能展示原始遥测日志。它只能链接到 `reports/adoption_drift_report.md`；任何隐私或模式违规均为阻塞项。

Runtime Permission Probes 关卡在打包后评估，因为它读取生成的目标适配器。缺少探测可在较轻量模式下告警，但受治理发布在没有 `reports/runtime_permission_probes.json` 的情况下不应声称目标权限就绪。

Review Waivers 关卡绝不能把一个阻塞项转成 `pass`。豁免只覆盖告警级风险，要求审核者、理由、范围与过期时间，并只能链接到 `reports/review_waivers.md`。

审查批注不是豁免。批注是审核者意见，附在 Review Studio 关卡与一个相对的源/报告路径及可选行号上。用批注保留审查上下文、请求的修改以及源行备注。打开的阻塞项批注应使 Review Studio 决策为 `blocked`，直到该批注被解决或在附审核者理由的前提下被延期。打开的告警批注可让包进入审查态，但它们不创建关卡专属的 `review_actions`；动作仍保留给非通过关卡。

## 审查动作

每个非通过关卡必须在 `reports/review-studio.json` 中产生一条 `review_actions` 条目。当所有关卡都通过时，`review_actions` 应为空列表，且页面应明确说明不存在阻塞或告警动作。

每条动作必须包含：

- `gate_key`
- `status`
- `summary`
- `why`
- `source_fix`
- `source_refs`
- `evidence`
- `verification_command`

`source_refs` 必须是结构化条目，包含相对的 `path`、人类可读标签、种类、存在标志、最佳努力行号、匹配模式、源短摘录以及在文件存在时的相对链接。它们应指向最小有用的报告或源文件，而不是宽泛的目录。HTML 页面应在链接旁渲染摘录，以便审核者在打开完整产物前理解行锚点的意义。

HTML 页面应在详细支撑段落之前渲染这些动作，让审核者可以从告警直接跳到修复。动作条目不改变关卡数或分数；它们让当前决策更具可操作性。

对于 `world-class-evidence`，动作还应为每个待定证据键暴露一个证据步骤卡片。每张卡片应展示提交路径、模板路径、被阻塞的源检查、命令交接、首批运行手册步骤、溯源要求、成功检查、证据产物以及隐私边界。这些卡片仅作为收集指引，绝不能视为已接受证据，也不能改变世界级就绪状态。

## 审查批注

`reports/review_annotations.json` 是结构化台账，`reports/review_annotations.md` 是人类可读的审查备注面。每条批注应包含：

- `gate_key`
- `target_path`
- 当存在有用源行时的 `line`
- `severity`
- `status`
- `reviewer`
- `body`
- 可选的 `suggested_action`

台账应拒绝绝对路径或逃逸出技能目录的路径。目标文件缺失可作为可见的证据缺口存在，但绝不能作为隐藏的失败。
