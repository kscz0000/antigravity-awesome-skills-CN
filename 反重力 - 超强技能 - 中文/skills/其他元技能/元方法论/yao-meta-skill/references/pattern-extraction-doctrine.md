# 模式提取准则

当一个技能从 GitHub 仓库、产品、论文、专家或用户提供的参考资料中借鉴思路时,使用本准则。目标是提取耐用的模式,而不是复制表面的风格。

## 原则

借鉴的模式必须在可靠性、清晰度或可移植性方面带来的提升,快于其增加的上下文成本。

## 四门模式检验

只有当某个模式通过了与该技能风险层级相匹配的门数时,才予以接受。

### 1. 复现性

该模式在多个真实案例、来源、工作流或使用场景中出现。

用此门拒绝那些看起来很惊艳但没有持久信号的一次性技巧。

### 2. 生成性

该模式能够指导新的案例,而不仅仅是解释原始示例。

用此门更倾向于操作原则、决策规则和工作流循环,而非个例。

### 3. 独特性

该模式比泛泛的好建议更为具体。

用此门拒绝诸如"讲清楚""有用""保证质量"之类的空洞主张,除非参考资料展示了具体做法。

### 4. 边界

该模式具有已知的局限:何时不适用、不应借鉴什么、或会引入何种成本。

用此门防止参考资料扫描演变为无止境的特性扩张。

## 接受规则

- `脚手架级`:当模式具备生成性与边界清晰性时予以接受。
- `生产级`:要求具备复现性、生成性与边界清晰性。
- `库级`:要求具备复现性、生成性、独特性与边界清晰性。
- `受控级`:要求同时满足全部四门,并附带审阅者可见的证据。

当证据不够充分时,将该模式移入迭代候选清单,而不是放入首批交付包。

## 应当借鉴的内容

应当借鉴:

- 高信号的工作流循环
- 有证据支撑的质量门
- 可重复执行的审查检查点
- 简洁的输出形态
- 能防止路径混淆的边界措辞
- 跨环境保留语义的可移植模式

## 不应借鉴的内容

不应借鉴:

- 来源品牌风格
- 长篇散文
- 与目标技能不匹配的角色扮演风格
- 针对低风险脚手架的重量级研究工作流
- 隐藏在通用方法内部的平台特定假设
- 无法验证或泛化的看似惊艳的示例

## 审阅者问题

在决定接受一个借鉴的模式之前,先问:

- 该模式还出现在哪些地方?
- 它能帮助解决什么新案例?
- 它比泛泛建议更具体在哪里?
- 何时本技能应拒绝使用它?
- 哪个文件、报告、评测或检查清单能证明它值得被引入?
