# Prompt Engineering Doctrine（提示工程准则）

当技能涉及创建、改进、审计或依赖提示词、角色指令、对话脚本、写作系统、教学指南、分析说明、可复用任务模板时，遵循本准则。

## 原则

提示词质量是技能设计的输入参数，而不是粘贴进 `SKILL.md` 的一长段提示词。

有效的抽象不是固定的 RTF 模板。有效的抽象是一层紧凑的推理层：

- 理解请求背后的真实需求
- 选择正确的任务类型与复杂度
- 将角色、任务与格式映射到技能结构中
- 在技能被视为可复用之前，对面向提示词的行为进行评分

## 需求模型

编写以提示词为主的技能前，先识别：

- 显式需求：用户明确提出的诉求
- 隐式需求：上下文暗示但用户未明确指出的诉求
- 场景：输出将在何处、如何被使用
- 用户水平：初学者、实践者、专家、评审者或运营人员
- 成功标准：什么能证明输出有效

若上述任一项会改变技能包的边界，就提出一个聚焦的澄清问题。若它们只影响实现细节，则将假设记录在报告中，而非打断用户。

## 任务族

- 创意生成：内容、创意、活动、变体、概念
- 分析推理：诊断、对比、综合、决策支持
- 执行操作：工作流步骤、任务完成、标准化操作
- 教学引导：讲解、课程、走查、辅导
- 对话交互：支持、访谈、角色扮演、探索、辅导
- 提示工程：提示词创建、提示词改进、提示词评审、提示词库

## 复杂度

- 简单：单一输出，少量约束，低歧义
- 中等：多步骤，需要一定判断力，标准适中
- 复杂：多输入，多权衡，对输出质量有高期待
- 专家：领域专业、评估、治理或安全敏感场景

复杂度应决定添加多少结构。它不应成为膨胀入口的借口。

## RTF 到技能的映射

| 提示层 | 技能层 | 评审者问题 |
| --- | --- | --- |
| 角色 | 运营姿态、专业度、语气 | 智能体身份是否匹配岗位与用户水平？ |
| 任务 | 工作流、关卡、脚本、参考 | 步骤是否可执行且可验证？ |
| 格式 | 输出契约、示例、报告 | 交付物是否有用、可读且可测试？ |

## 质量矩阵

从以下维度评估面向提示词的行为：

- 完整性：上下文、约束、输出都已充分说明
- 清晰度：用语无歧义、易于执行
- 一致性：角色、任务、格式、示例与边界相互吻合
- 实用性：输出可在不含隐性假设的情况下直接使用
- 专属性：用语贴合用户所在领域，而非泛化的提示词黑话

把创新视为可选项。可复用技能首先要做到清晰、可靠、专属。

## 反模式

- 将整段元提示词复制进 `SKILL.md`
- 工作流仅需狭窄能力时，却堆砌冗长的人格设定
- 事事都向用户索取，而非仅索取会改变设计的关键字段
- 产出打磨精美的提示词，却缺少测试、示例或输出校验
- 使用 RTF 标签做装饰，未将其与技能行为挂钩

## 评审者规则

评审面向提示词的技能时，评审者应看到需求模型、任务族、复杂度、RTF 到技能的映射以及质量矩阵。若这些缺失，技能包或许仍可运行，但其提示词行为未受治理。
