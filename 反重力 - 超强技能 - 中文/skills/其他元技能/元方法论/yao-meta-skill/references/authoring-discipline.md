# 编写规范

在创建、重构或审查一个技能包时使用此规范。它能保持系统的实用性，又不会让每个请求都变成一个庞大的框架。

## 原则

每一条新增的指令、文件、脚本、评估或治理规则，都必须能追溯回用户真实反复出现的任务。

## 1. 假设规范

不要在一个猜测出来的目标上加深技能包。

- 当用户的请求存在多种合理解读时，陈述当前的工作假设
- 当反复出现的任务、目标输出或排除边界不清晰时，简短地追问一句
- 当存在真实的设计冲突时，把它摆到台面上，而不是默默选择一个有风险的方向
- 仅在决策低风险且可逆时，才静默继续

好的澄清是轻量的。要问那个能改变技能包设计的问题，而不是填一整份接入表。

## 2. 范围规范

构建最小且可靠的技能包。

- 不添加用户没有要求、工作流也不需要的功能
- 在出现真正的差异之前，不添加通用的可配置性
- 不为了看起来完整而添加空文件夹、装饰性报告或宽泛的策略文本
- 一条强执行路径，胜过若干投机性的分支

技能包不会因为文件更多就更好。当反复出现的任务变得更清晰、更安全或更易验证时，它才更好。

## 3. 变更规范

改进现有技能时，要做外科手术式的改动。

- 只触碰直接支持本次请求改动的文件
- 沿用现有的风格和结构，除非它们本身就是问题
- 移除本次改动产生的、未被使用的产物
- 提及无关的死代码或偏离，但除非被要求，否则不要顺手清理

审查检验标准：每一处被改动的行，都应该能说清它服务于哪个用户目标、边界或验证需求。

## 4. 验证规范

把每一项有意义的改动与一项检查绑定起来。

- 触发词变更需要路由证据或邻近触发词证据
- 执行变更需要一份样例输入、脚本检查或手动运行说明
- 面向输出的变更需要输出一份风险画像，并至少配一项自检
- 新增引用需要说明它为何能降低歧义或上下文成本
- 借用的模式需要匹配技能层级的复用性、生成力、差异性或边界证据
- 新增治理需要明确负责人、生命周期预期或审查节奏
- 新增打包或可移植性声明需要具体的目标或兼容性检查

如果某项改动暂时还无法验证，就把它标记为下一步候选，而不是当作基线包的一部分直接发布。

## 审查者清单

在批准一个生成或被修改的技能之前，检查：

- 真实的反复任务是否已明确
- 未解决的假设是否已被命名或澄清
- 技能包没有超出任务所需
- 改动仅限于本次请求的范围
- 每一项新产物都有验证理由
- 借用的模式具有审查者可见的验收证据
- 在示例被批准之前，可能出现的输出错误是否已被命名
- 下一轮迭代的方向是聚焦的，而不是一束投机性的升级

## 失败模式

将以下情况视为编写失败：

- 为一次性回答创建一个技能
- 文字足够时却引入脚本
- 尚不存在路由风险时就加入评估
- 给没有复用压力的个人脚手架加治理
- 因为看起来相关就改动兄弟文件
- 给出建议却不点明其背后的假设
