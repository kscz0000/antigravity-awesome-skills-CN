# 技能图谱方法

技能图谱是承载大量技能的工作区的 2.0 运营层。

## 目的

单一技能的质量对于团队技能库而言并不足够。一个技能组合还需要揭示路由冲突、过时的归属、重复的资源以及反复出现的无路由机会。

## V0 检查项

- 枚举工作区下的每一个 `SKILL.md`。
- 提取名称、描述、归属、成熟度、目标对象、更新时间与评审节奏。
- 将相似描述检测为路由重叠候选。
- 检测重复的技能名称。
- 检测共享的脚本/参考文件名作为依赖信号。
- 标记缺失的归属或评审元数据。
- 基于 `updated_at` 与 `review_cadence` 标记过时技能。
- 从失败备注中提取无路由机会。
- 阅读聚合采用率漂移报告并标记遥测漂移，但不读取原始遥测日志。

## 范围策略

图谱保留完整目录，但发布门控应区分可执行的库技能与示例及测试夹具。

当路径前缀被刻意保留为示例、演化快照、嵌入式生成技能或校验器夹具时，使用 `skill_atlas/policy.json` 将其标记为不可执行。不可执行项仍会出现在完整报告、路由矩阵、过期列表与归属缺口列表中，但 Review Studio 应使用可执行项计数来评估发布就绪度。

## 遥测关联

图谱可读取每个技能的聚合 `reports/adoption_drift_report.json`，以浮出组合漂移信号，例如生产/库/受管技能无遥测、触发未命中、输出异常、资源缺失、脚本错误以及评审逾期计数。它不得读取或打包 `reports/telemetry_events.jsonl`；原始遥测仍是归属于该技能的本地专属证据。

将漂移输出写入 `skill_atlas/drift_signals.json`。不可执行的范围在该文件与 HTML 报告中保持可见，但只有可执行的漂移信号才应影响发布就绪度。

## 评审门控

在将单一技能提升至共享库之前，请先使用图谱。若出现可执行的路由冲突、归属缺失、过时的受管技能或遥测漂移信号，应先修正组合边界，再为某个技能叠加更多本地复杂度。不可执行问题应作为证据保持可见，而非作为发布阻塞项。
