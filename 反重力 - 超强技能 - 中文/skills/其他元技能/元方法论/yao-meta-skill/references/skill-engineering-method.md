# 技能工程方法

本文档定义了将混乱的工作流素材转化为可复用技能的默认方法，同时避免入口点膨胀。

## 核心循环

1. 先判断该需求是否真的应该成为一个技能。
2. 跑一轮简短的意图对话，捕捉真实任务、产出物、排除项和约束。
3. 选择最小可行的原型。
4. 设定一个清晰的能力边界。
5. 在扩展正文之前，先撰写并测试触发描述。
6. 套用编写纪律：标出未解决的假设、保持范围紧凑、让有意义的改动挂钩校验。
7. 为面向用户的产物添加输出风险画像。
8. 为报告、教程、可视化查看器、仪表盘、截图和可视化页面添加产物设计画像。
9. 只添加与风险匹配的门控。
10. 发布第一个可路由的包，再挑出三个最有价值的下一轮迭代方向。
11. 包装与治理技能时，范围仅与真实复用需求匹配。

## 阶段 1：资格筛选

只有满足以下任一条件时，才把需求提升为技能：

- 工作流会被重复使用
- 工作流容易被错误路由
- 确定性脚本能减少重复劳动
- 治理或可移植性很重要

以下情况应拒绝创建技能：

- 仅是解释
- 仅是总结
- 仅是翻译
- 头脑风暴
- 不带智能体执行的文档工作
- 一次性回答，无复用价值

参见 [非技能决策树](non-skill-decision-tree.md)。

## 阶段 1.5：编写纪律

在扩展包之前，先落实让工作保持扎实的执行纪律。

- 只澄清那些会改变包设计的假设
- 不要添加投机性功能、泛化可配置项或装饰性结构
- 编辑现有技能时，只触碰直接服务于本次改动的文件
- 让每一处有意义的改动都挂钩校验：路由证据、样本运行、资源边界检查、治理检查或评审员备注

参见 [编写纪律](authoring-discipline.md)。

## 阶段 1.6：问题诊断

当用户给出模糊的痛点而非明确的技能需求时，应先诊断可能的包形态，再去追问结构。

- 推断该需求最适合由脚手架、生产工作流、库能力、受治理资产，或干脆不创建技能来满足
- 最多推荐三个候选方向
- 解释每个候选为何合适、局限在哪、轻量首版应验证什么
- 意图足够清晰时，给出具体推荐而非备选菜单

这能让"能描述问题但讲不清技能架构"的用户也享受到有用的发现。

## 阶段 2：意图对话

在深入编写之前，只问那些会改变包设计的问题。

- 以平等、引导式的口吻开场，不要冷冰冰地扔出字段清单
- 让用户先自然作答；仅在必要时提供一个极简模板作为可选用快捷方式
- 该技能应承担什么样的重复性任务
- 用户会向它提供哪些真实输入
- 它必须产出哪些输出
- 哪些邻近需求应排除在范围之外
- 用户是否提供值得借鉴的参考系统、仓库或产品
- 哪些约束是重要的：隐私、命名、可移植性、治理或本地适配

参见 [意图对话](intent-dialogue.md)。

## 阶段 3：原型选择

选择能匹配任务的最小原型。

- `Scaffold`：探索性、个人性或短生命周期
- `Production`：团队复用、重视质量，但仍保持紧凑
- `Library`：广泛复用、可见证据、可移植性与维护预期
- `Governed`：组织敏感或运营关键；生命周期与评审显式

参见 [技能原型](skill-archetypes.md)。

## 阶段 4：边界设计

每个技能都应清晰地回答四个问题：

- 它承担什么样的重复性任务
- 它产出什么输出
- 哪些邻近需求不应路由到这里
- 哪些细节应放在 `SKILL.md` 之外

边界工作先于润色文笔。

## 阶段 5：参考扫描

在深入编写之前，先做一轮简短的基线扫描。

- 最多扫描 `3-5` 个参考对象
- 优先考虑高 star 的外部 GitHub 与官方基线来源
- 第二顺位是用户提供的参考，但只抽取其中的模式与标准
- 第三顺位是本地文件，仅用于适配性、隐私与兼容性校准
- 从方法、结构、执行、可移植性与领域模式中挑选
- 只抽取通过模式测试的内容：复发性、生成性、独特性、边界清晰度
- 记录"不借鉴什么"，让新技能保持轻量

参见 [参考扫描策略](reference-scan.md) 与 [模式抽取准则](pattern-extraction-doctrine.md)。

## 阶段 5.5：输出风险画像

在把包视为可用之前，先预测其最终面向用户输出中最可能出现的失误。

- 教程类技能应防范：泛化标题、模糊示例、缺失成功校验
- 报告与 Markdown 类技能应防范：薄弱表格、堆砌列表、层级混乱
- 截图或可视化类技能应防范：捕获错误、资产缺失、虚构视觉证据
- 研究或重引证类技能应防范：脚注堆叠、论断缺乏支撑
- 代码或命令类技能应防范：隐含的 cwd、输入、输出与副作用假设

生成 `reports/output-risk-profile.md` 并在添加更多结构前交给评审员。

参见 [输出质量风险](output-quality-risk.md)。

## 阶段 5.6：产物设计画像

在批准生成的报告或可视化输出之前，先定义产物应有的可读性与可扫读性。

- 选择产物族系：教程、报告、评审查看器、仪表盘、视觉证据、类幻灯片叙事，或代码/CLI 指南
- 让内容挑选视觉系统，而非套用固定的内部风格
- 借鉴 Kami 的文档纪律：按文档类型路由、提炼内容、校验布局关键假设
- 借鉴演示技能的幻灯片纪律：在写 HTML 前规划层级、密度、节奏与质量门
- 拒绝泛化标题、嘈杂引证、薄弱表格、错位截图、重复卡片栅格与装饰性的视觉默认值

生成 `reports/artifact-design-profile.md` 并在概览与评审查看器中展示。

参见 [产物设计准则](artifact-design-doctrine.md) 与 [输出视觉质量](output-visual-quality.md)。

## 阶段 5.7：提示词质量画像

当技能依赖提示行为、角色设计、对话质量或输出契约时，应将提示方法论转化为可证据化的画像。

- 识别显性需求、隐性需求、场景、用户水平与成功标准
- 把 Role、Task、Format 映射进技能行为，而不是把整段元提示塞进 `SKILL.md`
- 在添加结构前，先分类任务族系与复杂度
- 对完整性、清晰度、一致性、实用性、具体性进行评分
- 把完整推理展示给评审员，对用户侧的流程仍保持"以推荐为先"

生成 `reports/prompt-quality-profile.md` 并在概览与评审查看器中展示。

参见 [提示工程准则](prompt-engineering-doctrine.md)。

## 阶段 6：触发优先的编写

在扩展正文之前，先编写 frontmatter 中的 `description`。

- 以重复性任务为切入点
- 列出应路由到这里的触发动作
- 当容易混淆时，加入排除项
- 在扩展文件树之前先测试路由

触发质量通过以下手段改进：

- `trigger_eval.py`
- `optimize_description.py`
- 盲样本留出
- 评审员加持的盲样本留出
- 对抗性留出
- 路由混淆

## 阶段 7：门控选择

按风险选门控，不按习惯。

- 低风险脚手架：校验结构与上下文体积
- 生产类技能：触发评估加资源边界检查
- 库类技能：描述优化、路由混淆、打包检查
- 受治理类技能：治理评分、生命周期元数据、回归历史

参见 [门控选择](gate-selection.md)。

## 阶段 8：首轮迭代哲学

首版包是"可路由的基线"，而不是最终答案。

- 在扩充文笔前先改进触发与排除项
- 先加一个执行资产，再去补一堆文档
- 给出三个最有价值的下一步，避免作者四面出击
- 优先选用"以更少上下文换更高可靠性"的最小一步
- 把不可验证的想法移到下一步候选，不要把它作为基线结构发出

参见 [迭代哲学](iteration-philosophy.md) 与 [编写纪律](authoring-discipline.md)。

## 阶段 9：发布晋升

候选路由或包只有在以下条件都满足时才能晋升：

- 可见留出没有回退
- 盲样本留出没有回退
- 评审员加持的盲样本留出没有回退
- 对抗性留出没有回退
- 路由混淆保持干净
- 上下文与治理门控仍然通过

参见 [晋升策略](../evals/promotion_policy.md)。

## 设计原则

该方法只在"严谨性增长快于上下文成本"时才正确。如果新增一项检查或文档让技能更重、却没让它更可靠，那就移除或外迁。
