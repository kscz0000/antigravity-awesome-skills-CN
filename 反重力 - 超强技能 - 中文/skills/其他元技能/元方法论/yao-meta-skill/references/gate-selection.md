# 关卡选择

关卡选择应当与风险挂钩。并不是关卡越多就越好。

## 按风险划分的最低关卡

| 风险等级 | 典型原型 | 必须的关卡 |
| --- | --- | --- |
| 低 | 脚手架 | `validate_skill.py`、`resource_boundary_check.py` |
| 中 | 生产 | 低风险关卡，外加 `trigger_eval.py` |
| 高 | 库 | 中风险关卡，外加 `optimize_description.py`、路由混淆、打包验证 |
| 关键 | 受治理 | 高风险关卡，外加治理打分、回归历史、晋升策略 |

## 触发器关卡

在以下情况下使用触发器关卡：

- 某个路由可能与邻近技能混淆
- 该工作流会被多人复用
- 必须长期守住技能边界

触发器关卡，按顺序：

1. `trigger_eval.py`
2. `optimize_description.py`
3. 盲测留出
4. 由评审支持的盲测留出
5. 对抗性留出
6. 路由混淆

## 打包关卡

在以下情况下使用打包关卡：

- 输出必须被多个消费方使用
- 元数据漂移会破坏可发现性
- 可移植性是技能价值的一部分

打包关卡：

- `validate_skill.py`
- `cross_packager.py`
- 打包器失败夹具
- 适配器快照检查

## 治理关卡

在以下情况下使用治理关卡：

- 该技能是共享基础设施
- 该技能有所有者和审查节奏
- 生命周期状态至关重要

治理关卡：

- `governance_check.py`
- 回归历史
- 成熟度打分
- 晋升策略

## 上下文关卡

在每个非平凡的技能上都使用上下文关卡。

- `context_sizer.py`
- `resource_boundary_check.py`

如果严格度提高了，但初始加载超过了层级预算，那么应当把详细内容迁入 `references/`、`scripts/` 或 `reports/`，再增加更多检查。
