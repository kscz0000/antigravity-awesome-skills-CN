# SkillOps 决策策略

在将明确来源的会话证据转化为 SkillOps 机会、提案或发布工作时，请使用本策略。目标是让反复出现的用户信号变得可执行，同时不让自动化在未经审查的情况下写入持久化指令、技能、脚本或评估。

## 决策顺序

1. 将信号归类为：无操作、仅报告、Memory、AGENTS.md、现有技能补丁、候选技能、脚本、评估、报告、合并、拆分或归档。
2. 优先选择能修复反复出现摩擦的最小持久化载体。
3. 每一次写入操作都必须有证据支撑。
4. 任何源文件写入之前，必须先有提案或审批账本条目。
5. 每项拟议变更都必须映射到至少一条验证命令。

## 分数区间

SkillOps 机会使用 `0-100` 分数。分数仅供参考，绝不能绕过审批。

| 分数 | 决策 |
| ---: | --- |
| `85-100` | 准备进入审批评审 |
| `70-84` | 提案评审 |
| `50-69` | 继续观察证据 |
| `0-49` | 仅报告或不操作 |

高风险项目即便分数较高，也仍仅以提案形式存在。

## 动作映射

| 模式 | 默认动作 | 持久化载体 |
| --- | --- | --- |
| `language_default` | 补丁现有技能 | 报告模板或制品规范 |
| `report_ui` | 补丁现有技能 | 报告渲染器、制品规范、可视化测试 |
| `approval_safety` | 更新 AGENTS | 治理指南或审批策略 |
| `delivery_format` | 补丁现有技能 | CLI 输出、README、生成的摘要文案 |
| `evidence_testing` | 新增评估 | 聚焦回归、报告质量或发布门禁 |
| 未知模式 | 仅报告 | 人工复核队列 |

## 安全规则

- 不要隐式扫描私有日志；只能使用用户明确提供的来源。
- 不要在报告中存储原始会话内容；使用脱敏摘录与聚合计数。
- 不要在每日报告运行中写入源文件。
- 不要将 SkillOps 报告视为公开的世界级证据。
- 不要将计划中的工作、待提交的草稿或生成的提案视为已接受的证据。
