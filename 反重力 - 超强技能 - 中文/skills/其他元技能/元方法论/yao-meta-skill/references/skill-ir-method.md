# 技能 IR 方法

技能 IR（Intermediate Representation，中间表示）是 2.0 层，用于将持久的技能语义与平台封装分离开来。

## 用途

在面向生产、库、治理化或团队分发等场景进行平台特定封装之前，使用技能 IR。即便 OpenAI、Claude、Agent Skills、VS Code 或通用适配器在目录布局、元数据名称或激活行为上有所不同，IR 都应保留能力契约。

## 应纳入 IR 的内容

- 技能所负责的重复性任务
- frontmatter 中的触发描述
- 应触发、不应触发，以及近邻边界场景
- 工作流步骤、决策点与失败模式
- 包所使用的 references、scripts、assets 与 reports
- 触发与输出的评估计划
- 输出风险、执行风险与信任边界
- 负责人、成熟度、评审节奏与目标平台

## 不应纳入 IR 的内容

- 仅在封装之后才存在的平台特定文件路径
- 从外部基准复制过来的散文段落
- 客户端特定的 UI 标签
- 不属于技能包本体的本地私有路径
- 并非被要求或经过测试的投机性适配器

## 编写规则

在新增适配器、编译器、注册中心或一致性检查之前，先导出或更新技能 IR。如果某个字段无法从本地包的证据推导出来，应将其留空或使用显式的低置信度默认值，而不是凭空捏造细节。

## 评审者关卡

评审者应能回答以下问题：

1. 这个技能负责什么？
2. 何时应触发？
3. 何时不应触发？
4. 哪些 resources 与 scripts 承载了真实行为？
5. 哪些 evals 证明了契约？
6. 哪些目标平台可以在不损失语义的前提下消费该技能？
