# 参考扫描策略

在进入深度写作之前先做一轮简短的基准扫描。目的是从优秀的参考对象中借鉴可复用的模式，而不是把它们的措辞、权重或品牌语言引入新技能。

## 来源优先级

参考扫描有两个层次，绝不能同等对待：

1. **外部基准扫描**
   - 模式的主要来源
   - 使用高星标的公开 GitHub 仓库、官方文档、强公开示例和世界级产品
   - 这一层定义了质量上限
2. **用户参考输入**
   - 品味与标准层
   - 询问用户是否拥有欣赏的参考仓库、产品、页面、提示词或系统
   - 仅借鉴模式、结构、边界或质量标准；绝不复制措辞或私有素材
3. **本地适配校验**
   - 次级校准层
   - 仅将本地文件用于命名、隐私、兼容性、迁移和库适配约束
   - 这一层不应定义主要的设计模式

外部来源应当主导。用户参考应当明确方向。本地文件应当承担校准作用。

## 默认可见性

参考综合默认应当静默进行。

- 在后台完成基准扫描和模式综合
- 将结果转换为给用户的建议
- 仅向作者和评审者展示完整证据
- 仅在意图仍不确定或存在需要决策的真实设计冲突时，才向用户暴露底层权衡

## 这一步存在的理由

新技能常常失败，因为它从一个孤立的想法出发，而不是从一个经过验证的模式出发。控制良好的参考扫描能在技能成长之前就改进它：

- 更清晰的边界设计
- 更整洁的目录与元数据选择
- 更现实的质量门控
- 更强的可移植性决策
- 与用户自身品味和质量标准的更好对齐

## 规则

以下情况必须执行参考扫描：

- `Production` 类技能
- `Library` 类技能
- `Governed` 类技能
- meta-skill 或重打包的技能

以下情况参考扫描为可选：

- `Scaffold` 类技能
- 一个人探索性的技能

## 范围限制

不要把它变成开放式研究。

- 最多扫描 `3-5` 个参考对象
- 选取的类别不超过 `3` 个
- 提取模式，而非大段复制的内容
- 借鉴计划一旦清晰就立即停止
- 在将扫描视为完成前，至少要有 `2` 个外部基准对象
- 如果用户提供了参考，记录他们欣赏什么，以及明确不应复制什么

## 模式接受标准

借鉴的想法在影响首个包之前，必须通过一次轻量级的模式检验。

- `recurrence`（复现性）：模式在多个严肃的语境或来源中出现
- `generativity`（生成性）：模式能指导新案例，而不只是解释一个例子
- `distinctiveness`（差异性）：模式比泛泛的良好建议更具体
- `boundary`（边界性）：模式指明不适用的场景或它带来的代价

对于低风险的脚手架工作，生成性加上边界清晰度就够了。对于 Production、Library 和 Governed 工作，则需要在模式改变包形态之前收集更多证据。

参见 [模式提取原则](pattern-extraction-doctrine.md)。

## 参考类别

选择最小且相关的集合：

- `method`：循环、评估纪律、迭代结构
- `structure`：包结构、资源边界、元数据模式
- `execution`：操作流程、脚本、初始化与校验体验
- `portability`：中立元数据、适配器、降级策略
- `domain`：同一问题空间中顶级示例的工作流特定模式

## 输出格式

一次良好的扫描产出一份简短的报告，包含：

1. 当前技能锚点
2. 扫描焦点
3. 外部基准对象
4. 用户提供的参考
5. 本地适配约束
6. 可借鉴的内容
7. 不可借鉴的内容
8. 一份精简的借鉴计划

## 可借鉴的内容

借鉴：

- 可重复的循环
- 清晰的边界模式
- 经过验证的门控选择
- 可移植的元数据思路
- 清晰的面向操作的流程
- 高信号示例，展示最终体验而非仅展示内部方法

不要借鉴：

- 来源特有的品牌表达
- 大段复制的措辞
- 不必要的目录结构
- 超出技能风险等级的质量门控
- 伪装成最佳实践的平台锁定
- 比公开顶级基准更弱的本地历史习惯

## 设计原则

只有当扫描提升技能质量的速度快于提升上下文成本的速度时，它才算成功。如果基准素材使新技能变重，却没有让它更可靠，就丢弃它。
