# 编排批量重构

## 概述

用于安全执行高通量重构的技能。并行分析范围，综合为统一计划，然后用子代理执行独立的工作包。

## 适用场景
- 重构涉及多个文件或子系统，需要清晰的工作划分。
- 在并行实现之前需要依赖感知的规划。

## 输入

- 仓库路径和目标范围（路径、模块或功能区域）
- 目标类型：重构、重写或混合
- 约束条件：行为对等、API 稳定性、截止日期、测试要求

## 何时使用并行化

- 中大型范围涉及多个文件或子系统时使用此技能。
- 小规模编辑或高度耦合的单文件工作跳过多代理执行。

## 核心工作流

1. 定义范围和成功标准。
   - 列出目标路径/模块和非目标。
   - 说明行为约束（例如：保持外部行为不变）。
2. 先进行并行分析。
   - 将目标范围拆分为分析通道。
   - 并行生成 `explorer` 子代理分析每个通道。
   - 向每个代理获取：意图图、耦合风险、候选工作包、所需验证。
3. 构建一份依赖感知的计划。
   - 将 explorer 输出合并为单一工作图。
   - 创建工作包，明确文件归属和验证命令。
   - 按依赖级别排列工作包顺序；仅并行运行独立的工作包。
4. 使用 worker 代理执行。
   - 为每个独立的工作包生成一个 `worker`。
   - 分配明确的归属（文件/职责）。
   - 告知每个 worker 它们并非独占代码库，必须忽略无关的编辑。
5. 集成与验证。
   - 审查工作包输出，解决重叠，运行验证关卡。
   - 对每个工作包运行定向测试，然后对集成范围运行更广泛的测试套件。
6. 报告与收尾。
   - 总结工作包成果、关键重构、已解决的冲突和残余风险。

## 工作包规则

- 每个执行波次中，每个文件只有一个归属者。
- 重叠文件集不得并行编辑。
- 工作包目标应保持狭窄且可衡量。
- 包含明确的完成标准和必要的检查项。
- 除非用户明确要求行为变更，否则优先采用保持行为的重构。

## 规划契约

每个工作包必须包含：

1. 工作包 ID 和目标。
2. 归属文件。
3. 依赖项（无或工作包 ID）。
4. 需保持的风险和不变量。
5. 必要的检查项。
6. 主线程的集成说明。

使用 [`references/work-packet-template.md`](references/work-packet-template.md) 获取确切格式。

## 代理提示词契约

- 使用 [`references/agent-prompt-templates.md`](references/agent-prompt-templates.md) 中的提示词模板。
- Explorer 提示词侧重于分析和拆解。
- Worker 提示词侧重于实现和验证，具有严格的归属边界。

## 安全护栏

- 计划综合完成前不得开始 worker 执行。
- 不得跨未解决的依赖进行并行化。
- 任何必要的工作包检查失败时不得声称完成。
- 工作包边界导致反复合并冲突时，停下来重新规划。

## 验证策略

按以下顺序执行：

1. 工作包级检查（快速且有范围限制）。
2. 跨工作包集成检查。
3. 范围较广时进行全项目安全检查。

优先使用快速反馈循环，但绝不能跳过必要的行为检查。

## 局限性
- 仅在任务明确匹配上述范围时使用此技能。
- 输出结果不能替代针对特定环境的验证、测试或专家评审。
- 如果缺少必要的输入、权限、安全边界或成功标准，请停下来请求澄清。
