# 精炼与评估标准

在阶段二（评估与收敛）期间使用此量表来压力测试创意方向。并非每条标准都适用于每个创意——根据具体上下文判断哪些维度最重要。

## 核心评估维度

### 1. 用户价值

最重要的维度。如果价值不清晰，其他都不重要。

**止痛药 vs. 维生素：**
- **止痛药：** 解决急迫、频繁的问题。用户会主动寻找。他们会从现有方案切换。信号：人们带着情绪描述问题、已经搭建了变通方案、愿意为解决方案付费。
- **维生素：** 锦上添花。让某件事稍微好一点。用户不会专门去找。信号：人们礼貌地点头，说"挺酷的"，然后不改变行为。

**要问的问题：**
- 你能说出 3 个现在就有这个问题的具体的人吗？
- 他们现在在用什么替代？（真正的竞争者始终是当前的变通方案。）
- 他们会从现有方式切换吗？什么能让他们切换？
- 他们多频繁遇到这个问题？（日常问题 > 月度问题）
- 这是一个"拉力"问题（用户在要求这个）还是"推力"问题（你觉得他们应该想要这个）？

**红旗信号：**
- "每个人都能用这个"——如果你说不出一个具体用户，价值就不清晰
- "就是比 X 更好"——边际改进很少驱动采用
- 问题真实但罕见——高频率但低频次的产品很少合理

### 2. 可行性

你真的能构建这个吗？不只是技术上，而是实际上。

**技术可行性：**
- 核心技术是否存在且可靠运行？
- 最难的技术问题是什么？是已知的难题还是全新的？
- 是否依赖你无法控制的第三方、API 或数据源？
- 最小技术栈需要什么？（如果答案是"很多"，那就是信号。）

**资源可行性：**
- 构建 MVP 的最小团队/投入是多少？
- 是否需要你目前没有的专业知识？
- 是否有监管、法律或合规要求？

**价值实现时间：**
- 多快能把东西放到用户面前？
- 是否有一个能在数天/数周（而非数月）内交付价值的版本？
- 关键路径是什么？什么必须先完成？

**红旗信号：**
- "我们只需要先解决 [非常难的研究问题]"
- 多个依赖需要同时运作
- MVP 仍需数月工作——可能不够最小化

### 3. 差异化

是什么让它真正不同？不是更好——是*不同*。

**要问的问题：**
- 如果用户向朋友描述这个，他们会说什么？那个描述有吸引力吗？
- 这个做到的哪一件事是其他任何东西都做不到的？（如果你说不出一个，那就有问题。）
- 这个差异化是否可持续？竞争者能在一周内复制吗？
- 差异是用户真正在乎的，还是只是构建者觉得有趣的？

**差异化类型（从强到弱）：**
1. **新能力：** 做到以前不可能的事
2. **10 倍提升：** 在关键维度上好到改变行为
3. **新受众：** 将现有能力带给被排除在外的人
4. **新场景：** 在现有方案失败的场景中工作
5. **更好的体验：** 同样能力，体验大幅简化
6. **更便宜：** 同样的东西，更低成本（最弱——容易被竞争抹平）

**红旗信号：**
- 差异化完全在于技术，而非用户体验
- "我们更快/更便宜/更好看"而没有结构性的原因
- 差异化的特性不是用户最在乎的特性

## 假设审计

对每个创意方向，在三个类别中明确列出假设：

### 必须为真（致命假设）
如果错误就会彻底扼杀创意的假设。这些需要在构建前验证。

示例："用户会与我们分享数据"——如果他们不愿意，整个产品就不能运作。

### 应该为真（重要假设）
显著影响成功但不会扼杀创意的假设。如果这些错误，你可以调整方法。

示例："用户偏好自助服务而非与人沟通"——如果错误，你需要不同的市场策略，但核心产品仍可运作。

### 可能为真（锦上添花假设）
关于次要功能或优化的假设。在核心验证前不要验证这些。

示例："用户会想与队友分享结果"——增长功能，非核心价值主张。

## 决策框架

在方向之间选择时，按此矩阵排序：

|                    | 高可行性 | 低可行性 |
|--------------------|-------------------|-----------------|
| **高价值**     | 先做这个     | 值得冒险   |
| **低价值**      | 仅当很轻松   | 别做    |

然后用差异化作为同一象限内选项的决胜因素。

## MVP 范围界定原则

为选定方向定义 MVP 范围时：

1. **一个任务，做到极致。** MVP 应该精准搞定一个用户任务。不是三个任务各做一半。
2. **先验证最冒险的假设。** MVP 的主要目的是测试最可能出错的假设。
3. **按时间框定，而非功能清单。**"我们在 [时间范围] 内能构建和测试什么？"比"我们需要什么功能？"更好。
4. **"不做清单"是必须的。** 明确说出你砍掉什么以及为什么。这防止范围蔓延并迫使诚实的优先级排序。
5. **如果不觉得尴尬，你等太久了。** 第一版对构建者来说应该感觉不完整。如果不是，你就过度构建了。
