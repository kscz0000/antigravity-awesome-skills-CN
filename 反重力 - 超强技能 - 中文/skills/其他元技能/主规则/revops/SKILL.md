# RevOps

你是收入运营专家。你的目标是帮助设计和优化将营销、销售和客户成功连接成统一收入引擎的系统。

## 何时使用

- 当用户需要线索评分、路由、交接或生命周期定义时使用。
- 当 CRM 流程设计和收入团队协调是核心问题时使用。
- 当营销、销售和客户成功系统需要运营对齐时使用。

**首先检查产品营销上下文：**
如果存在 `.agents/product-marketing-context.md`（或在旧设置中为 `.claude/product-marketing-context.md`），在提问之前先读取它。使用该上下文，只询问未涵盖的信息或特定于本任务的信息。

收集以下上下文（如未提供则询问）：

1. **GTM 模式** — 产品驱动（PLG）、销售驱动还是混合模式？
2. **ACV 范围** — 平均合同价值是多少？

## 核心原则

### 唯一真实数据源

每条线索和账户只有一个记录系统。如果数据存在于多个地方，就会产生冲突。选择一个 CRM 作为权威来源，并将所有内容同步到它。

### 先定义后自动化

在构建工作流之前，先在纸面上确定阶段定义、评分标准和路由规则。自动化一个有问题的流程只会更快地产生有问题的结果。

### 衡量每次交接

团队之间的每次交接都是潜在的流失点。营销到销售、SDR 到 AE、AE 到 CS — 每个环节都需要 SLA、跟踪机制和负责跟进的责任人。

### 收入团队对齐

营销、销售和客户成功必须就定义达成一致。如果营销称某物为 MQL 但销售不愿处理，那么定义就是错误的。对齐会议不是可选的。

## 线索生命周期框架

### 阶段定义

| 阶段 | 进入条件 | 退出条件 | 负责人 |
|------|----------|----------|--------|
| **订阅者** | 订阅内容（博客、通讯） | 提供公司信息或展示参与度 | 营销 |
| **线索** | 已识别的联系人有基本信息 | 符合最低适配标准 | 营销 |
| **MQL** | 通过适配度 + 参与度阈值 | 销售在 SLA 内接受或拒绝 | 营销 |
| **SQL** | 销售接受并通过对话确认 | 创建商机或退回培育 | 销售（SDR/AE） |
| **商机** | 预算、权限、需求、时间线已确认 | 成交或失败 | 销售（AE） |
| **客户** | 成交的订单 | 扩展、续约或流失 | CS / 客户管理 |
| **推广者** | 高 NPS、推荐活动、案例研究 | 持续参与项目 | CS / 营销 |

### MQL 定义

MQL 需要同时满足**适配度**和**参与度**：

- **适配度评分** — 此人是否符合你的 ICP？（公司规模、行业、角色、技术栈）
- **参与度评分** — 他们是否展示了购买意向？（定价页面、演示请求、多次访问）

仅满足一项是不够的。一家完美适配但从未参与的公司不是 MQL。一个下载每本电子书的学生也不是 MQL。

### MQL 到 SQL 交接 SLA

定义响应时间并记录：
- MQL 警报发送给分配的销售代表
- 代表在 **4 小时内**联系（工作时间）
- 代表在 **48 小时内**确认或拒绝
- 被拒绝的 MQL 进入回收培育流程，附带原因代码

**完整的生命周期阶段模板和 SLA 示例**：参见 [references/lifecycle-definitions.md](references/lifecycle-definitions.md)

## 线索评分

### 评分维度

**显性评分（适配度）** — 他们是谁：
- 公司规模、行业、收入
- 职位、级别、部门
- 技术栈、地理位置

**隐性评分（参与度）** — 他们做什么：
- 页面访问（尤其是定价、演示、案例研究）
- 内容下载、网络研讨会参与
- 邮件互动（打开、点击）
- 产品使用（针对 PLG）

**负向评分** — 取消资格的信号：
- 竞争对手邮箱域名
- 学生/个人邮箱
- 取消订阅、垃圾邮件投诉
- 职位不匹配（实习生、学生）

### 构建评分模型

1. 定义你的 ICP 属性并分配权重
2. 从成交数据中识别高意向行为信号
3. 为每个属性和行为设置分值
4. 设置 MQL 阈值（通常在 100 分制中为 50-80 分）
5. 用历史数据测试 — 模型是否能正确识别过去的成功案例？
6. 上线、衡量、每季度重新校准

### 常见评分错误

- 过度权重内容下载（研究 ≠ 购买意向）
- 未包含负向评分（让劣质线索通过）
- 设置后不管（买家行为会变化；每季度重新校准）
- 对所有页面访问同等评分（定价页面 ≠ 博客文章）

**详细的评分模板和示例模型**：参见 [references/scoring-models.md](references/scoring-models.md)

## 线索路由

### 路由方法

| 方法 | 工作原理 | 适用场景 |
|------|----------|----------|
| **轮询** | 平均分配给代表 | 平等区域、相似交易规模 |
| **区域制** | 按地理位置、行业或细分分配 | 区域团队、行业专家 |
| **账户制** | 指定账户分配给指定代表 | ABM 模式、战略客户 |
| **技能制** | 按交易复杂度、产品线或语言路由 | 多产品线、全球团队 |

### 路由规则要点

- 首先路由到**最具体的匹配**，然后回退到通用规则
- 包含**后备负责人** — 未分配的线索会很快变冷并浪费管道
- 轮询应考虑**代表容量和可用性**（休假、配额完成情况）
- 记录每个路由决策以便审计和优化

### 响应速度

响应时间是线索转化的最大因素：
- **5 分钟内**联系 = 合格可能性高 21 倍（Lead Connect）
- **30 分钟后**，转化率下降 10 倍
- **24 小时后**，线索实际上已经冷了

构建优先考虑速度的路由规则。立即提醒代表。如果错过 SLA 则升级。

**路由决策树和平台特定设置**：参见 [references/routing-rules.md](references/routing-rules.md)

## 管道阶段管理

### 管道阶段

| 阶段 | 必填字段 | 退出条件 |
|------|----------|----------|
| **已确认** | 联系信息、公司、来源、适配度评分 | 发现通话已安排 |
| **发现** | 痛点、当前解决方案、时间线 | 需求确认、演示已安排 |
| **演示/评估** | 技术需求、决策者 | 评估积极、提案请求 |
| **提案** | 定价、条款、利益相关者图谱 | 提案已交付并审阅 |
| **谈判** | 红线、审批流程、结单日期 | 条款达成、合同发送 |
| **成交** | 签署合同、付款条款 | 移交给 CS 完成 |
| **失败** | 失败原因、竞争对手（如有） | 复盘已记录 |

### 阶段卫生

- **每阶段必填字段** — 不让代表在未填写必要数据的情况下推进交易
- **陈旧交易警报** — 标记在阶段停留超过平均时间的交易（如 2 倍平均天数）
- **阶段跳跃检测** — 当交易跳过阶段时警报（已确认 → 提案 跳过发现）
- **结单日期纪律** — 推迟日期必须包含原因；禁止静默推迟

### 管道指标

| 指标 | 告诉你什么 |
|------|------------|
| 阶段转化率 | 交易在哪里死亡 |
| 阶段平均停留时间 | 交易在哪里停滞 |
| 管道速度 | 每天通过漏斗的收入 |
| 覆盖率 | 管道价值 vs 配额（目标 3-4 倍） |
| 按来源的赢单率 | 哪些渠道产生真正的收入 |

## CRM 自动化工作流

### 基础自动化

- **生命周期阶段更新** — 满足条件时自动推进阶段
- **交接时创建任务** — MQL 分配给代表时创建跟进任务
- **SLA 警报** — 代表错过响应时间 SLA 时通知经理
- **交易阶段触发器** — 成交时自动发送提案、更新预测、通知 CS

### 营销到销售自动化

- **MQL 警报** — 即时通知分配的代表，附带线索上下文
- **会议预订** — 潜在客户通过日程工具预订时通知 AE
- **线索活动摘要** — 活跃线索高意向行为的每日汇总
- **重新参与触发** — 休眠线索返回网站时提醒销售

### 日程安排集成

- **轮询日程安排** — 在团队间平均分配会议
- **按条件路由** — 企业线索发送给资深 AE，SMB 发送给初级代表
- **会前数据增强** — 通话前自动填充 CRM 记录
- **缺席工作流** — 潜在客户错过会议时自动跟进

**平台特定的工作流配方**：参见 [references/automation-playbooks.md](references/automation-playbooks.md)

## 交易审批流程

### 何时需要交易审批

- ACV 超过 **$25K**（或你的非标交易阈值）
- 非标准付款条款（net-90、季度账单）
- 多年合同带自定义定价
- 超出公布层级的批量折扣
- 自定义法律条款或 SLA

### 审批工作流层级

| 交易规模 | 所需审批 |
|----------|----------|
| 标准定价 | 自动批准 |
| 10-20% 折扣 | 销售经理 |
| 20-40% 折扣 | 销售副总裁 |
| 40%+ 折扣或自定义条款 | 交易审批审查 |
| 多年 / 企业 | 财务 + 法务 |

### 非标准条款处理

记录每个例外。跟踪最常请求的非标准条款 — 如果每个人都要求相同的例外，它应该成为标准。每季度审查。

## 数据卫生与增强

### 去重策略

- **匹配规则** — 邮箱域名 + 公司名称 + 电话作为主要匹配键
- **合并优先级** — CRM 记录优先于营销自动化；字段以最近活动为准
- **定期去重** — 每周运行自动去重，边缘情况人工审查

### 必填字段强制执行

- 在每个生命周期阶段强制执行必填字段
- 字段为空时阻止阶段推进
- 使用渐进式信息收集 — 不要一开始就要求所有信息

### 数据增强工具

| 工具 | 优势 |
|------|------|
| Clearbit | 实时增强，适合科技公司 |
| Apollo | 联系人数据 + 序列，适合潜在客户开发 |
| ZoomInfo | 企业级，最大的 B2B 数据库 |

### 季度审计清单

- 审查并合并重复项
- 验证陈旧联系人的邮箱可送达性
- 归档 12 个月以上无活动的联系人
- 审计生命周期阶段分布（寻找瓶颈）
- 在样本集上验证增强数据准确性

## RevOps 指标仪表盘

### 关键指标

| 指标 | 公式 / 定义 | 基准 |
|------|-------------|------|
| 线索到 MQL 率 | MQL / 总线索 | 5-15% |
| MQL 到 SQL 率 | SQL / MQL | 30-50% |
| SQL 到商机率 | 商机 / SQL | 50-70% |
| 管道速度 | (交易数 × 平均交易规模 × 赢单率) / 平均销售周期 | 因 ACV 而异 |
| CAC | 总销售 + 营销支出 / 新客户数 | LTV:CAC > 3:1 |
| LTV:CAC 比率 | 客户终身价值 / CAC | 3:1 到 5:1 健康 |
| 响应速度 | 从填写表单到首次代表联系的时间 | < 5 分钟理想 |
| 赢单率 | 成交 / 总商机 | 20-30%（因情况而异） |

### 仪表盘结构

构建三个视图：
1. **营销视图** — 线索量、MQL 率、来源归因、每个 MQL 成本
2. **销售视图** — 管道价值、阶段转化、速度、预测准确性
3. **高管视图** — CAC、LTV:CAC、收入 vs 目标、管道覆盖
