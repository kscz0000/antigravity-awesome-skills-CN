# MISSION.md 格式

`MISSION.md` 位于工作空间根目录。它记录用户学习该主题的*原因*。每一项教学决策——下一步教什么、呈现哪些资源、设计怎样的练习——都应当回溯到本文档。

## 模板

```md
# 使命：{主题}

## Why（为什么）
{1-3 句话。用户所追求的、具体的现实目标。当他们掌握这项技能后，他们的工作或生活会发生怎样的变化？避免"理解 X"之类的抽象表述——深挖背后的实质结果。}

## Success looks like（成功是什么样子）
- {一件具体的、可观察到的、用户能完成的事}
- {另一件具体的事}
- {…}

## Constraints（约束）
- {时间、预算、已有承诺、学习偏好，任何限制做法的事项}

## Out of scope（不在范围内）
- {用户明确表示当前不打算触及的相邻主题——用以保护最近发展区}
```

## 规则

- **每个工作空间只对应一个使命。** 如果用户想学两件互不相关的事，那就对应两个工作空间。
- **具体优先于抽象。** "在十月前完成半马"胜过"更健康"。"把一个 Rust 命令行工具交付给团队"胜过"学习 Rust"。
- **对模糊说不。** 如果用户无法说清"为什么"，在动笔之前先访谈他们。错误的使命比没有使命更糟糕。
- **现实变化时同步更新。** 使命会变。当用户的目标迁移时，更新该文件——不要让过期的使命继续影响未来的会话。
- **保持简练。** 如果 `MISSION.md` 长过一屏，它就已经不再是"指南针"，而是变成了"计划书"。
