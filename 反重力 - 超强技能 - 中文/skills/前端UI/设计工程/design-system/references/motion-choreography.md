# 动效编排——可泛化的动画指南

何时动画、哪些过渡值得保留、时长控制，以及让动效从抖动走向
有意为之的工艺细节。**框架无关**——这些规则同时适用于
纯 CSS、Web Animations API、浏览器 View Transition API、JS 动效库，
或 React 的 `<ViewTransition>`。当 UI 涉及状态变化、导航、列表变化
或值得动画化的揭示时，从 design-system 的动效一节加载本文件。

> 改编并泛化自 Web Interface Guidelines（`vercel-labs/web-interface-guidelines` @ `4e799d4`，2026-04-06）以及 React View Transitions skill（`vercel-labs/agent-skills` @ `f8a72b9`，2026-06-10）。它们的 API 细节绑定 React/Next；下面的*编排*部分是可移植的。

## 1. 何时动画——每个过渡都要物有所值

只有当动效**传达**以下信息之一时才使用动画：空间关系、
连续性（"同一个东西，换了个位置"）、因果关系、数据已到达，或
刻意的趣味感。**如果一句话说不清这个过渡在传达什么，删掉它。**
没有信息的动效就是噪音，只会让性能和注意力受损。

- **由输入驱动，绝不自动播放。** 响应用户操作或他们引起的状态变化来动画——
  而不是由定时器对他们播放。
- 这是"克制原则"在动效上的具体体现：默认是*不*做动画；
  只有明确的传达目的才能让回答变成"是"。

## 2. 哪些过渡值得保留——优先级排序

当多种变化同时发生时，把所有*适用*的都实现（不是"挑一个"），
大致按下面的价值顺序：

1. **共享元素**——同一对象在不同视图间持续存在（缩略图 → 主图）。
   表达"这是同一个东西，进入下一层"。价值最高，最值得投入。
2. **揭示**——骨架/加载 → 真实内容。表达"数据已加载"。
3. **列表身份**——集合重排/筛选时各项目保持身份。表达"还是这些项目，
   只是换了个排列"（动位置，不要整体淡出）。
4. **状态变化**——某个元素进入/离开（面板、提示、行）。表达"它出现了/离开了"。
5. **路由/章节切换**——进入新位置。

只有在 UI 不存在对应变化时才跳过某一级。后台刷新/静默
重新校验应当**完全**不动效。

## 3. 按导航*类型*选风格——方向必须诚实

动画风格不能暗示一种并不存在的空间关系：

| 导航类型 | 动画 | 原因 |
|---|---|---|
| **层级**（列表 → 详情，父 → 子） | 定向滑动（从一侧进入，从另一侧离开） | 方向编码深度 |
| **有序序列**（上/下一张图、轮播、分页） | 定向滑动；"下一张"从右，"上一张"从左 | 方向编码位置 |
| **横向/同级**（标签 ↔ 标签，无序） | **淡入 / 交叉淡入**——而非滑动 | 没有深度，滑动会撒谎 |
| **揭示**（骨架 → 内容） | 淡入或短上滑 | 内容到达 |
| **后台刷新/重新校验** | 无 | 没有用户需要追踪的变化 |

最常见的动效错误就是在横向导航上做定向滑动——它错误地在
同级之间暗示了前进/后退的深度。

## 4. 时长与缓动（作为起点，不是定律）

| 交互 | 时长 |
|---|---|
| 直接切换（展开/收起、开关） | 100–200 毫秒 |
| 路由/章节切换（滑动） | 150–250 毫秒 |
| 揭示（骨架 → 内容） | 200–400 毫秒 |
| 共享元素变形 | 300–500 毫秒 |

- **缓动匹配变化：** 进入用 `ease-out`（减速进入），离开用 `ease-in`
  （加速离开），位置移动用 `ease-in-out`。按"在变什么"来选
  （大小、距离、触发器）——更大/更远就稍长一点。
- 俏皮/玩具般的方向（design-thinking）可以拉长这些时长并加入 spring/bounce；
  克制/专业的方向保持紧凑。让手感匹配所选方向，而不是默认套用数字。

## 5. 机制——动什么、怎么动（任何技术栈）

- **只使用合成器友好的属性：** 只动 `transform` 和 `opacity`。**绝不**动画
  布局属性（`width`、`height`、`top`、`left`）——它们会触发重排并造成抖动。
- **不要写 `transition: all`**——明确列出属性；`all` 会悄悄动画
  影响布局的属性，导致抖动。
- **正确的 `transform-origin`**——把运动锚定在它"物理上"开始的地方（菜单
  从触发点开始，而不是屏幕中心）。
- **可中断**——新的用户输入取消/重定向进行中的动画；动效从不
  成为强制等待。
- **SVG 变换**——应用到 `<g>` 包装层，并设置 `transform-box: fill-box;
  transform-origin: center;`（避免 Safari 的原点 bug）。
- **技术栈偏好：** CSS > Web Animations API > JS 库。优先使用平台；
  只有在平台无法表达时才引入库。

## 6. 区分精致与业余的工艺细节

- **变形时的运动模糊。** 共享元素变形如果在中段（约 30% 处）加一小段
  模糊（例如 `filter: blur(3px)`），最后回到清晰，会读起来既快又有物理感。
- **不要栅格缩放文本。** 不同字号间的共享元素变形（如 `h3 → h1`）会把
  小字号的*位图*放大 → 变成模糊的鬼影。改为保持/交叉淡入文本
  （以完整分辨率显示新文本，隐藏旧快照），而不是缩放它。
- **隔离持久 Chrome。** 在过渡中持续显示的顶栏、导航、侧边栏、固定工具栏
  必须**不**随页面内容一起滑动——给它们独立的过渡身份（或排除它们），
  让它们在内容移动时保持原位。
- **防闪烁揭示。** 如果要显示 spinner/骨架，加上较短的展示延迟（约 150–300 毫秒）
  以及最小可见时间（约 300–500 毫秒），避免快速响应时 loader 一闪而过。（参见
  design-ux 交互附加项。）

## 7. 无障碍——减少动效

对于**公开/多用户**构建，尊重 `prefers-reduced-motion`：去掉补间（保留
最终态 / 0 秒交叉淡入），不要抹掉动效传达的信息。对于
**个人/单用户**项目，本仓库刻意覆盖该标志——参见
`motion-preference-rule`。无论哪种情况，都要避免视差、跨视口的大幅变换，
以及频闪。

## 一句话检验

在加入一个过渡之前：*它在传达什么，这种风格是否诚实地表达了它？*
如果第一个问题答不上来，就删掉它；如果风格暗示了并不存在的深度/位置
（同级之间的滑动），就换一种风格。
