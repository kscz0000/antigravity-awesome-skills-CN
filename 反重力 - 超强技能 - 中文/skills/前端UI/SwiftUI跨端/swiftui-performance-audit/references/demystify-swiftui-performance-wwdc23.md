# 揭秘 SwiftUI 性能（WWDC23）（摘要）

背景：WWDC23 演讲，关于建立 SwiftUI 性能心智模型和分诊卡顿/掉帧。

## 性能循环

- 测量 → 识别 → 优化 → 重新测量。
- 聚焦于具体症状（导航缓慢、动画异常、旋转指示器）。

## 依赖与更新

- 视图形成依赖图；动态属性是更新的常见来源。
- 仅在调试时使用 `Self._printChanges()` 检查额外依赖。
- 通过提取视图或缩小状态来消除不必要的依赖。
- 考虑使用 `@Observable` 实现更精细的属性追踪。

## 更新缓慢的常见原因

- 昂贵的视图 body（字符串插值、过滤、格式化）。
- 动态属性实例化和 `body` 中的状态初始化。
- 列表/表格中缓慢的身份解析。
- 隐藏工作：bundle 查找、堆分配、重复字符串构造。

## 避免在视图 body 中缓慢初始化

- 不要在视图 body 中同步创建重型模型。
- 使用 `.task` 获取异步数据，保持 `init` 轻量。

## 列表和表格身份规则

- 稳定的身份标识对性能和动画至关重要。
- 确保 `ForEach` 中每个元素的视图数量恒定。
- 避免在 `ForEach` 中内联过滤；预先过滤并缓存集合。
- 避免在列表行中使用 `AnyView`；它隐藏身份标识并增加开销。
- 尽可能展平嵌套的 `ForEach` 以减少开销。

## 表格特定说明

- `TableRow` 解析为单行；行数必须恒定。
- 优先使用精简的 `Table` 初始化器以强制恒定行数。
- 需要向后部署时使用显式 ID。

## 调试辅助

- 使用 Instruments 分析卡顿和掉帧。
- 在调试期间使用 `_printChanges` 验证依赖假设。
