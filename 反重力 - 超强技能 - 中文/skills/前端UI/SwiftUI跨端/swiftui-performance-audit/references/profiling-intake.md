# 性能分析收集清单

## 用途

当代码审查无法解释 SwiftUI 性能问题，需要从用户获取运行时证据时使用此清单。

## 首先询问

- 精确症状：CPU 飙升、掉帧、内存增长、卡顿或视图更新过多。
- 精确交互：滚动、打字、初始加载、导航 push/pop、动画、sheet 弹出或后台刷新。
- 目标设备和系统版本。
- 问题是在真机还是仅在模拟器上复现。
- 构建配置：Debug 还是 Release。
- 用户是否已有基线或变更前后的对比数据。

## 默认性能分析请求

请用户：
- 尽可能在 Release 构建中运行应用。
- 使用 SwiftUI Instruments 模板。
- 仅复现问题交互足够长时间以采集到问题即可。
- 同时采集 SwiftUI 时间线和 Time Profiler。
- 导出 trace 或提供关键 SwiftUI 通道和 Time Profiler 调用树的截图。

## 请求以下产物

- 相关 SwiftUI 通道的 trace 导出或截图
- Time Profiler 调用树截图或导出
- 设备/系统/构建配置
- 简要说明采集时正在执行的操作
- 如涉及内存，提供内存图或 Allocations 数据（如有）

## 何时请求更多信息

- 如第一次采集混入了多个交互，请求第二次采集。
- 如用户已尝试修复，请求变更前后的对比采集。
- 如问题仅在模拟器出现或滚动流畅度很重要，请求真机采集。

## 常见陷阱

- Debug 构建会扭曲 SwiftUI 的时序和分配行为。
- 模拟器 trace 可能遗漏仅在真机出现的渲染或内存问题。
- 单次采集中混入多个交互会增加归因难度。
- 没有复现说明的截图难以解读。
