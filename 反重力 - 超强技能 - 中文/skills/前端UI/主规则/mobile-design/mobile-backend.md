# 移动端后端模式

> **本文件覆盖移动客户端专属的后端/API模式。**
> 通用后端模式见 `nodejs-best-practices` 和 `api-patterns`。
> **移动端后端 ≠ Web后端。约束不同，模式不同。**

---

## 🧠 移动端后端思维

```
移动客户端与Web客户端完全不同：
├── 网络不稳定（2G、地铁、电梯）
├── 电池受限（减少唤醒）
├── 存储有限（不能缓存一切）
├── 会话频繁中断（来电、通知）
├── 设备碎片化（旧手机到旗舰机）
└── 二进制更新慢（应用商店审核）
```

**你的后端必须为以上所有情况兜底。**

## 🚫 AI移动端后端反模式

### AI构建移动后端时的常见错误：

| ❌ AI默认做法 | 问题 | ✅ 移动端正确做法 |
|---------------|------|-------------------|
| Web和移动共用同一API | 移动需要精简响应 | 独立移动端点或字段选择 |
| 返回完整对象 | 浪费带宽和电量 | 部分响应、分页 |
| 不考虑离线 | 无网络时应用崩溃 | 离线优先设计、同步队列 |
| 万物皆WebSocket | 耗电 | 推送通知 + 轮询降级 |
| 无应用版本管理 | 无法强制更新，破坏性变更 | 版本头、最低版本检查 |
| 通用错误消息 | 用户无法解决问题 | 移动端专属错误码 + 恢复操作 |
| 基于Session的认证 | 移动应用会重启 | 基于令牌 + 刷新机制 |
| 忽略设备信息 | 无法调试问题 | 请求头携带设备ID、应用版本 |

## 1. 推送通知

### 平台架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    你的后端                                      │
├─────────────────────────────────────────────────────────────────┤
│                         │                                        │
│              ┌──────────┴──────────┐                            │
│              ▼                     ▼                            │
│    ┌─────────────────┐   ┌─────────────────┐                    │
│    │   FCM (Google)  │   │  APNs (Apple)   │                    │
│    │   Firebase      │   │  直连或通过FCM   │                    │
│    └────────┬────────┘   └────────┬────────┘                    │
│             │                     │                              │
│             ▼                     ▼                              │
│    ┌─────────────────┐   ┌─────────────────┐                    │
│    │  Android 设备   │   │    iOS 设备     │                    │
│    └─────────────────┘   └─────────────────┘                    │
└─────────────────────────────────────────────────────────────────┘
```

### 推送类型

| 类型 | 用途 | 用户看到 |
|------|------|----------|
| **展示型** | 新消息、订单更新 | 通知横幅 |
| **静默型** | 后台同步、内容更新 | 无（后台处理） |
| **数据型** | 应用自定义处理 | 取决于应用逻辑 |

### 反模式

| ❌ 绝不 | ✅ 始终 |
|----------|----------|
| 推送中发送敏感数据 | 推送只说"新消息"，应用自行获取内容 |
| 推送轰炸 | 合并、去重、尊重免打扰时段 |
| 所有人同一消息 | 按用户偏好、时区分段 |
| 忽略失效令牌 | 定期清理无效令牌 |
| iOS跳过APNs | 仅FCM不保证iOS送达 |

### 令牌管理

```
令牌生命周期：
├── 应用注册 → 获取令牌 → 发送到后端
├── 令牌可能变化 → 应用启动时必须重新注册
├── 令牌过期 → 从数据库清理
├── 用户卸载 → 令牌失效（通过错误检测）
└── 多设备 → 每用户存储多个令牌
```

## 2. 离线同步与冲突解决

### 同步策略选择

```
什么类型的数据？
        │
        ├── 只读（新闻、目录）
        │   └── 简单缓存 + TTL
        │       └── ETag/Last-Modified 做失效
        │
        ├── 用户独占（笔记、待办）
        │   └── 后写覆盖（简单）
        │       └── 或基于时间戳合并
        │
        ├── 协作型（共享文档）
        │   └── 需要 CRDT 或 OT
        │       └── 考虑 Firebase/Supabase
        │
        └── 关键型（支付、库存）
            └── 服务器是唯一真相源
                └── 乐观UI + 服务器确认
```

### 冲突解决策略

| 策略 | 工作方式 | 适用场景 |
|------|----------|----------|
| **后写覆盖** | 最新时间戳覆盖 | 简单数据、单用户 |
| **服务器胜出** | 服务器始终权威 | 关键交易 |
| **客户端胜出** | 离线变更优先 | 离线重度应用 |
| **合并** | 逐字段组合变更 | 文档、富内容 |
| **CRDT** | 数学上无冲突 | 实时协作 |

### 同步队列模式

```
客户端：
├── 用户操作 → 写入本地DB
├── 加入同步队列 → { action, data, timestamp, retries }
├── 网络可用 → FIFO处理队列
├── 成功 → 从队列移除
├── 失败 → 退避重试（最多5次）
└── 冲突 → 应用解决策略

服务端：
├── 接受变更，附带客户端时间戳
├── 与服务端版本比较
├── 应用冲突解决
├── 返回合并后的状态
└── 客户端用服务端响应更新本地
```

## 3. 移动端API优化

### 响应体积缩减

| 技术 | 节省 | 实现 |
|------|------|------|
| **字段选择** | 30-70% | `?fields=id,name,thumbnail` |
| **压缩** | 60-80% | gzip/brotli（自动） |
| **分页** | 不定 | 移动端用游标分页 |
| **图片变体** | 50-90% | `/image?w=200&q=80` |
| **增量同步** | 80-95% | 仅返回时间戳之后的变更记录 |

### 分页：游标 vs 偏移

```
偏移（移动端不推荐）：
├── 第1页: OFFSET 0 LIMIT 20
├── 第2页: OFFSET 20 LIMIT 20
├── 问题：新增条目 → 重复！
└── 问题：大偏移 = 慢查询

游标（移动端推荐）：
├── 首次: ?limit=20
├── 下一页: ?limit=20&after=cursor_abc123
├── 游标 = 编码(id + 排序值)
├── 数据变更不会重复
└── 性能稳定
```

### 批量请求

```
而非：
GET /users/1
GET /users/2  
GET /users/3
（3次往返，3倍延迟）

使用：
POST /batch
{ requests: [
    { method: "GET", path: "/users/1" },
    { method: "GET", path: "/users/2" },
    { method: "GET", path: "/users/3" }
]}
（1次往返）
```

## 4. 应用版本管理

### 版本检查端点

```
GET /api/app-config
Headers:
  X-App-Version: 2.1.0
  X-Platform: ios
  X-Device-ID: abc123

Response:
{
  "minimum_version": "2.0.0",
  "latest_version": "2.3.0",
  "force_update": false,
  "update_url": "https://apps.apple.com/...",
  "feature_flags": {
    "new_player": true,
    "dark_mode": true
  },
  "maintenance": false,
  "maintenance_message": null
}
```

### 版本比较逻辑

```
客户端版本 vs 最低版本：
├── client >= minimum → 正常使用
├── client < minimum → 显示强制更新页
│   └── 阻止使用直到更新
└── client < latest → 显示可选更新提示

功能开关：
├── 无需应用更新即可启停功能
├── 按版本/设备做A/B测试
└── 渐进发布（10% → 50% → 100%）
```

## 5. 移动端认证

### 令牌策略

```
访问令牌：
├── 短期（15分钟 - 1小时）
├── 存在内存中（不持久化）
├── 用于API请求
└── 过期时刷新

刷新令牌：
├── 长期（30-90天）
├── 存在SecureStore/Keychain
├── 仅用于获取新访问令牌
└── 每次使用后轮换（安全）

设备令牌：
├── 标识此设备
├── 支持"登出所有设备"
├── 与刷新令牌一起存储
└── 服务端追踪活跃设备
```

### 静默重新认证

```
请求流程：
├── 用访问令牌发起请求
├── 401 未授权？
│   ├── 有刷新令牌？
│   │   ├── 是 → 调用 /auth/refresh
│   │   │   ├── 成功 → 重试原始请求
│   │   │   └── 失败 → 强制登出
│   │   └── 否 → 强制登出
│   └── 令牌刚过期（非无效）
│       └── 自动刷新，用户无感知
└── 成功 → 继续
```

## 6. 移动端错误处理

### 移动端专属错误格式

```json
{
  "error": {
    "code": "PAYMENT_DECLINED",
    "message": "Your payment was declined",
    "user_message": "Please check your card details or try another payment method",
    "action": {
      "type": "navigate",
      "destination": "payment_methods"
    },
    "retry": {
      "allowed": true,
      "after_seconds": 5
    }
  }
}
```

### 错误分类

| 状态码范围 | 类别 | 移动端处理 |
|------------|------|------------|
| 400-499 | 客户端错误 | 显示消息，需用户操作 |
| 401 | 认证过期 | 静默刷新或重新登录 |
| 403 | 禁止访问 | 显示升级/权限页 |
| 404 | 未找到 | 从本地缓存移除 |
| 409 | 冲突 | 显示同步冲突UI |
| 429 | 限流 | 按Retry-After头退避重试 |
| 500-599 | 服务端错误 | 退避重试，显示"稍后再试" |
| 网络错误 | 无连接 | 使用缓存数据，排队同步 |

## 7. 媒体与二进制处理

### 图片优化

```
客户端请求：
GET /images/{id}?w=400&h=300&q=80&format=webp

服务端响应：
├── 即时缩放或使用CDN
├── Android用WebP（更小）
├── iOS 14+用HEIC（如支持）
├── JPEG降级
└── Cache-Control: max-age=31536000
```

### 分块上传（大文件）

```
上传流程：
1. POST /uploads/init
   { filename, size, mime_type }
   → { upload_id, chunk_size }

2. PUT /uploads/{upload_id}/chunks/{n}
   → 上传每个分块（1-5 MB）
   → 中断后可续传

3. POST /uploads/{upload_id}/complete
   → 服务端组装分块
   → 返回最终文件URL
```

### 流媒体音频/视频

```
要求：
├── iOS用HLS（HTTP Live Streaming）
├── Android用DASH或HLS
├── 多画质级别（自适应码率）
├── Range请求支持（拖动进度）
└── 离线下载分块

端点：
GET /media/{id}/manifest.m3u8  → HLS清单
GET /media/{id}/segment_{n}.ts → 视频分片
GET /media/{id}/download       → 完整文件用于离线
```

## 8. 移动端安全

### 设备认证

```
验证真实设备（非模拟器/机器人）：
├── iOS: DeviceCheck API
│   └── 服务端向Apple验证
├── Android: Play Integrity API（替代SafetyNet）
│   └── 服务端向Google验证
└── 失败时拒绝：认证不过则拦截
```

### 请求签名

```
客户端：
├── 创建签名 = HMAC(timestamp + path + body, secret)
├── 发送: X-Signature: {signature}
├── 发送: X-Timestamp: {timestamp}
└── 发送: X-Device-ID: {device_id}

服务端：
├── 验证时间戳（5分钟内）
├── 用相同输入重建签名
├── 比对签名
└── 不匹配则拒绝（检测到篡改）
```

### 限流

```
移动端专属限制：
├── 按设备（X-Device-ID）
├── 按用户（认证后）
├── 按端点（敏感端点更严格）
└── 优先滑动窗口

响应头：
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1609459200
Retry-After: 60（429时）
```

## 9. 监控与分析

### 移动端必须发送的请求头

```
每个移动端请求应包含：
├── X-App-Version: 2.1.0
├── X-Platform: ios | android
├── X-OS-Version: 17.0
├── X-Device-Model: iPhone15,2
├── X-Device-ID: uuid（持久化）
├── X-Request-ID: uuid（每请求，用于追踪）
├── Accept-Language: zh-CN
└── X-Timezone: Asia/Shanghai
```

### 日志记录

```
每个请求记录：
├── 以上所有请求头
├── 端点、方法、状态码
├── 响应时间
├── 错误详情（如有）
└── 用户ID（如已认证）

告警：
├── 单版本错误率 > 5%
├── P95延迟 > 2秒
├── 特定版本崩溃激增
├── 认证失败激增（攻击？）
└── 推送送达失败激增
```
