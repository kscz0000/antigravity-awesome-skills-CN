## 1. 框架选择

### 主决策树

```
你要构建什么？
        │
        ├── 需要不经应用商店审核的OTA更新？
        │   │
        │   ├── 是 → React Native + Expo
        │   │         ├── 开发用 Expo Go
        │   │         ├── 生产OTA用 EAS Update
        │   │         └── 适合：快速迭代、Web团队
        │   │
        │   └── 否 → 继续 ▼
        │
        ├── 需要跨平台像素级一致的自定义UI？
        │   │
        │   ├── 是 → Flutter
        │   │         ├── 自定义渲染引擎
        │   │         ├── iOS + Android 统一UI
        │   │         └── 适合：品牌化、视觉型应用
        │   │
        │   └── 否 → 继续 ▼
        │
        ├── 重度原生功能（ARKit、HealthKit、特定传感器）？
        │   │
        │   ├── 仅iOS → SwiftUI / UIKit
        │   │              └── 最大化原生能力
        │   │
        │   ├── 仅Android → Kotlin + Jetpack Compose
        │   │                  └── 最大化原生能力
        │   │
        │   └── 双端 → 原生开发 + 共享逻辑
        │              └── Kotlin Multiplatform 共享
        │
        ├── 已有Web团队 + TypeScript代码库？
        │   │
        │   └── 是 → React Native
        │             ├── React开发者熟悉的范式
        │             ├── 与Web共享代码（有限）
        │             └── 生态丰富
        │
        └── 企业级且已有Flutter团队？
            │
            └── 是 → Flutter
                      └── 发挥现有专业优势
```

### 框架对比

| 因素 | React Native | Flutter | 原生 (Swift/Kotlin) |
|------|-------------|---------|---------------------|
| **OTA更新** | ✅ Expo | ❌ 无 | ❌ 无 |
| **学习曲线** | 低（React开发者） | 中 | 较高 |
| **性能** | 良好 | 优秀 | 最佳 |
| **UI一致性** | 平台原生 | 完全一致 | 平台原生 |
| **包体积** | 中等 | 较大 | 最小 |
| **原生访问** | 通过桥接 | 通过Channel | 直接 |
| **热重载** | ✅ | ✅ | ✅ (Xcode 15+) |

### 何时选择原生

```
选择原生的情况：
├── 需要极致性能（游戏、3D）
├── 需要深度系统集成
├── 平台特有功能是核心
├── 团队有原生开发经验
├── 应用商店展示是首要目标
└── 长期维护优先

避免原生的情况：
├── 预算/时间有限
├── 需要快速迭代
├── 双平台UI完全一致
├── 团队以Web为主
└── 跨平台是优先项
```

## 2. 状态管理选择

### React Native 状态决策

```
状态复杂度如何？
        │
        ├── 简单应用，少量页面，共享状态极少
        │   │
        │   └── Zustand（或直接 useState/Context）
        │       ├── 极少样板代码
        │       ├── 易于理解
        │       └── 可扩展到中等规模
        │
        ├── 以服务端数据为主（API驱动）
        │   │
        │   └── TanStack Query (React Query) + Zustand
        │       ├── Query管理服务端状态
        │       ├── Zustand管理UI状态
        │       └── 出色的缓存与重新获取
        │
        ├── 复杂应用，功能众多
        │   │
        │   └── Redux Toolkit + RTK Query
        │       ├── 可预测、可调试
        │       ├── RTK Query处理API
        │       └── 适合大团队
        │
        └── 原子化、细粒度状态需求
            │
            └── Jotai
                ├── 基于Atom（类似Recoil）
                ├── 最小化重渲染
                └── 适合派生状态
```

### Flutter 状态决策

```
状态复杂度如何？
        │
        ├── 简单应用，正在学Flutter
        │   │
        │   └── Provider（或 setState）
        │       ├── 官方推荐，简单
        │       ├── Flutter内置
        │       └── 适合小型应用
        │
        ├── 现代、类型安全、可测试
        │   │
        │   └── Riverpod 2.0
        │       ├── 编译时安全
        │       ├── 代码生成
        │       ├── 适合中大型应用
        │       └── 新项目推荐
        │
        ├── 企业级，需要严格模式
        │   │
        │   └── BLoC
        │       ├── 事件 → 状态模式
        │       ├── 高度可测试
        │       ├── 样板代码较多
        │       └── 适合大团队
        │
        └── 快速原型
            │
            └── GetX（谨慎使用）
                ├── 实现快
                ├── 模式不够严格
                └── 规模大了容易混乱
```

### 状态管理反模式

```
❌ 禁止：
├── 全局状态用于一切
├── 混用多种状态管理方案
├── 把服务端状态存到本地状态
├── 跳过状态规范化
├── 过度使用Context（重渲染严重）
└── 把导航状态放入应用状态

✅ 应当：
├── 服务端状态 → Query库
├── UI状态 → 最小化，优先本地
├── 只在必要时提升状态
├── 每个项目只选一种方案
└── 状态尽量靠近使用位置
```

## 3. 导航模式选择

```
有多少个顶级目的地？
        │
        ├── 2个目的地
        │   └── 考虑：顶部Tab或简单Stack
        │
        ├── 3-5个（同等重要）
        │   └── ✅ Tab栏 / 底部导航
        │       ├── 最常见模式
        │       └── 易于发现
        │
        ├── 5个以上
        │   │
        │   ├── 全都重要 → 抽屉导航
        │   │               └── 隐藏但选项多
        │   │
        │   └── 部分次要 → Tab栏 + 抽屉混合
        │
        └── 单一线性流程？
            └── 仅Stack导航
                └── 引导页、结账等
```

### 按应用类型选导航

| 应用类型 | 模式 | 原因 |
|----------|------|------|
| 社交（Instagram） | Tab栏 | 频繁切换 |
| 电商 | Tab栏 + Stack | 分类作为Tab |
| 邮件（Gmail） | 抽屉 + 列表-详情 | 文件夹多 |
| 设置 | 仅Stack | 深层下钻 |
| 引导页 | Stack向导 | 线性流程 |
| 即时通讯 | Tab（聊天）+ Stack | 会话线程 |

## 4. 存储策略选择

```
什么类型的数据？
        │
        ├── 敏感数据（令牌、密码、密钥）
        │   │
        │   └── ✅ 安全存储
        │       ├── iOS: Keychain
        │       ├── Android: EncryptedSharedPreferences
        │       └── RN: expo-secure-store / react-native-keychain
        │
        ├── 用户偏好（设置、主题）
        │   │
        │   └── ✅ 键值存储
        │       ├── iOS: UserDefaults
        │       ├── Android: SharedPreferences
        │       └── RN: AsyncStorage / MMKV
        │
        ├── 结构化数据（实体、关系）
        │   │
        │   └── ✅ 数据库
        │       ├── SQLite (expo-sqlite, sqflite)
        │       ├── Realm (NoSQL, 响应式)
        │       └── WatermelonDB (大数据集)
        │
        ├── 大文件（图片、文档）
        │   │
        │   └── ✅ 文件系统
        │       ├── iOS: Documents / Caches 目录
        │       ├── Android: 内部/外部存储
        │       └── RN: react-native-fs / expo-file-system
        │
        └── 缓存的API数据
            │
            └── ✅ Query库缓存
                ├── TanStack Query (RN)
                ├── Riverpod async (Flutter)
                └── 自动失效
```

### 存储对比

| 存储 | 速度 | 安全性 | 容量 | 用途 |
|------|------|--------|------|------|
| 安全存储 | 中 | 🔒 高 | 小 | 令牌、密钥 |
| 键值存储 | 快 | 低 | 中 | 设置项 |
| SQLite | 快 | 低 | 大 | 结构化数据 |
| 文件系统 | 中 | 低 | 很大 | 媒体、文档 |
| Query缓存 | 快 | 低 | 中 | API响应 |

## 5. 离线策略选择

```
离线有多关键？
        │
        ├── 锦上添花（能连就连）
        │   │
        │   └── 缓存上次数据 + 展示过期内容
        │       ├── 实现简单
        │       ├── TanStack Query 配 staleTime
        │       └── 显示"上次更新"时间戳
        │
        ├── 核心需求（主要功能离线可用）
        │   │
        │   └── 离线优先架构
        │       ├── 本地数据库作为唯一真相源
        │       ├── 上线后同步到服务器
        │       ├── 冲突解决策略
        │       └── 操作排队等待同步
        │
        └── 实时关键（协作、聊天）
            │
            └── WebSocket + 本地队列
                ├── 乐观更新
                ├── 最终一致性
                └── 复杂冲突处理
```

### 离线实现模式

```
1. 缓存优先（简单）
   请求 → 检查缓存 → 过期则获取 → 更新缓存
   
2. 过期重验证
   请求 → 返回缓存 → 后台获取 → 更新UI
   
3. 离线优先（复杂）
   操作 → 写入本地DB → 排队同步 → 上线后同步
   
4. 同步引擎
   使用：Firebase、Realm Sync、Supabase realtime
   自动处理冲突解决
```

## 6. 认证模式选择

```
需要什么认证方式？
        │
        ├── 简单邮箱/密码
        │   │
        │   └── 基于令牌（JWT）
        │       ├── 安全存储刷新令牌
        │       ├── 访问令牌放内存
        │       └── 静默刷新流程
        │
        ├── 社交登录（Google、Apple等）
        │   │
        │   └── OAuth 2.0 + PKCE
        │       ├── 使用平台SDK
        │       ├── Deep link回调
        │       └── iOS必须支持Apple登录
        │
        ├── 企业/SSO
        │   │
        │   └── OIDC / SAML
        │       ├── WebView或系统浏览器
        │       └── 正确处理重定向
        │
        └── 生物识别（FaceID、指纹）
            │
            └── 本地认证 + 安全令牌
                ├── 生物识别解锁存储的令牌
                ├── 不能替代服务端认证
                └── 降级为PIN/密码
```

### 认证令牌存储

```
❌ 绝不存储令牌到：
├── AsyncStorage（明文）
├── Redux/状态（未正确持久化）
├── Local Storage等价物
└── 日志或调试输出

✅ 始终存储令牌到：
├── iOS: Keychain
├── Android: EncryptedSharedPreferences
├── Expo: SecureStore
├── 如可用，加生物识别保护
```

## 7. 项目类型模板

### 电商应用

```
推荐技术栈：
├── 框架：React Native + Expo（OTA更新价格）
├── 导航：Tab栏（首页、搜索、购物车、账户）
├── 状态：TanStack Query（商品）+ Zustand（购物车）
├── 存储：SecureStore（认证）+ SQLite（购物车缓存）
├── 离线：缓存商品，排队购物车操作
└── 认证：邮箱/密码 + 社交登录 + Apple Pay

关键决策：
├── 商品图片：懒加载，积极缓存
├── 购物车：通过API跨设备同步
├── 结账：安全，步骤最少
└── Deep link：商品分享、营销
```

### 社交/内容应用

```
推荐技术栈：
├── 框架：React Native 或 Flutter
├── 导航：Tab栏（动态、搜索、创建、通知、个人）
├── 状态：TanStack Query（信息流）+ Zustand（UI）
├── 存储：SQLite（信息流缓存、草稿）
├── 离线：缓存信息流，排队发帖
└── 认证：社交登录为主，Apple必选

关键决策：
├── 信息流：无限滚动，memo化条目
├── 媒体：上传排队，后台上传
├── 推送：Deep link到内容
└── 实时：WebSocket处理通知
```

### 效率/SaaS应用

```
推荐技术栈：
├── 框架：Flutter（UI一致）或 RN
├── 导航：抽屉或Tab栏
├── 状态：Riverpod/BLoC 或 Redux Toolkit
├── 存储：SQLite（离线）、SecureStore（认证）
├── 离线：完整离线编辑，同步
└── 认证：企业用SSO/OIDC

关键决策：
├── 数据同步：冲突解决策略
├── 协作：实时还是最终一致？
├── 文件：大文件处理
└── 企业：MDM、合规
```

## 8. 决策检查清单

### 启动任何项目前

- [ ] 目标平台已定义（iOS/Android/双端）？
- [ ] 基于条件选择了框架？
- [ ] 确定了状态管理方案？
- [ ] 选择了导航模式？
- [ ] 每种数据类型都有存储策略？
- [ ] 离线需求已定义？
- [ ] 认证流程已设计？
- [ ] Deep link从开始就规划？

### 需要向用户确认的问题

```
项目细节模糊时，主动询问：

1. "是否需要不经应用商店审核的OTA更新？"
   → 影响框架选择（Expo = 是）

2. "iOS和Android需要完全一致的UI吗？"
   → 影响框架（Flutter = 一致）

3. "离线需求是什么级别？"
   → 影响架构复杂度

4. "是否已有后端/认证系统？"
   → 影响认证和API方案

5. "目标设备？仅手机还是包含平板？"
   → 影响导航和布局

6. "企业级还是消费级？"
   → 影响认证（SSO）、安全、合规
```

## 9. 反模式决策

### ❌ 决策反模式

| 反模式 | 问题 | 更好做法 |
|--------|------|----------|
| **简单应用用Redux** | 严重过度 | Zustand或Context |
| **MVP用原生** | 开发慢 | 跨平台MVP |
| **3个分区用抽屉** | 导航被隐藏 | Tab栏 |
| **令牌存AsyncStorage** | 不安全 | SecureStore |
| **不考虑离线** | 地铁里就坏了 | 从一开始规划 |
| **所有项目同一技术栈** | 不符合上下文 | 按项目评估 |

## 10. 速查参考

### 框架速选

```
需要OTA？           → React Native + Expo
UI完全一致？        → Flutter
极致性能？          → 原生
有Web团队？         → React Native
快速原型？          → Expo
```

### 状态速选

```
简单应用？          → Zustand / Provider
服务端数据为主？    → TanStack Query / Riverpod
企业级？            → Redux / BLoC
原子化状态？        → Jotai
```

### 存储速选

```
密钥？              → SecureStore / Keychain
设置项？            → AsyncStorage / UserDefaults
结构化数据？        → SQLite
API缓存？           → Query库
```
