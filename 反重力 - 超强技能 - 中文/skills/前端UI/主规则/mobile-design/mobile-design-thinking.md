# 移动端设计思维

> **本文件防止AI使用记忆化模式，强制真正思考。**

## 🔍 组件分解（强制）

### 每个屏幕的分解分析

设计任何屏幕前，执行此分析：

```
屏幕：[屏幕名称]
├── 主要操作：[主要动作是什么？]
│   └── 在拇指区内吗？[是/否 → 为什么？]
│
├── 触控目标：[所有可点击元素]
│   ├── [元素1]：[尺寸]pt → 够大吗？
│   ├── [元素2]：[尺寸]pt → 够大吗？
│   └── 间距：[间距]pt → 误触风险？
│
├── 可滚动内容：
│   ├── 是列表吗？→ FlatList/FlashList [为什么选这个？]
│   ├── 条目数量：~[N] → 性能考量？
│   └── 固定高度？→ 需要getItemLayout吗？
│
├── 状态需求：
│   ├── 本地状态够吗？
│   ├── 需要提升状态吗？
│   └── 需要全局状态吗？[为什么？]
│
├── 平台差异：
│   ├── iOS：[需要什么不同？]
│   └── Android：[需要什么不同？]
│
├── 离线考量：
│   ├── 这个屏幕应该离线工作吗？
│   └── 缓存策略：[是/否/哪种？]
│
└── 性能影响：
    ├── 有重型组件吗？
    ├── 需要memoization吗？
    └── 动画性能？
```

## 🎯 模式质疑矩阵

对每个默认模式提出这些问题：

### 导航模式质疑

| 假设 | 质疑 | 替代方案 |
|------|------|----------|
| "用Tab栏" | 有几个目的地？ | 3个→精简Tab，6+→抽屉 |
| "5个Tab" | 都同等重要吗？ | "更多"Tab？抽屉混合？ |
| "底部导航" | 支持iPad/平板？ | Navigation Rail替代 |
| "Stack导航" | 考虑了Deep link吗？ | URL结构 = 导航结构 |

### 状态模式质疑

| 假设 | 质疑 | 替代方案 |
|------|------|----------|
| "用Redux" | 应用有多复杂？ | 简单：Zustand，服务端：TanStack |
| "全局状态" | 这个状态真的是全局的吗？ | 本地提升、Context选择器 |
| "Context Provider" | 重渲染会是问题吗？ | Zustand、Jotai（Atom方案） |
| "BLoC模式" | 样板代码值得吗？ | Riverpod（代码更少） |

### 列表模式质疑

| 假设 | 质疑 | 替代方案 |
|------|------|----------|
| "FlatList" | 性能关键吗？ | FlashList（更快） |
| "标准renderItem" | memo化了吗？ | useCallback + React.memo |
| "Index作key" | 数据顺序会变吗？ | 用item.id |
| "ListView" | 有分隔线吗？ | ListView.separated |

### UI模式质疑

| 假设 | 质疑 | 替代方案 |
|------|------|----------|
| "FAB右下角" | 用户惯用手？ | 无障碍设置 |
| "下拉刷新" | 这个列表需要刷新吗？ | 仅在必要时 |
| "弹窗底部弹出" | 内容有多少？ | 全屏弹窗可能更好 |
| "滑动操作" | 可发现性？ | 可见按钮替代 |

## 🧪 反记忆化测试

### 每次给出方案前自问

```
┌─────────────────────────────────────────────────────────────────┐
│                    反记忆化检查清单                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  □ 我选择这个方案是因为"一直这么做"吗？                          │
│    → 如果是：停下。考虑替代方案。                                │
│                                                                 │
│  □ 这是训练数据中频繁出现的模式吗？                              │
│    → 如果是：它真的适合本项目吗？                                │
│                                                                 │
│  □ 我是否不假思索地写出了这个方案？                              │
│    → 如果是：退一步，做分解。                                    │
│                                                                 │
│  □ 我考虑了替代方案吗？                                          │
│    → 如果否：想出至少2个替代方案，再决定。                       │
│                                                                 │
│  □ 我是否按平台分别思考了？                                      │
│    → 如果否：分别分析iOS和Android。                              │
│                                                                 │
│  □ 我考虑了这个方案的性能影响吗？                                │
│    → 如果否：内存、CPU、电池影响是什么？                         │
│                                                                 │
│  □ 这个方案适合本项目的上下文吗？                                │
│    → 如果否：根据上下文定制。                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 📊 基于上下文的决策协议

### 根据项目类型差异化思考

```
确定项目类型：
        │
        ├── 电商应用
        │   ├── 导航：Tab（首页、搜索、购物车、账户）
        │   ├── 列表：商品网格（memo化、图片优化）
        │   ├── 性能：图片缓存至关重要
        │   ├── 离线：购物车持久化、商品缓存
        │   └── 特殊：结账流程、支付安全
        │
        ├── 社交/内容应用
        │   ├── 导航：Tab（动态、搜索、创建、通知、个人）
        │   ├── 列表：无限滚动、复杂条目
        │   ├── 性能：信息流渲染至关重要
        │   ├── 离线：信息流缓存、草稿
        │   └── 特殊：实时更新、媒体处理
        │
        ├── 效率/SaaS应用
        │   ├── 导航：抽屉或自适应（手机Tab、平板Rail）
        │   ├── 列表：数据表格、表单
        │   ├── 性能：数据同步
        │   ├── 离线：完整离线编辑
        │   └── 特殊：冲突解决、后台同步
        │
        ├── 工具应用
        │   ├── 导航：极简（可能仅Stack）
        │   ├── 列表：可能极少
        │   ├── 性能：快速启动
        │   ├── 离线：核心功能离线
        │   └── 特殊：小组件、快捷方式
        │
        └── 媒体/流媒体应用
            ├── 导航：Tab（首页、搜索、媒体库、个人）
            ├── 列表：横向轮播、纵向信息流
            ├── 性能：预加载、缓冲
            ├── 离线：下载管理
            └── 特殊：后台播放、投屏
```

## 🔄 交互分解

### 每个手势的分析

添加任何手势前：

```
手势：[手势类型]
├── 可发现性：
│   └── 用户如何发现这个手势？
│       ├── 有视觉提示吗？
│       ├── 会在引导页展示吗？
│       └── 有按钮替代吗？（必须）
│
├── 平台惯例：
│   ├── 这个手势在iOS上意味着什么？
│   ├── 这个手势在Android上意味着什么？
│   └── 我是否偏离了平台惯例？
│
├── 无障碍：
│   ├── 运动障碍用户能执行这个手势吗？
│   ├── 有VoiceOver/TalkBack替代方案吗？
│   └── 开关控制能用吗？
│
├── 冲突检查：
│   ├── 是否与系统手势冲突？
│   │   ├── iOS：边缘滑动返回
│   │   ├── Android：返回手势
│   │   └── Home指示器滑动
│   └── 与应用内其他手势一致吗？
│
└── 反馈：
    ├── 触觉反馈定义了吗？
    ├── 视觉反馈够吗？
    └── 需要音频反馈吗？
```

## 🎭 精神高于清单（移动端版）

### 通过清单不够！

| ❌ 自我欺骗 | ✅ 诚实评估 |
|-------------|-------------|
| "触控目标是44px"（但在边缘，够不到） | "用户能单手够到吗？" |
| "用了FlatList"（但没memo化） | "滚动流畅吗？" |
| "平台专属导航"（但只是图标不同） | "iOS有iOS的感觉，Android有Android的感觉吗？" |
| "有离线支持"（但错误消息是通用的） | "用户离线时实际能做什么？" |
| "有加载状态"（但只是个转圈） | "用户知道要等多久吗？" |

> 🔴 **通过清单不是目的。创造出色的移动端UX才是目的。**
