# 移动端测试模式

> **移动端测试 ≠ Web测试。约束不同，策略不同。**
> 本文件教授何时使用每种测试方法以及为什么。
> **代码示例精简——聚焦决策。**

---

## 🧠 移动端测试思维

```
移动端测试与Web的区别：
├── 真机至关重要（模拟器隐藏bug）
├── 平台差异（iOS与Android行为不同）
├── 网络条件变化剧烈
├── 测试中关注电池/性能
├── 应用生命周期（后台、被杀、恢复）
├── 权限和系统对话框
└── 触控交互 vs 点击
```

---

## 🚫 AI移动端测试反模式

| ❌ AI默认做法 | 问题 | ✅ 移动端正确做法 |
|---------------|------|-------------------|
| 仅Jest测试 | 遗漏原生层 | Jest + 设备E2E |
| Enzyme模式 | 已废弃，面向Web | React Native Testing Library |
| 浏览器E2E（Cypress） | 无法测试原生功能 | Detox / Maestro |
| 万物皆Mock | 遗漏集成bug | 真机测试 |
| 跳过平台测试 | iOS/Android不同 | 平台专属用例 |
| 跳过性能测试 | 移动端性能至关重要 | 低端设备上Profile |
| 只测正常路径 | 移动端有更多边缘情况 | 离线、权限、中断 |
| 追求100%单元覆盖率 | 虚假安全感 | 金字塔平衡 |
| 照搬Web测试模式 | 环境不同 | 移动端专属工具 |

## 1. 测试工具选择

### 决策树

```
你要测试什么？
        │
        ├── 纯函数、工具、辅助方法
        │   └── Jest（单元测试）
        │       └── 无需特殊移动端配置
        │
        ├── 单个组件（隔离）
        │   ├── React Native → React Native Testing Library
        │   └── Flutter → flutter_test（Widget测试）
        │
        ├── 带hooks、context、导航的组件
        │   ├── React Native → RNTL + Mocked Providers
        │   └── Flutter → integration_test包
        │
        ├── 完整用户流程（登录、结账等）
        │   ├── Detox（React Native，快速，可靠）
        │   ├── Maestro（跨平台，YAML配置）
        │   └── Appium（旧项目，慢，最后手段）
        │
        └── 性能、内存、电池
            ├── Flashlight（RN性能）
            ├── Flutter DevTools
            └── 真机Profiling（Xcode/Android Studio）
```

### 工具对比

| 工具 | 平台 | 速度 | 可靠性 | 适用场景 |
|------|------|------|--------|----------|
| **Jest** | RN | ⚡⚡⚡ | ⚡⚡⚡ | 单元测试、逻辑 |
| **RNTL** | RN | ⚡⚡⚡ | ⚡⚡ | 组件测试 |
| **flutter_test** | Flutter | ⚡⚡⚡ | ⚡⚡⚡ | Widget测试 |
| **Detox** | RN | ⚡⚡ | ⚡⚡⚡ | E2E、关键流程 |
| **Maestro** | 双端 | ⚡⚡ | ⚡⚡ | E2E、跨平台 |
| **Appium** | 双端 | ⚡ | ⚡ | 旧项目、最后手段 |

## 2. 移动端测试金字塔

```
                    ┌───────────────┐
                    │    E2E测试    │  10%
                    │  （真机）      │  慢、昂贵、不可或缺
                    ├───────────────┤
                    │   集成测试    │  20%
                    │               │  组件 + 上下文
                    ├───────────────┤
                    │   组件测试    │  30%
                    │               │  隔离UI
                    ├───────────────┤
                    │   单元测试    │  40%
                    │   （Jest）    │  纯逻辑
                    └───────────────┘
```

### 为什么是这个比例？

| 层级 | 原因 |
|------|------|
| **E2E 10%** | 慢、不稳定，但能发现集成bug |
| **集成 20%** | 测试真实用户流程，无需完整应用 |
| **组件 30%** | UI变更的快速反馈 |
| **单元 40%** | 最快、最稳定、逻辑覆盖 |

> 🔴 **如果90%是单元测试、0%是E2E，你测错了东西。**

## 3. 各层级测什么

### 单元测试（Jest）

```
✅ 测试：
├── 工具函数（formatDate、calculatePrice）
├── 状态reducer（Redux、Zustand store）
├── API响应转换器
├── 校验逻辑
└── 业务规则

❌ 不测：
├── 组件渲染（用组件测试）
├── 导航（用集成测试）
├── 原生模块（Mock它们）
└── 第三方库
```

### 组件测试（RNTL / flutter_test）

```
✅ 测试：
├── 组件正确渲染
├── 用户交互（点击、输入、滑动）
├── 加载/错误/空状态
├── 无障碍标签存在
└── Props变更行为

❌ 不测：
├── 内部实现细节
├── 万物皆快照（仅关键组件）
├── 样式细节（脆弱）
└── 第三方组件内部
```

### 集成测试

```
✅ 测试：
├── 表单提交流程
├── 屏幕间导航
├── 跨屏幕状态持久化
├── API集成（Mock服务器）
└── Context/Provider交互

❌ 不测：
├── 所有可能路径（用单元测试）
├── 第三方服务（Mock它们）
└── 后端逻辑（后端测试）
```

### E2E测试

```
✅ 测试：
├── 关键用户旅程（登录、购买、注册）
├── 离线 → 在线切换
├── Deep Link处理
├── 推送通知导航
├── 权限流程
└── 支付流程

❌ 不测：
├── 每个边缘情况（太慢）
├── 视觉回归（用快照测试）
├── 非关键功能
└── 纯后端逻辑
```

## 4. 平台专属测试

### iOS与Android有什么不同？

| 领域 | iOS行为 | Android行为 | 都要测？ |
|------|---------|-------------|----------|
| **返回导航** | 边缘滑动 | 系统返回键 | ✅ 是 |
| **权限** | 问一次，去设置 | 每次都问，显示理由 | ✅ 是 |
| **键盘** | 外观不同 | 行为不同 | ✅ 是 |
| **日期选择器** | 滚轮/模态 | Material对话框 | ⚠️ 自定义UI时 |
| **推送格式** | APNs载荷 | FCM载荷 | ✅ 是 |
| **Deep Link** | Universal Links | App Links | ✅ 是 |
| **手势** | 有独特手势 | Material手势 | ⚠️ 自定义时 |

### 平台测试策略

```
每个平台：
├── 运行单元测试（两端相同）
├── 运行组件测试（两端相同）
├── 在真机上运行E2E
│   ├── iOS：iPhone（不只是模拟器）
│   └── Android：中端设备（不是旗舰机）
└── 分别测试平台专属功能
```

## 5. 离线与网络测试

### 需要测试的离线场景

| 场景 | 验证什么 |
|------|----------|
| 离线启动应用 | 显示缓存数据或离线提示 |
| 操作中途断网 | 操作排队，不丢失 |
| 恢复网络 | 队列同步，无重复 |
| 慢速网络（2G） | 加载状态、超时正常工作 |
| 不稳定网络 | 重试逻辑、错误恢复 |

### 如何测试网络条件

```
方法：
├── 单元测试：Mock NetInfo，测试逻辑
├── 集成测试：Mock API响应，测试UI
├── E2E（Detox）：使用device.setURLBlacklist()
├── E2E（Maestro）：使用网络条件配置
└── 手动：使用Charles Proxy / Network Link Conditioner
```

## 6. 性能测试

### 测量什么

| 指标 | 目标 | 如何测量 |
|------|------|----------|
| **应用启动** | < 2秒 | Profiler、Flashlight |
| **屏幕转场** | < 300ms | React DevTools |
| **列表滚动** | 60 FPS | Profiler、手感 |
| **内存** | 稳定，无泄漏 | Instruments / Android Profiler |
| **包体积** | 最小化 | Metro bundler分析 |

### 何时做性能测试

```
性能测试时机：
├── 发布前（必须）
├── 添加重型功能后
├── 升级依赖后
├── 用户反馈慢时
└── CI中（可选，自动化基准测试）

测试环境：
├── 真机（必须）
├── 低端设备（Galaxy A系列、旧iPhone）
├── 不在模拟器上（性能数据不可靠）
└── 使用类生产数据（不是3条记录）
```

## 7. 无障碍测试

### 验证什么

| 元素 | 检查项 |
|------|--------|
| 交互元素 | 有accessibilityLabel |
| 图片 | 有替代文本或标记为装饰性 |
| 表单 | 标签关联到输入框 |
| 按钮 | Role = button |
| 触控目标 | ≥ 44×44（iOS）/ 48×48（Android） |
| 颜色对比度 | WCAG AA最低 |

### 如何测试

```
自动化：
├── React Native：jest-axe
├── Flutter：测试中的Accessibility检查器
└── 缺少标签的Lint规则

手动：
├── 开启VoiceOver（iOS）/ TalkBack（Android）
├── 用屏幕阅读器导航整个应用
├── 测试放大文字
└── 测试减少动效
```

## 8. CI/CD集成

### 各阶段运行什么

| 阶段 | 测试 | 设备 |
|------|------|------|
| **PR** | 单元 + 组件 | 无（快速） |
| **合并到main** | + 集成 | 模拟器/仿真器 |
| **发布前** | + E2E | 真机（设备农场） |
| **每夜** | 完整套件 | 设备农场 |

### 设备农场选项

| 服务 | 优势 | 劣势 |
|------|------|------|
| **Firebase Test Lab** | 免费额度，Google设备 | 偏Android |
| **AWS Device Farm** | 选择广泛 | 贵 |
| **BrowserStack** | 体验好 | 贵 |
| **本地设备** | 免费，可靠 | 品种有限 |

## 📝 移动端测试检查清单

### PR前
- [ ] 新逻辑有单元测试
- [ ] 新UI有组件测试
- [ ] 测试中无console.log
- [ ] CI测试通过

### 发布前
- [ ] 真机iOS上跑过E2E
- [ ] 真机Android上跑过E2E
- [ ] 低端设备上测试过
- [ ] 离线场景已验证
- [ ] 性能可接受
- [ ] 无障碍已验证

### 可以有意识跳过的
- [ ] 100%覆盖率（追求有意义的覆盖）
- [ ] 每种视觉排列（少量使用快照）
- [ ] 第三方库内部
- [ ] 后端逻辑（单独测试）

## 🎯 测试前先回答

1. **什么可能坏？** → 测那个
2. **什么对用户关键？** → E2E测那个
3. **什么逻辑复杂？** → 单元测那个
4. **什么是平台专属的？** → 双平台都测
5. **离线会怎样？** → 测那个场景

> **切记：** 好的移动端测试是测对的东西，不是测所有东西。一个不稳定的E2E测试比没有测试更糟。一个能发现bug的失败单元测试，胜过100个通过的无关测试。
