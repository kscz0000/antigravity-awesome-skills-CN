# 移动端导航参考

> 导航模式、Deep Link、返回处理与Tab/Stack/Drawer决策。
> **导航是应用的骨架——搞错了，一切都会感觉不对。**

---

## 1. 导航选择决策树

```
什么类型的应用？
        │
        ├── 3-5个顶级分区（同等重要）
        │   └── ✅ Tab栏 / 底部导航
        │       示例：社交、电商、工具
        │
        ├── 深层分级内容（逐级下钻）
        │   └── ✅ Stack导航
        │       示例：设置、邮件文件夹
        │
        ├── 多个目的地（5个以上顶级）
        │   └── ✅ 抽屉导航
        │       示例：Gmail、复杂企业应用
        │
        ├── 单一线性流程
        │   └── ✅ 仅Stack（向导/引导页）
        │       示例：结账、设置流程
        │
        └── 平板/折叠屏
            └── ✅ Navigation Rail + 列表-详情
                示例：iPad上的邮件、笔记
```

## 2. Tab栏导航

### 何时使用

```
✅ 使用Tab栏的情况：
├── 3-5个顶级目的地
├── 目的地同等重要
├── 用户频繁切换
├── 每个Tab有独立的导航栈
└── 应用在短会话中使用

❌ 避免Tab栏的情况：
├── 超过5个目的地
├── 目的地有明显层级
├── 各Tab使用频率差异很大
└── 内容按顺序流转
```

### Tab栏最佳实践

```
iOS Tab栏：
├── 高度：49pt（含Home指示器83pt）
├── 最多项数：5
├── 图标：SF Symbols，25×25pt
├── 标签：始终显示（无障碍）
├── 选中指示：主题色

Android底部导航：
├── 高度：80dp
├── 最多项数：5（3-5最理想）
├── 图标：Material Symbols，24dp
├── 标签：始终显示
├── 选中指示：药丸形状 + 填充图标
```

### Tab状态保持

```
规则：每个Tab维护自己的导航栈。

用户旅程：
1. 首页Tab → 下钻到商品 → 加入购物车
2. 切换到个人Tab
3. 切回首页Tab
→ 应回到"加入购物车"页面，而非首页根

实现：
├── React Navigation：每个Tab有自己的导航器
├── Flutter：IndexedStack保持状态
└── 切换Tab时绝不重置栈
```

## 3. Stack导航

### 核心概念

```
栈隐喻：卡片叠放

Push：在顶部添加屏幕
Pop：移除顶部屏幕（返回）
Replace：替换当前屏幕
Reset：清空栈，设新根

视觉：新屏幕从右侧滑入（LTR）
返回：屏幕向右滑出
```

### Stack导航模式

| 模式 | 用途 | 实现 |
|------|------|------|
| **简单栈** | 线性流程 | 逐步Push |
| **嵌套栈** | 有子导航的分区 | Tab内嵌Stack |
| **模态栈** | 聚焦任务 | 模态弹出 |
| **认证栈** | 登录 vs 主界面 | 条件根 |

### 返回按钮处理

```
iOS：
├── 从左边缘滑动（系统级）
├── 导航栏返回按钮（可选）
├── 交互式pop手势
└── 没有充分理由绝不覆盖滑动返回

Android：
├── 系统返回按钮/手势
├── 工具栏Up按钮（可选，用于下钻）
├── 预测性返回动画（Android 14+）
└── 必须正确处理返回（Activity/Fragment）

跨平台规则：
├── 返回始终向上回退栈
├── 绝不劫持返回做其他用途
├── 丢弃未保存数据前确认
└── Deep link应允许完整回退
```

## 4. 抽屉导航

### 何时使用

```
✅ 使用抽屉的情况：
├── 超过5个顶级目的地
├── 较少访问的目的地
├── 功能众多的复杂应用
├── 导航中需要品牌/用户信息
└── 平板/大屏幕使用持久抽屉

❌ 避免抽屉的情况：
├── 5个或更少目的地（用Tab）
├── 所有目的地同等重要
├── 移动优先的简单应用
└── 可发现性至关重要（抽屉是隐藏的）
```

### 抽屉模式

```
模态抽屉：
├── 覆盖内容打开（背景遮罩）
├── 从边缘滑动打开
├── 汉堡图标（ ☰ ）触发
└── 移动端最常见

持久抽屉：
├── 始终可见（大屏幕）
├── 内容移位
├── 适合效率应用
└── 平板、桌面

Navigation Rail（Android）：
├── 窄竖条
├── 图标 + 可选标签
├── 竖屏平板使用
└── 80dp宽度
```

## 5. 模态导航

### 模态 vs Push

```
Push（栈）：                      模态：
├── 水平滑入                      ├── 垂直向上滑入（Sheet）
├── 层级的一部分                  ├── 独立任务
├── 返回键回退                    ├── 关闭（X）回退
├── 同一导航上下文                ├── 独立导航上下文
└── "下钻"                       └── "聚焦任务"

模态适用于：
├── 创建新内容
├── 设置/偏好
├── 完成交易
├── 自包含工作流
├── 快速操作
```

### 模态类型

| 类型 | iOS | Android | 用途 |
|------|-----|---------|------|
| **Sheet** | `.sheet` | Bottom Sheet | 快速任务 |
| **全屏** | `.fullScreenCover` | Full Activity | 复杂表单 |
| **警告** | Alert | Dialog | 确认操作 |
| **操作表** | Action Sheet | Menu/Bottom Sheet | 选项选择 |

### 模态关闭

```
用户期望通过以下方式关闭模态：
├── 点击X / 关闭按钮
├── 下拉滑动（Sheet）
├── 点击遮罩（非关键时）
├── 系统返回（Android）
├── 硬件返回键（旧Android）

规则：仅在有未保存数据时阻止关闭。
```

## 6. Deep Link

### 为什么从第一天就规划Deep Link

```
Deep Link支持：
├── 推送通知导航
├── 分享内容
├── 营销活动
├── Spotlight/搜索集成
├── 小组件导航
├── 外部应用集成

事后补建很困难：
├── 需要重构导航
├── 屏幕依赖关系不清
├── 参数传递复杂
└── 始终在开始时规划Deep Link
```

### URL结构

```
Scheme://host/path?params

示例：
├── myapp://product/123
├── https://myapp.com/product/123（Universal/App Link）
├── myapp://checkout?promo=SAVE20
├── myapp://tab/profile/settings

层级应匹配导航：
├── myapp://home
├── myapp://home/product/123
├── myapp://home/product/123/reviews
└── URL路径 = 导航路径
```

### Deep Link导航规则

```
1. 完整栈构建
   Deep link到 myapp://product/123 应：
   ├── 将首页放在栈底
   ├── Push商品页面到顶部
   └── 返回按钮回到首页

2. 认证感知
   如果Deep Link需要认证：
   ├── 保存目标目的地
   ├── 重定向到登录
   ├── 登录后导航到目标

3. 无效链接
   如果Deep Link目标不存在：
   ├── 导航到降级页（首页）
   ├── 显示错误消息
   └── 绝不崩溃或白屏

4. 有状态导航
   活跃会话中的Deep Link：
   ├── 不要清空当前栈
   ├── Push到顶部 或
   ├── 询问用户是否导航离开
```

## 7. 导航状态持久化

### 需要持久化的

```
应当持久化：
├── 当前Tab选择
├── 列表滚动位置
├── 表单草稿数据
├── 近期导航栈
└── 用户偏好

不应持久化：
├── 模态状态（对话框）
├── 临时UI状态
├── 过期数据（返回时刷新）
├── 认证状态（用安全存储）
```

### 实现

```javascript
// React Navigation - 状态持久化
const [isReady, setIsReady] = useState(false);
const [initialState, setInitialState] = useState();

useEffect(() => {
  const loadState = async () => {
    const savedState = await AsyncStorage.getItem('NAV_STATE');
    if (savedState) setInitialState(JSON.parse(savedState));
    setIsReady(true);
  };
  loadState();
}, []);

const handleStateChange = (state) => {
  AsyncStorage.setItem('NAV_STATE', JSON.stringify(state));
};

<NavigationContainer
  initialState={initialState}
  onStateChange={handleStateChange}
>
```

## 8. 转场动画

### 平台默认值

```
iOS转场：
├── Push：从右滑入
├── 模态：从底部滑入（Sheet）或淡入
├── Tab切换：交叉淡入
├── 交互式：滑动返回

Android转场：
├── Push：淡入 + 从右滑入
├── 模态：从底部滑入
├── Tab切换：交叉淡入或无
├── 共享元素：Hero动画
```

### 自定义转场

```
何时自定义：
├── 品牌识别需要
├── 共享元素关联
├── 特殊揭示效果
└── 保持微妙，<300ms

何时用默认：
├── 大多数时候
├── 标准下钻
├── 平台一致性
└── 性能关键路径
```

### 共享元素转场

```
连接屏幕间的元素：

屏幕A：商品卡片带图片
            ↓（点击）
屏幕B：商品详情带同一图片（放大）

图片从卡片位置动画到详情位置。

实现：
├── React Navigation：shared element库
├── Flutter：Hero widget
├── SwiftUI：matchedGeometryEffect
└── Compose：Shared element transitions
```

## 9. 导航反模式

### ❌ 导航罪过

| 反模式 | 问题 | 解决方案 |
|--------|------|----------|
| **返回不一致** | 用户困惑，无法预测 | 始终pop栈 |
| **隐藏导航** | 功能不可发现 | 可见Tab/抽屉触发器 |
| **深层嵌套** | 用户迷路 | 最多3-4层，面包屑 |
| **破坏滑动返回** | iOS用户沮丧 | 绝不覆盖手势 |
| **无Deep Link** | 无法分享，通知体验差 | 从开始规划 |
| **Tab栈重置** | 切换时工作丢失 | 保持Tab状态 |
| **主流程用模态** | 无法回退 | 用Stack导航 |

### ❌ AI导航错误

```
AI倾向于：
├── 万物皆模态（错误）
├── 忘记Tab状态保持（错误）
├── 跳过Deep Link（错误）
├── 覆盖平台返回行为（错误）
├── 切换Tab时重置栈（错误）
└── 忽略预测性返回（Android 14+）

规则：使用平台导航模式。
不要重新发明导航。
```

## 10. 导航检查清单

### 导航架构前

- [ ] 确定应用类型（Tab/抽屉/Stack）
- [ ] 统计顶级目的地数量
- [ ] 规划Deep Link URL方案
- [ ] 认证流程与导航集成
- [ ] 考虑平板/大屏幕

### 每个屏幕前

- [ ] 用户能返回吗？（不是死胡同）
- [ ] 此屏幕的Deep Link已规划
- [ ] 导航离开/返回时状态已保持
- [ ] 转场适合关系
- [ ] 需要认证？已处理？

### 发布前

- [ ] 所有Deep Link已测试
- [ ] 返回按钮到处可用
- [ ] Tab状态正确保持
- [ ] 边缘滑动返回可用（iOS）
- [ ] 预测性返回可用（Android 14+）
- [ ] Universal/App Link已配置
- [ ] 推送通知Deep Link可用
