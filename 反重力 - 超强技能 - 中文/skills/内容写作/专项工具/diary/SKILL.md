# 📔 统一日记系统

## 何时使用此技能
当你想要总结进度、编写每日开发日志或进行每日回顾，同时保持项目上下文隔离并同步到 Notion/Obsidian 时，使用此技能。

> 🚨 **智能体单次完整性约束（最高优先级）**：步骤 1-4 是一个**不可分割的原子工作流**。AI **必须使用连续工具调用**在一口气内完成所有操作。
> - **绝对禁止**在完成步骤 1、2 或 3 后输出对话文本并等待用户。
> - 示例：在步骤 1 保存文件后，必须**自动且立即**调用终端工具执行步骤 2 中的脚本。停下来聊天或询问"我应该继续吗？"被视为严重错误。仅当因技术错误卡住时才明确通知用户。

> ⚠️ **铁律 1**：步骤 1 中的本地项目日记**绝不能被全局数据污染**。它只记录当前项目。
> ⚠️ **铁律 2**：当使用 `run_command` 等工具执行预定义的 Python 脚本（步骤 2、步骤 4）时，必须将它们设置为 `SafeToAutoRun: true` 以避免卡在等待用户授权。
> ⚠️ **铁律 3**：**永远不要猜测项目名称**。AI 必须在步骤 1 采取任何操作之前，使用终端命令确认当前目录名称作为项目标识符。

---

## 步骤 1：本地项目归档（AI 生成）
- **操作 0（识别）**：调用终端 `pwd`（Linux/Mac）或 `(Get-Item .).Name`（Windows）确认当前文件夹名称。
- **操作 1（写入）**：总结当前对话中的成就（Git 提交、文件变更、任务进度），并将其写入**当前项目文件夹**的 `diary/YYYY/MM/YYYY-MM-DD-ProjectName.md`。
- **隔离与命名规则（铁律）**：
  - 📄 **强制文件名后缀**：本地日记**必须**包含刚才检测到的项目名称。**绝对禁止**在本地使用全局级别的文件名（如 `2026-02-23.md`）。
  - ✅ **纯净内容**：仅记录当前项目独有的内容。不要混入其他项目。
  - 📝 **追加模式**：如果项目日记已存在，使用"追加"方式更新，切勿覆盖原始内容。
  - 📁 **自动创建**：根据年月创建子文件夹 `diary/YYYY/MM/`。
  - ⚡ **强制继续**：写入完成后，**不要中断对话；立即调用终端工具并进入步骤 2。**

## 步骤 1.5：刷新项目上下文（自动化脚本）
- **前提条件**：你已确认当前项目目录路径（来自操作 0 的 `pwd` 结果）。
- **操作**：调用终端执行以下命令，自动扫描项目状态并生成/更新 `AGENT_CONTEXT.md`：
  ```powershell
  python {diary_system_path}/scripts/prepare_context.py "<Project_Root_Path>"
  ```
- **SafeToAutoRun**：true（安全操作；纯读写本地文件）。
- **结果**：项目目录中的 `AGENT_CONTEXT.md` 刷新到最新状态。
- **完成后**：强制继续到步骤 2；不要等待用户确认。

## 步骤 2：提取全局与项目素材（脚本执行）
- **操作**：调用提取脚本，**传入步骤 1 刚写入的项目日记的绝对路径**。脚本将精确打印"今日全局进度"和"当前项目进度"。
- **执行命令**：
  ```powershell
  python {diary_system_path}/scripts/fetch_diaries.py "<Absolute_Path_to_Step1_Project_Diary>"
  ```
- **结果**：终端将并排打印两组素材。AI 必须直接读取终端输出并准备心智融合。

## 步骤 3：AI 智能融合与全局归档（AI 执行）🧠
- **操作**：基于步骤 2 终端打印的两组素材，在心智中完成**无缝融合**，然后写入全局日记：`{diary_system_path}/diary/YYYY/MM/YYYY-MM-DD.md`。
- **上下文防火墙（核心机制）**：
  1. **无标签漂移**：读取"全局进度素材"时，可能包含其他项目的进度。**严禁将今日对话成就归类到属于其他项目的现有项目标题下。**
  2. **优先级定义**：步骤 2 中标记为 `📁 [当前项目最新进度]` 的内容是今日日记的主角。
- **重写规则**：
  1. **安全优先**：如果全局日记"已存在"，保留原始内容并追加/融合新项目进度。**不要覆盖。**
  2. **精确分区**：确保为此项目设置专门的 `### 📁 ProjectName` 区域。不要将内容混入其他项目区域。
  3. **经验总结**：合并并去重；为每条记录附加行动项。
  4. **清理**：全局写入或融合后，你**必须**强制删除创建的任何临时文件以避免编码问题（如 `temp_diary.txt`、`fetched_diary.txt`），保持工作区整洁。

## 步骤 4：云端同步与经验提取（脚本 + 人工）🛑
- **操作 1（同步）**：调用主脚本将全局日记推送到 Notion 和 Obsidian。
- **执行命令**：
  ```powershell
  python {diary_system_path}/scripts/master_diary_sync.py --sync-only
  ```
- **操作 2（提取与强制暂停）**：
  1. AI 从全局日记中提取"改进与学习"。
  2. 确认是否包含过去完全缺失的关键点（📌 新规则），或更好的方法（🔄 进化规则）。
  3. 列出结果并**等待用户确认**（用户说"执行"或"同意"）。
  4. 用户确认后，更新 `{Knowledge_Base_Path}/` 中的 `.md` 文件并执行 `qmd embed`（如适用）。

---
**🎯 任务验收标准**：
1. ✅ 项目本地日记已生成（无污染）。
2. ✅ 已使用绝对路径调用 `fetch_diaries.py` 并成功打印素材。
3. ✅ AI 执行高质量重写并精确写入全局日记（如文件存在则成功追加）。
4. ✅ `--sync-only` 成功推送到 Notion + Obsidian。
5. ✅ 经验提取已呈现给用户并获得授权。

## 📝 模板与写作指南

在步骤 1（本地）和步骤 3（全局融合）中严格应用以下 Markdown 模板以确保清晰。

### 💡 写作指南（供 AI）
1. **动态替换**：模板中的 `{Project Name}` 必须严格使用步骤 1 中 `pwd` 获取的文件夹名称。
2. **简洁去重**：在步骤 3 编写全局日记时，AI 必须精简本地日记中的"🛠️ 执行细节"。全局日记仅关注"总体方向与输出结果"。
3. **强制复选框**：所有"下一步"和"行动项"必须使用 Markdown `* [ ]` 格式，以便后续在 Obsidian/Notion 中勾选。

### 📝 模板 1：项目本地日记（步骤 1 专用）

```markdown
# 项目开发日志：{Project Name}
* **📅 日期**：YYYY-MM-DD
* **🏷️ 标签**：`#Project` `#DevLog`

---

> 🎯 **进度摘要**
> （简述完成的核心任务，例如："完成 auto-video-editor 的 Google Colab 环境测试"）

### 🛠️ 执行细节与变更
* **Git 提交**：（如有请列出）
* **核心文件修改**：
  * 📄 `路径/文件名`：变更说明。
* **技术实现**：
  * （记录关键逻辑或架构结构变更）

### 🚨 问题排查
> 🐛 **遇到的问题**：（如：API 错误、包冲突）
> 💡 **解决方案**：（最终修复方式，保留关键命令）

### ⏭️ 下一步
- [ ] （具体任务 1）
- [ ] （具体任务 2）
```
