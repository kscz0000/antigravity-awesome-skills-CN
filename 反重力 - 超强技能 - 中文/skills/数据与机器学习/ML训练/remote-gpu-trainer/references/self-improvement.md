# 随时间改进 — 捕获新陷阱 + 个性化（不损坏技能）

本技能是静态参考，因此它**不会**自行进化。但每次真实运行都会教会一些东西 — 一个新的平台怪癖、一个目录中没有的训练 bug、或用户自己的设置。本文件是在**正确位置、以正确门槛沉淀知识、而不静默改写技能**的协议。每当运行暴露了目录尚未覆盖的内容时，应用此协议。

快速跳转：`grep -in '<keyword>' references/self-improvement.md`。

## 目录
1. 门槛 — 什么符合可保留的陷阱
2. 路由学习 — 用户记忆 vs 目录 vs 上游 PR
3. 提议，不要自动编辑
4. 首次运行个性化 — 将用户设置捕获到记忆
5. 新鲜度 — 平台事实会过期；打标并重新验证

## 1. 门槛 — 不要将偶发事件神圣化

一个令人意外的失败是一个**假设，不是陷阱**（原则 #3；**必需：** `verifying-dl-experiments`）。仅当以下三条全部满足时才沉淀新陷阱：

- **已根因分析** — 机制已被理解，不是仅"重试后就好了"。
- **已复现或机制明确** — 不是单次偶发事件（瞬态网络抖动不是陷阱）。
- **可泛化** — 另一个用户在此平台 / 训练设置下也会遇到。

如果未达门槛，则是*项目笔记*（→ §4 记忆），不是目录条目。将未验证的偶发事件神圣化正是此门槛要防止的目录腐化。

## 2. 路由学习

| 学到了什么 | 去哪里 | 形式 |
|---|---|---|
| **用户/个人/主机特定** — 此账户的怪癖、用户的偏好、常用的 GPU 计划、"在我的机器上 X 为真" | **主机的 `memory/` 系统**（主机特定、个人、可能临时） | 一条 `reference` / `project` / `feedback` 事实，每文件一条，去重 |
| **项目级事实或用户在其项目上反复遇到的错误** — 一个配置怪癖、"总是先运行 X"、必须为 Y 的路径、必须激活的环境 | 用户仓库中的**项目指令文件** — `CLAUDE.md` / `AGENTS.md` / `.cursorrules`（跨会话、跨工具、对协作者持久，不像主机记忆） | 短命令式规则；**提议，不要自动写入**（§3） |
| **可泛化的平台陷阱** | `profiles/<platform>.md` §7（平台限定）或 `references/gotchas_universal.md`（跨平台） | `症状 → 根因 → 修复` + 来源 URL |
| **可泛化的训练调试陷阱** | `references/training/<topic>.md` | 同样形式 |
| **纠正**（此处的某个事实现在错误/过时） | 编辑该文件；重新打标其 `verified <month>` | 注明旧 → 新 + URL |

因为本技能是开源的，可泛化的新增/纠正也是**上游 PR** 的候选 — 主动提议开一个，让每个用户受益，不仅是本安装。

## 3. 提议，不要自动编辑

**永远不要从单次运行静默改写技能文件** — 错误的"修复"或破坏的结构比缺失的条目更糟。相反：

- 起草条目（`症状 → 根因 → 修复`，附来源）并**展示给用户审批。**
- 对于超出范围或更大的改动，分拆出去（主机的任务 / PR 机制），而不是膨胀当前运行。
- 仅在用户同意后应用；然后重新运行技能自身的检查 — 交叉引用可解析、**无隐秘值写入**、结构 + 目录完好。

## 4. 首次运行个性化 → 记忆

本技能第一次为用户运行时，将其**实际设置**捕获到记忆中，使后续运行预参数化而非重复询问：

- 他们租用的平台，以及每平台 §8 SCRIPT OVERRIDES 中有效的值（路径、代理钩子、凭证位置）；
- 项目仓库路径 / 训练入口点 / 配置布局；
- tracker 实体（wandb / trackio）以及**密钥存放位置**（其环境变量名或文件路径 — 永远不是值本身）；
- 常用 GPU 计划 + 磁盘预算。

存储为 `project` / `reference` 记忆。仅记录凭证的*名称或路径*，永远不记录密钥本身。下次会话时 Profile 覆盖 + `run_one` 参数自动填充。

## 5. 新鲜度 — 本技能有保质期

平台价格、计费动词和限制**会变**。每个平台事实在编写时标注 `verified <month>`。在将**金钱或数据**押在拆除/计费事实上之前 — 不可逆的那些（`terminate` / `destroy` / release）— **在同一会话中根据平台当前文档重新验证。** 如果事实过时，修复它（§2 纠正行）并重新打标日期。每季度重新验证每个 Profile 的 §5 TEARDOWN/BILLING 节可保持高风险事实的诚实 — 可通过主机的 `/schedule` 调度。运行 `scripts/check_staleness.py` 列出所有超过 N 个月的 `verified` 标记（这是一个关于*什么*需要重新检查的机械提醒 — 它不验证事实本身）。
