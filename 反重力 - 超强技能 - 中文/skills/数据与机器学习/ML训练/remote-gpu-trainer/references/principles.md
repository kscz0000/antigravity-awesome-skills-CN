# 操作原则 — 10 条不变量（展开）

这些是每个阶段和陷阱背后的*原因*。它们在任何**计费、隔离、租赁的 GPU** 上都成立
— AutoDL、RunPod、vast.ai、Lambda、Paperspace、中国平台、裸 SSH 主机、Slurm 或 K8s。
只有具体路径/CLI 不同（那些在 `profiles/<platform>.md` 中）。内化这些；具体操作方法随之而来。一行式版本在 `SKILL.md` 中；本文件承载跨平台的细微差别。

跳转：`grep -n '^## ' references/principles.md`。

---

## 1. 最小化付费挂钟时间

计费器在机器启动的*整个*时间运行，不只是 GPU 计算时。三个推论：
在租用前**本地 CPU 上冒烟测试**正确性（原则 #2）；**分离启动并交还控制权**而非看护一个阻塞的 `sleep`；验证通过后**立即释放**（原则 #9 掌管*谁决定*）。每一个空闲的付费分钟 — 一个卡住的下载、一个过夜忘记关的机器、一个需要人工介入的暂停 — 都是钱。

*通用。* 即使在 Slurm 上，"计费器"是挂钟时间/fairshare 配额而非美元，同样的纪律也适用：不要让分配空闲着。

---

## 2. 便宜检查先于昂贵计算

CPU 冒烟测试（1–2 个 batch，logger 关闭，微小 shape）几乎免费地消灭 import 错误、配置漂移、tensor-shape 和度量**尺度** bug，**在它们消耗 GPU 小时之前**。它是*必要但不充分*的 — 它不会捕获收敛问题 — 但它捕获那些否则只有在实例启动后才会暴露的愚蠢且昂贵的失败。

*边界：* 本技能拥有*何时*运行冒烟测试（租用前门控）。冒烟测试的*内容* — 断言什么、如何缩小问题 — 属于 **`verifying-dl-experiments`**。不要在这里重复。

---

## 3. 信任你加载过的产物，不是声称成功的日志行

"synced / saved / done / 100% complete" 是一个**声明**，声明在静默写入失败时会撒谎 — 磁盘满、inode 耗尽、一个被吞掉的错误、一个半上传的 blob。在释放唯一副本之前确认文件**存在且可加载**。

**监视器自身状态也是声明**，不是事实真相。一个你取代了其作业的异步条件等待器会轮询一个永远不会到达的标记（一个永远循环的僵尸）。一个会话范围的监视器在上下文重置时消亡而作业仍在运行。对照作业的*真实*进程和产物
（`tmux ls` / `squeue` / `pgrep`，输出 `mtime`，一次加载测试）协调监视器，当你取代其作业时拆除监视器，并将监视器的生命周期匹配等待的持续时间。

> **此原则依赖的监控物理：** 前台 Bash 硬性上限 600 秒（长前台等待在 10 分钟时被杀死）；
> `run_in_background` **没有**上限并在退出时通知；一个从不*退出*的监视器永不通知；
> 一个未加引号的 `|` 在轮询正则内部会分裂为管道命令，第一个读取 stdin → 永远挂起。参见 `references/monitoring_patterns.md`。

*通用 — 承载脊柱。* 它是 `superpowers:verification-before-completion` 铁律的平台实例（"没有新鲜验证证据就没有完成声明"）。与 `verifying-dl-experiments` 共享。

## 4. 知道什么在 stop 与 destroy 后存活

**最大的可移植性陷阱。** AutoDL 在关机后保留 `/root` — 所以 AutoDL 习惯是"关机就行，数据好好的。" 这个假设在**几乎所有其他地方都是错误的**：

- **RunPod** stop 后擦除*容器磁盘*；只有*卷磁盘*（`/workspace`）在 stop 后存活，且只有 **Network Volume** 在 terminate 后存活。
- **vast.ai** stop 后保留磁盘但**永远计费**；destroy 丢失一切。
- **K8s** 每次重新调度时擦除 pod 文件系统，除非挂载了 PVC。
- **Colab** 断开后丢失 `/content` 和 RAM。

所以原则不是一个路径 — 它是一个**纪律**：对每个平台，在 Phase 0 之前，读取 profile 的存储生存矩阵，将检查点写入在你打算使用的拆除动词后仍存活的挂载点。你最常需要的数据默认位于*易失*层。

*混合：* *规则*是通用的；*哪个挂载点*的值是一个 profile 事实。

---

## 5. 存储在你未监控的维度 — 和位置 — 上失败

磁盘在 **inode 先于字节**死亡（`df -h` 显示 34% 而 `cp` 因 "No space left" 失败，因为 `df -i` 已达 100% — 在一个充满许多小文件 eval 输出的共享 FS 上是经典情况）。真正的空间占用者常常隐藏在你看不到的地方 — 一个**符号链接缓存**（`~/.cache/huggingface` 映射到数据盘上）可能超过你创建的 `runs/`。**用 `du` 审计实际挂载点，而非假设。** 按**价值**清理：保留微小不可替代的证据（metric/eval JSON），丢弃大量可复现的临时数据（周期性检查点、未使用的模型缓存 — 一次观察到的 sweep 留下了 **179 GB** 的被取代 `latest.pt`/`epoch_*.pt`，而真正的证据仅 **<200 MB** 的 JSON）。预先计算预算；监控 `df -i`，不只是 `df -h`。

*混合：* inode 上限*数字*是 profile 事实（AutoDL/中国强制约 200K；RunPod/vast/Lambda 规定 GB 配额，无文档化 inode 上限）。"审计真实挂载点，按价值清理"的纪律是核心。多小文件陷阱的通用形式是**分片为 tar**（WebDataset）— 参见 `references/gotchas_universal.md` U25。

---

## 6. 永远不要在活跃运行下修改输入

运行中的作业按字节偏移量将其脚本**保持在内存中**。tmux 保持 `run_queue.sh` 的加载状态；bash 通过寻址到保存的偏移量读取脚本，所以运行中 `scp` 新版本会让 bash 落在*不同*文件的中间并重新执行代码块（重复运行、停滞的队列）。版本化文件名；只在没有进程读取时编辑（`pgrep -af <script>` 为空）。

*通用 — 纯 bash/tmux 物理规律。* 在每个 SSH 后端上完全相同。

## 7. 为重试设计 — 失败是概率性的，传输是脆弱的，镜像是路由特定的

一部分相同启动会死亡（`wandb.init` 期间的网络抖动、瞬态内核故障、spot 抢占）。封装器必须是**幂等且可恢复的**；重试**相同配置**而非手动修补一次运行（这会破坏可比性 — 参见 `verifying-dl-experiments`）。

**批量传输是典型的脆弱步骤：** 用 `timeout`+恢复重试循环包装 — 停滞≠永久失败，可恢复的下载在多次杀死间积累进度。一个加速**镜像/代理/缓存只加速一条路由，不是全部** — 它可能覆盖 metadata/API 路径而 bulk-data 路径（CDN/blob 后端）仍然失败，且一个*国内*源通过*国外*加速代理路由反而更慢。将路由匹配源；在真实传输使用的**相同路由**上验证速度测试（对代理传输做无代理探测测量不了任何东西）。

*通用。* **spot/抢占**子情况是 profile 参数化的（在 vast/RunPod 上是核心；在 Lambda/Paperspace/中国上中断是 auto-shutdown/auto-release/容量不足）— 参见原则 #8 和 `references/spot-resilience.md`。

## 8. 检查点到持久存储 + 幂等恢复是通用脊柱

分离作业是必要的但不充分的。**唯一**能在每种失败模式下存活的机制 — SSH 断开、Slurm 挂钟时间杀死、K8s pod 重新调度、spot 抢占、Colab 断开 — 是：

1. **在周期性定时器上将完整状态检查点到平台的持久位置**（模型 + 优化器 +
   LR-scheduler + epoch/step + RNG + dataloader 位置），**原子地**写入（`tmp`→`fsync`→
   `os.rename`），使得写入中途被杀死永远不会损坏最新的良好检查点。
2. **启动时无条件加载最新状态**，这样*相同的启动命令*恢复而非重新开始。这正是原则 #7 "重试相同配置"实际恢复进度的方式。

**分离原语是可插拔的** — 裸机上的 tmux，Slurm 上的 `sbatch --requeue`，K8s 上的 Job 清单，Kaggle 上的 Save&Run 提交，Colab 上的 checkpoint-to-Drive 循环。Checkpoint+resume 是所有这些之下不变的东西。

*通用。* 频率是公式，不是猜测 — Young/Daly `W = √(2·μ·C)`（μ = 两次抢占间的平均时间，C = 检查点写入时间）；向下取整到迭代边界。托管框架（SkyPilot Managed Jobs、SageMaker）替您移动机器但**从头重启进程 — 您的检查点加载才是恢复进度的东西。** 详情 + 计算示例在 `references/spot-resilience.md`。

## 9. 成本和破坏性操作由用户决定

永远不要自动释放/终止实例，永远不要在未经明确确认的情况下删除持久/共享文件，且如果您自己的清理无法释放足够空间，**要求扩展磁盘**（说明需要的 GB 数）而非静默缩小实验（更少的种子、更小的 eval、上限化的可视化）。

在多平台场景下这被强化而非软化：在 RunPod/vast/Lambda 上，停止计费的操作是**不可逆的** `terminate`/`destroy`，会删除磁盘 — 所以确认门控*更加*重要。将其操作化为**拆除铁律**（SKILL.md Phase 5）：检查点 pull 到本地并通过加载验证且用户批准具体影响成本的操作之前，不得拆除。

*通用。* 共享 FS 也是多项目的：在您项目自己的文件夹内工作，只删除您自己的冗余，永远不要删除您未创建的顶级目录。

## 10. 教用户平台，不只是操作它

大多数用户 — 尤其是只偶尔租用某个平台的用户 — 不知道其非显而易见的**便利设施**或**危险时钟**，技能的工作不只是操作机器，而是*告诉他们*。首次接触平台时，主动展示：
- **他们否则会错过的便利设施：** 一键 SSH 密钥注册（这样智能体可以非交互连接）、GPU 可用性通知、内置面板（JupyterLab / TensorBoard 磁贴）。
- **导致数据或金钱损失的危险时钟：** *已停止*实例上的自动释放/自动删除定时器
  （AutoDL 在**15 天**后释放关机的机器 → 数据磁盘消失；多个中国平台约 10 天）、一个仍持续计费的 stop（vast.ai 永远计费，RunPod 2 倍）、低余额/欠费清除。

每个平台的列表位于各 profile 的**前置告知用户**块中。这与 #9 配对：#9 阻止智能体*做*危险的事；#10 让智能体在危险时钟触发前*警告人类*。租赁硬件上最昂贵的意外是静默定时器（一个停泊的机器被释放、一个停止的磁盘仍在计费），而非可见的失败 — 提前展示它们是最便宜的保险。
