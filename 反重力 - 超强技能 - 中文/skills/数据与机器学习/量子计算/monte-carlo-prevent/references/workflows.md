# 工作流详情

每个 Monte Carlo Prevent 工作流的详细分步说明。
这些内容从主 SKILL.md 引用 — 执行工作流时参考对应章节。

---

## 工作流 1：表健康度检查 — 打开或编辑模型时

用户打开 dbt 模型或提及表时，自动运行此序列：

```
1. search(query="<table_name>") → 获取完整 MCON/表标识符
2. getTable(mcon="<mcon>") → schema、新鲜度、行数、重要性评分、监控状态
3. getAssetLineage(mcon="<mcon>") → 上游来源、下游依赖
4. getAlerts(created_after="<7天前>", created_before="<现在>", table_mcons=["<mcon>"]) → 活跃告警
```

为用户总结：
- **健康度**：最后更新时间、行数、是否被监控？
- **血缘**：N 个上游来源，M 个下游消费者（列出重要者）
- **告警**：任何活跃/未确认的事件 — 若存在则优先展示
- **风险信号**（轻量）：标记重要性评分高、关键资产在下游、或告警已触发 — 这些表明修改前需格外谨慎

打开 dbt 模型文件时主动提供的示例摘要：
> "表 `orders_status` 最后更新于 2 小时前，共 142K 行。有 3 个下游依赖，包括 `order_status_snapshot`（关键资产）。存在 2 个活跃新鲜度告警 — 修改前需格外谨慎。需要运行完整的变更影响评估吗？"

**自动升级规则 — 完成上述步骤 1-4 后：**

首先，检查用户在本会话中是否表达了修改模型的意图（如提及变更、要求添加/编辑/修复某项）。

若已表达变更意图，且以下任一条件成立：
  - 表上存在一个或多个活跃/未确认告警
  - 一个或多个下游依赖是关键资产
  - 表的重要性评分高于 0.8
→ 在运行工作流 4 前询问用户：
  "这是一个高重要性表，有 [N 个活跃告警 / 关键资产依赖 / 重要性评分 0.989]。是否在继续前运行完整的变更影响评估？（是/否）"
→ 等待确认。若是 → 运行工作流 4。
  若否 → 继续但注明："按您的要求跳过影响评估。"

若存在风险信号但未表达变更意图：
→ 仅呈现健康度摘要并标注风险信号：
  "这是一个具有关键资产依赖的高重要性表。准备好修改时，说'运行影响评估'或直接描述变更，我会自动运行。"
→ 不运行工作流 4。不询问是否运行工作流 4。

### 新建模型变体

用户创建新的 .sql dbt 模型文件时（非编辑现有文件）：

1. 从 SQL 中解析所有 `{{ ref('...') }}` 和 `{{ source('...', '...') }}` 调用
2. 对每个引用的表，运行标准工作流 1 健康度检查：
   search() → getTable() → getAlerts()
3. 呈现汇总的上游健康度摘要：
   "您的新模型引用了 N 个上游表。当前健康度如下："
   - 列出每个表：最后更新时间、活跃告警（如有）、关键资产标记
4. 将任何有活跃告警的上游表标记为风险：
   "⚠️ <table_name> 有 <N> 个活跃告警 — 新模型将继承此数据质量问题"

新建模型跳过 getAssetLineage — 它们尚无下游依赖。
新建模型跳过工作流 4 — 没有现有爆炸半径需要评估。

## 工作流 2：添加监控器 — 添加新转换逻辑时

> **详细的监控器创建指导** — 包括参数验证、字段类型兼容性检查和常见错误预防 — 参见 `monitor-creation` 技能（`skills/monitor-creation/SKILL.md`）。以下工作流是 prevent 会话中常见"刚加了一列，建议一个监控器"场景的快速路径。

用户添加新列、过滤条件或业务规则时，建议添加监控器。首先根据新逻辑的作用选择监控器类型：

```
- 新列包含行级条件（空值检查、范围、正则）
  → createValidationMonitorMac

- 新的聚合指标（行数、求和、平均值、百分位数随时间变化）
  → createMetricMonitorMac

- 逻辑应匹配另一个表或先前时间段
  → createComparisonMonitorMac

- 不符合上述的复杂业务规则
  → createCustomSqlMonitorMac
```

然后运行对应序列：

```
1. 读取正在编辑的 SQL 文件以提取具体的转换逻辑：
   - 从对话上下文确认文件路径（不要猜测或假设）
   - 若文件路径不明确，询问工程师："新逻辑在哪个文件中？"
   - 提取具体的新列定义、过滤条件或业务规则
   - 在步骤 3 构建监控器条件时直接使用此逻辑

2. 对于验证监控器：getValidationPredicates() → 显示可用的验证类型
   对于所有类型：从上面的选择指南确定正确的工具
3. 调用选定的 create*MonitorMac 工具：
   - createValidationMonitorMac(mcon, description, condition_sql) → 返回 YAML
   - createMetricMonitorMac(mcon, description, metric, operator) → 返回 YAML
   - createComparisonMonitorMac(source_table, target_table, metric) → 返回 YAML
   - createCustomSqlMonitorMac(mcon, description, sql) → 返回 YAML
   ⚠ 若 createValidationMonitorMac 失败（如列在正式表中尚不存在），
     改用 createCustomSqlMonitorMac 并提供显式 SQL 查询。
3. 将 YAML 保存到 <project>/monitors/<table_name>.yml
4. 运行：montecarlo monitors apply --dry-run（预览）
5. 运行：montecarlo monitors apply --auto-yes（应用）
```

**重要 — `monitors apply` 的 YAML 格式：**
所有 `create*MonitorMac` 工具返回的 YAML 不能直接用于 `montecarlo monitors apply`。需将输出重新格式化为以 `montecarlo:` 为根键的独立监控器文件。第二级键对应监控器类型：`custom_sql:`、`validation:`、`metric:` 或 `comparison:`。以下示例使用 `custom_sql:` — 其他监控器类型替换为对应的键。

```yaml
# monitors/<table_name>.yml  ← 仅监控器定义，不是 montecarlo.yml
montecarlo:
  custom_sql:
    - warehouse: <warehouse_name>
      name: <monitor_name>
      description: <description>
      schedule:
        interval_minutes: 720
        start_time: '<ISO timestamp>'
      sql: <your validation SQL>
      alert_conditions:
        - operator: GT
          threshold_value: 0.0
```

`montecarlo.yml` 项目配置是项目根目录中的**独立文件**，仅包含：
```yaml
# montecarlo.yml  ← 仅项目配置，不是监控器定义
version: 1
namespace: <your-namespace>
default_resource: <warehouse_name>
```

不要将 `version:`、`namespace:` 或 `default_resource:` 放入监控器定义文件。

## 工作流 3：告警分诊 — 调查活跃事件时

```
1. getAlerts(
     created_after="<开始时间>",
     created_before="<结束时间>",
     order_by="-createdTime",
     statuses=["NOT_ACKNOWLEDGED"]
   ) → 列出未确认告警
2. getTable(mcon="<受影响表的mcon>") → 检查当前表状态
3. getAssetLineage(mcon="<mcon>") → 识别上游原因或下游爆炸半径
4. getQueriesForTable(mcon="<mcon>") → 可能解释异常的近期查询
```

响应告警：
- `updateAlert(alert_id="<id>", status="ACKNOWLEDGED")` — 确认告警
- `setAlertOwner(alert_id="<id>", owner="<email>")` — 分配负责人
- `createOrUpdateAlertComment(alert_id="<id>", comment="<text>")` — 添加上下文

## 工作流 4：变更影响评估 — 修改模型前必需

**触发条件：** 任何添加、重命名、删除或修改列、join、过滤条件或模型逻辑的意图。立即运行 — 在编写任何代码之前 — 即使用户未要求。

### Bug 修复和回退同样需要影响评估

用户说"修复"、"回退"、"恢复"或"撤销"时，在编写任何代码前运行此工作流 — 即使变更看起来很小或安全。

回退操作撤销列添加或更改 join 逻辑，其爆炸半径与原始变更相同。下游模型可能已适应了"不正确"的行为，意味着修复本身可能破坏它们。

特别注意：
- 回退是否会移除其他模型现在依赖的列
- 下游模型是否引用了正在回退的具体逻辑
- 活跃告警是否可能与正在回退的变更相关

当用户即将重命名或删除列、更改 join 条件、修改过滤条件或重构模型逻辑时，运行此序列在提交任何变更前呈现爆炸半径：

```
1. search(query="<table_name>") + getTable(mcon="<mcon>")
   → 重要性评分、查询量（每日读/写）、关键资产标记

2. getAssetLineage(mcon="<mcon>")
   → 下游依赖完整列表；标注每个是否为关键资产

3. 对每个关键下游资产执行 getTable(mcon="<downstream_mcon>")
   → 重要性评分、最后更新时间、监控状态

4. getAlerts(
     created_after="<7天前>",
     created_before="<现在>",
     table_mcons=["<mcon>", "<downstream_mcon_1>", ...],
     statuses=["NOT_ACKNOWLEDGED"]
   )
   → 任何已影响此表或其依赖的活跃事件

5. getQueriesForTable(mcon="<mcon>")
   → 近期查询；扫描对正在变更的具体列的引用
   → 对模糊情况使用 getQueryData(query_id="<id>") 获取完整 SQL

5b. 补充本地搜索下游 dbt 引用：
   - 在本地 models/ 目录中搜索 ref('<table_name>')（仅单跳）
   - 将结果与步骤 2 的 getAssetLineage 输出对比
   - 若本地模型引用了此表但不在 MC 血缘结果中：
     "⚠️ 发现 N 个本地模型引用此表但尚未在 MC 血缘中：[列表]"
   - 若当前项目不存在 models/ 目录，静默跳过
   - MC 血缘仍为权威来源 — 本地 grep 仅作补充

6. getMonitors(mcon="<mcon>")
   → 哪些监控器正在监视受变更影响的列或指标
```

### 风险等级评估

| 等级 | 条件 |
|---|---|
| 🔴 高 | 关键资产在下游，或活跃告警已触发，或每日读取 >50 |
| 🟡 中 | 非关键资产在下游，或受影响列上有监控器，或查询量中等 |
| 🟢 低 | 无下游依赖，无活跃告警，查询量低 |

### 多模型变更

用户在同一会话或同一领域内变更多个模型时（如 3 个时序模型、4 个 criticality_score 模型）：

- 跨所有变更表运行单一合并影响评估
- 去重下游依赖 — 若两个变更表共享同一下游依赖，仅计一次并注明受多个上游变更影响
- 呈现统一的爆炸半径报告而非 N 个独立报告
- 若合并爆炸半径大于任何单个表，提升风险等级

合并报告标题示例：
"## 变更影响：timeseries 领域的 3 个模型
合并下游爆炸半径：28 个表（已去重）
最高风险表：timeseries_detector_routing（22 个下游引用）"

### 报告格式

```
## 变更影响：<table_name>

风险：🔴 高 / 🟡 中 / 🟢 低

下游爆炸半径：
  - <N> 个表依赖此模型
  - 受影响的关键资产：<列表或"无">

活跃事件：
  - <告警标题、状态> 或"无"

列暴露（正在变更的列）：
  - 在 <N> 条近期查询中出现（如 <查询片段>）

监控器覆盖：
  - <监控器名称> 监视 <指标> — 将受此变更影响
  - 若零个自定义监控器 → 追加：
    "⚠️ 此表无自定义监控器。完成变更后，
    我会为新逻辑建议一个监控器 — 或说'添加监控器'
    立即执行。"

建议：
  - <具体建议，如"部署前通知 downstream_table 的负责人"，
     "与新鲜度告警负责人协调"，"为新列添加监控器">
```

若风险为 🔴 高：
1. 调用 `getAudiences()` 获取已配置的通知受众
2. 在建议中包含："通知：<受众名称/渠道>"
3. 主动建议：
   - 通知下游关键资产负责人（对活跃告警使用 `setAlertOwner` / `createOrUpdateAlertComment`）
   - 部署前为新逻辑添加监控器（工作流 2）
   - 变更后运行 `montecarlo monitors apply --dry-run` 验证无破坏

### 综合分析：将发现转化为代码建议

呈现影响报告后，利用发现来指导代码建议。
不要呈现 MC 数据后又忽略数据编写代码。
明确将每个关键发现与具体建议关联：

- 表上活跃告警已触发：
  → 建议推迟或最小化变更范围，直到告警解决
  → 解释："此表有 N 个活跃告警 — 现在变更可能加剧现有数据质量问题"

- 关键资产在下游：
  → 建议防御性编码模式：空值保护、向后兼容变更、尽可能仅做增量 schema 变更
  → 解释："X 个下游关键资产依赖此表 — 建议编写为[具体模式]以避免破坏[具体依赖]"

- 受影响列上有监控器：
  → 指出变更将影响监控器覆盖
  → 建议在代码变更同时更新监控器（提供工作流 2）
  → 解释："[列]上的现有监控器需要更新以适应此变更"

- 新增输出列或逻辑：
  → 影响评估后始终提供工作流 2，无论现有监控器覆盖情况
  → 即使风险等级为 🟢 低也不跳过此步骤
  → 明确说："这添加了新的输出逻辑 — 是否需要我生成一个监控器？
    我可以添加空值检查、范围验证或自定义 SQL 规则。"
  → 等待用户回复后再继续编辑

- 高读取量（每日 >50 次读取）：
  → 对列重命名或删除建议格外谨慎
  → 建议向后兼容过渡（添加新列，弃用旧列）
  → 解释："此表每日有 [N] 次读取 — 没有过渡期的列重命名会立即破坏下游消费者"

- 列重命名，即使位于 CTE 内部：
  → 不要假设 CTE 内部的重命名是安全的。始终检查：
    1. 此列是否出现在最终 SELECT 中，直接或通过馈入最终 SELECT 的 CTE？
    2. 若是 — 视为破坏性变更。建议向后兼容过渡：添加正确命名的列，
       暂时保留旧列，在后续 PR 中移除。
    3. 若确实为内部使用且从未出现在输出中 — 在继续前明确确认这一点。
  → 解释："即使此列定义在 CTE 中，若它出现在最终 SELECT 中则是公共输出列 —
    重命名会破坏任何按名称选择它的下游模型。"

## 工作流 5：变更验证查询 — 代码变更完成后

**触发条件：** 仅工程师明确意图时。当工程师说以下内容时激活：
- "生成验证查询"、"验证这个变更"、"我完成了这个变更"
- "让我测试一下"、"写查询检查这个"、"准备提交"

**必需的会话上下文 — 缺少任一则不激活：**
1. 本会话中已为此表运行工作流 4（变更影响评估）
2. 已对该表的 `.sql` 或 dbt 模型文件进行了编辑

**不要在文件编辑后自动激活。不要在工作流 4 或文件编辑后主动提供。工程师准备好时会主动请求。**

---

### 本工作流的功能

利用会话中已有的上下文 — 工作流 4 发现、文件 diff 和 `getTable` 结果 — 生成 3-5 条针对性 SQL 验证查询，直接测试此具体变更是否按预期执行。

这些不是通用模板。使用工作流 4 上下文中变更的语义含义：哪些列变了及原因、影响了什么业务逻辑、什么下游模型依赖此表、存在哪些监控器。对新的 `days_since_contract_start` 列的空值检查应验证它永不为负且对有 `contract_start_date` 的行永不为空 — 而非仅泛泛检查空值。

---

### 步骤 1 — 从会话上下文识别变更类型

从工作流 4 发现和文件 diff 中，对主要变更进行分类。一个变更可能跨越多个类型 — 分类主导类型并标注次要类型：

- **新列** — SELECT 中添加了新的输出列
- **过滤条件变更** — WHERE 子句、IN 列表或 CASE 条件被修改
- **Join 变更** — JOIN 条件或 join 目标被修改
- **列重命名或删除** — 现有输出列被重命名或移除
- **参数变更** — 硬编码阈值、常量或数值被更改
- **新建模型** — 文件为新创建，无生产基线

---

### 步骤 2 — 从工作流 4 确定数仓上下文

从会话上下文中已有的 `getTable` 结果提取：
- **完全限定表名** — 如 `analytics.prod_internal_bi.client_hub_master`
- **数仓类型** — Snowflake、BigQuery、Redshift、Databricks
- **Schema** — 已解析，无需重新推导

使用对应数仓类型的正确 SQL 方言。关键差异：

| 数仓 | 日期差 | 当前时间戳 | 备注 |
|---|---|---|---|
| Snowflake | `DATEDIFF('day', a, b)` | `CURRENT_TIMESTAMP()` | 支持 `QUALIFY` |
| BigQuery | `DATE_DIFF(a, b, DAY)` | `CURRENT_TIMESTAMP()` | 使用子查询代替 `QUALIFY` |
| Redshift | `DATEDIFF('day', a, b)` | `GETDATE()` | |
| Databricks | `DATEDIFF(a, b)` | `CURRENT_TIMESTAMP()` | |

对于开发数据库，使用占位符 `<YOUR_DEV_DATABASE>` 并附加注释指示工程师替换。不要猜测开发数据库名称。

---

### 步骤 3 — 应用数据库定位规则（强制）

这些规则不可协商 — 违反会导致查询在运行时失败：

- **仅变更后存在的列或逻辑** → 仅开发数据库。永远不要查询生产环境中尚不存在的列。
- **比较查询（变更前后对比）** → 生产库和开发库均查询
- **新建模型（无生产基线）** → 所有查询仅开发库
- **行数比较** → 始终包含，始终查询两个数据库

---

### 步骤 4 — 生成针对性验证查询

无论变更类型如何，始终包含行数比较 — 它是意外情况的基础信号。

然后根据此变更类型需要验证的内容生成变更特定查询。使用 diff 和工作流 4 发现中的确切条件、列名和业务逻辑 — 而非通用占位符。每种变更类型的目标：

**新列：** 验证列在应为非空的地方非空（基于其业务含义）、值范围合理、分布符合底层数据预期。仅查询开发库。

**过滤条件变更：** 验证仅预期行被重新分类 — 使用 diff 中的确切过滤逻辑生成变更前后计数，显示新条件添加或移除了多少行，以及变更分类的行样本。样本帮助工程师确认正确的记录被移动。

**Join 变更：** 验证 join 未引入重复 — join 键的唯一性检查必不可少。同时验证行数未意外变化。开发库查询唯一性，两个数据库查询行数。

**列重命名或删除：** 验证旧列名在开发 schema 中不存在，新列（若为重命名）存在。同时验证引用旧列名的下游模型已被识别 — 使用工作流 4 中的本地 ref() grep 结果（如有）。

**参数或阈值变更：** 验证受变更影响的值分布 — 多少行移至新阈值之上或之下，计数是否符合工程师预期。查询两个数据库进行前后对比。

**新建模型：** 无法与生产环境对比。验证行数非零且合理、样本行看起来正确、关键列非空。仅查询开发库。

---

### 步骤 5 — 为每条查询添加变更特定上下文

每条查询包含一个 SQL 注释块，说明：
- 查询在检查什么
- **对此具体变更**健康的结果是什么样的
- 什么表明存在问题

从工作流 4 发现中推导此上下文。使用变更的业务含义，而非通用描述。例如，添加 `days_since_contract_start` 时：

```sql
/*
空值率检查：days_since_contract_start（新列，仅开发库）
关注点：
  - 空值计数应等于没有 contract_start_date 的工作区数
  - 所有有 contract_start_date 的行应有非空、非负值
  - 值超过 3650（约10年）可疑，可能表示数据问题
*/
```

这正是这些查询区别于通用验证的地方 — 注释告诉工程师对他们具体变更而言通过和失败分别长什么样。
