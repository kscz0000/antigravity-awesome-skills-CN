# 推送数据的异常检测

推送的数据量和新鲜度数据与 pull 模型使用相同的异常检测器。检测器不会立即激活 — 它们需要足够的历史数据来学习预期行为，然后才能对偏差发出警报。

## 推荐的推送频率：每小时

- 最多**每小时推送一次** — 更频繁的推送会产生不可预测的检测器行为，因为训练管道将数据聚合到小时桶中
- **持续推送** — 超过几天的间隔会延迟激活或停用先前活跃的检测器

## 新鲜度检测器

新鲜度检测器学习表的更新频率，并在表未按预期更新时触发。

**训练内容**：跨推送的 `last_update_time` 值之间的连续差异（`delta_sec`）。仅当 `last_update_time` 实际发生变化时，推送才算数。

**激活要求：**
| 要求 | 值 |
|---|---|
| 最小样本数 | 7 次 `last_update_time` 发生变化的推送（或慢表覆盖率 ≥ 0.8） |
| 最小覆盖率 | 0.15（= `median_update_secs × n_samples / 22 天`） |
| 训练窗口 | 35 天 |
| 支持的更新周期 | 5 分钟 – 7.7 天 |
| 最小表龄 | 在较旧的仓库上约 14 天 |

**停用触发条件：**
- **14 天**无推送 → `"no recent data"`
- 快速表（中位更新 ≤ 26.4 小时）在过去 14 天内间隔 > 7 天 → `"gap of over a week in last 2 weeks"`

## 数据量检测器（Volume Change + Unchanged Size）

检测行数或字节数的意外峰值/下降。

**激活要求：**
| 要求 | 值 |
|---|---|
| 最小样本数（每日） | 10 |
| 最小样本数（次每日，约 12 次/天） | 48 |
| 最小样本数（每周） | 5 |
| 最小覆盖率 | 0.30（= `N × median_update_secs / 42 天`） |
| 训练窗口 | 42 天 |
| 最小表龄 | 5 天 |
| 规律性检查 | 更新间隔的第 75/25 百分位 ≥ 0.2 |

**停用**：没有硬性间隔限制，但随着 42 天窗口在没有新数据的情况下推进，覆盖率会下降。最终降至 0.3 以下并停用。

## 总结表

| | 新鲜度 | Volume Change / Unchanged Size |
|---|---|---|
| 推荐频率 | 每小时 | 每小时 |
| 最大频率 | 每小时一次 | 每小时一次 |
| 训练窗口 | 35 天 | 42 天 |
| 最小样本数 | 7 | 10（每日）/ 48（次每日）/ 5（每周） |
| 最小覆盖率 | 0.15 | 0.30 |
| 硬性停用间隔 | 14 天 | 无（覆盖率下降） |
| 快速表间隔警告 | 过去 14 天内 7 天 | 不适用 |

## 如何回答客户

当客户问"为什么我的异常检测不起作用？"时：

1. **检查检测器状态** — 在 MC UI 或通过 GraphQL（`getTable.thresholds.freshness.status`）。`"training"` 状态表示数据不足。`"inactive"` 表示触发了停用条件 — 检查原因代码。

2. **验证推送频率** — 他们是否恰好每小时推送一次？过快和过慢的速率都会导致问题。

3. **验证 `last_update_time` 是否变化** — 要让新鲜度累积训练样本，每次推送必须携带与前一次*不同*的 `last_update_time`。如果表实际上没有更新，推送仍然会到达但不会推进样本计数。

4. **设定合理预期** — 新鲜度检测器需要约 1-2 周的每小时推送。每日表的数据量检测器需要 10+ 天，次每日表需要最多 42 天。异常检测不是即时的。

5. **不要中断推送后恢复** — 如果客户暂停推送一周后恢复，新鲜度检测器可能会停用。即使表没有变化，他们也应该继续推送（只需重复相同的 `last_update_time`）以保持覆盖率，尽管该特定推送不会算作新的新鲜度样本。
