# 信息论在代码和系统中的应用

## 1. Shannon 熵在软件中的应用

### 应用定义
```
H(X) = -Σ p(xᵢ) · log₂ p(xᵢ)

在代码中的解读：
- X = 随机变量"系统处于哪个状态"
- p(xᵢ) = 处于状态 i 的概率
- H(X) = 关于状态的不确定性 → 测试复杂度

VoicePipeline 示例：
状态：IDLE(70%), RECORDING(15%), TRANSCRIBING(5%),
      QUERYING_LLM(5%), SPEAKING(4%), ERROR(1%)

H = -(0.70·log₂0.70 + 0.15·log₂0.15 + 0.05·log₂0.05 +
      0.05·log₂0.05 + 0.04·log₂0.04 + 0.01·log₂0.01)
H ≈ 1.45 bits

理论最大值（均匀分布）：log₂(6) ≈ 2.58 bits
熵效率：1.45/2.58 ≈ 56% — 低熵 = 系统结构良好
```

### 接口熵
```
对函数 f: I → O
H(O) = 可能输出的熵
H(O|I) = 给定 I 时 O 的熵（残余不确定性）

互信息 I(I;O) = H(O) - H(O|I)
  = 输入减少了多少关于输出的不确定性

理想目标：I(I;O) = H(O) → 输入完全确定输出
  等价于函数是确定性的（无非确定性）

Auri 中的问题案例：
BluetoothController.connect() — 输出取决于蓝牙设备状态
H(success|deviceAddress) > 0（连接可能因外部原因失败）
→ 此点系统本质上是非确定性的
→ 错误处理是必须的，而非可选的
```

## 2. Kolmogorov 复杂度

### 定义
```
K(x) = 产生 x 的最短程序长度

实际解读：
- K(代码) = 代码的内在算法复杂度
- 代码不能被压缩到 K(代码) 以下
- 若代码有重复模式 → K(代码) << |代码|
  → 抽象/重构的机会
```

### 应用：检测重复代码
```
原理：若 compress(file1 + file2) << |file1| + |file2|
则 file1 和 file2 有共享结构 → 提取抽象

实用工具：测量压缩比
ratio = compress(代码) / |代码|

启发式阈值：
ratio < 0.3：大量重复代码 → 紧急重构
ratio 0.3-0.5：部分重复 → 重构机会
ratio > 0.5：代码多样/富有表达力 → 可接受
```

## 3. 信道容量与吞吐量

### 蓝牙系统的噪声模型
```
含噪声的 Shannon 信道：
- 容量 C = B · log₂(1 + S/N) bits/s
  B = 带宽（Hz），S/N = 信噪比

对 SCO（蓝牙耳机麦克风）：
- B = 8kHz（窄带）或 16kHz（宽带）
- 语音质量：安静环境下典型 S/N 20-30dB
- C ≈ 8000 · log₂(101) ≈ 53,000 bits/s ≈ 53 kbps

App 中的 PCM 录音：
- 16kHz, 16-bit → 256 kbps 原始数据
- BT SCO 通道有效压缩：64 kbps（CVSD 编码）
- 质量损失：约 75% 的 PCM 分辨率
→ 解释了为何有 BLE Audio 时应优先于 SCO
```

### 吞吐量流水线
```
瓶颈模型（串联排队论）：
流水线：音频 → STT → LLM → TTS

各阶段容量：
阶段         | 速率       | 缓冲    | 备注
音频采集     | 256 kbps   | 环形缓冲 | 连续
STT 在线     | ~500ms/请求| 1 请求   | 延迟主导
LLM (API)    | ~500 tok/s | 1 请求   | 高吞吐
LLM (Ollama) | ~3 tok/s   | 1 请求   | 瓶颈 A04
TTS          | ~200 ms    | 1 请求   | 快速

流水线吞吐量 = min(各阶段速率) = 3 tok/s（Ollama A04）
约等于 ~15 词/分钟（对流畅对话来说太慢）
→ 解释了实时对话应使用 OpenAI API 的建议
```

## 4. 编码论在 API 中的应用

### 冗余与鲁棒性
```
有冗余的协议：
- 码率：r = k/n（k = 有效比特，n = 传输比特）
- r = 1：无冗余（脆弱）
- r < 1：有冗余（容错）

在 REST/LLM API 中：
- 重试逻辑 = 时间冗余：r = 1/(尝试次数)
- 幂等键 = 去重：保证恰好 1 次处理
- 校验和/哈希 = 验证冗余

对 Auri AuriToolExecutor：
- 无幂等键：r = 1（重试时脆弱）
- 有幂等键：r = 1/max_retries（鲁棒）
```

### LLM 上下文的压缩与效率
```
LLM 上下文有限（如 Claude 的 128k token）
每次对话消耗：Σ len(消息_i) token

上下文优化即压缩问题：
- 仅保留下次回复所需的关键信息
- 摘要历史以节省 token
- 策略性"遗忘"无关上下文

Auri 的最优上下文管理策略：
1. 保留最近 N 轮完整对话（N=5-10）
2. 将更早的对话摘要为 1-2 句话
3. 始终保留：系统上下文 + 用户最新消息
4. 此策略下每次请求的估计成本：~200-500 token
```

## 5. 贝叶斯决策论

### Bug 诊断即贝叶斯推断
```
P(bug=B | 观测=O) = P(O|B) · P(B) / P(O)

其中：
- P(B) = 先验：此类 bug 的历史频率
- P(O|B) = 似然：给定此 bug 出现此日志/行为的概率
- P(O) = 证据：归一化

Auri 示例：
Bug："流水线卡在 TRANSCRIBING"
观测："STT 返回空结果，UI 冻结"

假设及估计先验：
H1：STT 结果处理器未更新状态（P = 0.3）
H2：协程在处理结果前被取消（P = 0.25）
H3：emit() 中有静默异常（P = 0.2）
H4：MainActivity 生命周期问题（P = 0.15）
H5：其他原因（P = 0.1）

代码分析后：
P(空结果|H1) = 0.9 → P(H1|空结果) ≈ 0.52 — 最可能
P(空结果|H2) = 0.7 → P(H2|空结果) ≈ 0.34
P(空结果|H3) = 0.3 → P(H3|空结果) ≈ 0.11

贝叶斯诊断：优先调查 H1（52%），其次 H2（34%）
```

### 分析的置信度估计
```
当 Prof. Euler 做出断言时，应包含校准：

"[断言] — 置信度：X%"

典型校准：
90%+：基于已读代码 + 充分验证的模式
70-89%：合理推断 + 类似模式经验
50-69%：合理假设，需验证
<50%：推测，明确标注
```

## 6. 最小描述长度（MDL）

### 架构选择的 MDL 原则
```
原则：选择使下式最小的架构：
MDL = 长度(模型) + 长度(数据|模型)

应用于设计模式：
- MVVM：紧凑模型（3 层），数据组织良好 → MDL 低
- God Class：紧凑模型（1 个类），数据混乱 → MDL 高
- 微服务：复杂模型，数据分布良好 → MDL 中等

对 Auri MainViewModel：
若 MainViewModel CC > 50 且 300+ 行：
- MDL(当前) = 1 个大文件 = 模型开销低，理解成本高
- MDL(重构后) = 5 个专用 ViewModel = 模型开销，每个成本低
- 当总复杂度 > 阈值 ≈ CC 30 时，MDL(重构后) < MDL(当前)

建议：每文件 CC 总量 > 30 时分解
```

## 7. 信息论在日志中的应用

### 日志熵（异常检测）
```
正常日志：消息遵循平稳分布
- 计算基线：H_baseline = 前 N 条日志的熵
- 监控：H_current = 滑动窗口熵

异常条件：|H_current - H_baseline| > 2σ

可检测的异常类型：
1. 大量相同消息（刷屏）→ 熵骤降
2. 意料之外的消息（新类型错误）→ 熵上升
3. 事件序列异常 → 日志间互信息变化

对 Auri FileLogWriter：
- 记录时间戳 + 事件类型 + 模块
- 后处理：按模块按分钟计算熵
- 阈值：H(模块, 分钟) < 0.5 或 > 3.5 bits 时告警
```

### 日志压缩即模式检测
```
compress(日志) / |日志| = 压缩比

低压缩比（< 0.2）：日志高度重复 → 可能有循环或刷屏
高压缩比（> 0.7）：日志变化多端 → 可能有不稳定状态

健康系统：压缩比 ≈ 0.3-0.5
```
