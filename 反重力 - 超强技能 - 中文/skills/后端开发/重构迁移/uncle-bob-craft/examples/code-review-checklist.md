# Uncle Bob 工程实践 — 代码审查清单

使用 uncle-bob-craft 技能进行基于原则的代码审查时，直接复制粘贴此清单。单独运行项目的 linter/formatter；本清单聚焦于结构和设计。

---

## 1. 依赖规则与边界

- [ ] 依赖**指向内层**（用例/领域不依赖 UI、DB 或框架细节）。
- [ ] 外层依赖内层定义的接口（如 repository、gateway）。
- [ ] 审查中的核心/用例代码没有直接导入框架或驱动。

## 2. SOLID 在上下文中

- [ ] **SRP** — 每个类/模块只有一个变更理由（一个参与者）。
- [ ] **OCP** — 通过新实现扩展，而非编辑现有核心逻辑。
- [ ] **LSP** — 子类型可替换；对具体类型无隐藏假设。
- [ ] **ISP** — 接口聚焦；调用方不依赖未使用的方法。
- [ ] **DIP** — 高层代码依赖抽象；具体实现在边缘注入。

## 3. 异味

- [ ] **僵化** — 小改动不会迫使多处编辑。
- [ ] **脆弱** — 修改不会破坏不相关的区域。
- [ ] **不可移植** — 在合理的情况下可以复用。
- [ ] **粘滞** — 做正确的事不比走捷径更难。
- [ ] **不必要复杂** — 无投机性或未使用的抽象。
- [ ] **不必要重复** — DRY；无明显重复。
- [ ] **不透明** — 命名和结构让意图清晰。

## 4. 设计模式

- [ ] 使用的任何模式都有明确的**理由**（重复、变化或边界）。
- [ ] 无盲目套用的模式（如只有一个实现且无变化计划的 Factory）。
- [ ] 模式提高了可读性或可测试性，而非相反。

## 5. 测试与职业素养

- [ ] 被修改的代码有或被测试覆盖（在适用的情况下）。
- [ ] 没有明显违反可持续节奏或质量的"以后再修"或"TODO: refactor"。
- [ ] 提交/PR 是连贯的，不会让代码库变得更差。

---

## 建议的审查输出格式

1. **边界**：关于依赖方向和任何违规的一两句话。
2. **SOLID**：列出违规项，附文件/函数和原则（如"SRP: `OrderService` 既解析又持久化——应拆分"）。
3. **异味**：列出发现的异味及位置（如"僵化：修改折扣规则涉及 4 个文件"）。
4. **具体重构**：一两个具体建议（如"从 `process` 中提取 `applyDiscount`"；"引入 `OrderRepository` 接口并在用例中注入"）。
5. **测试/职业素养**：关于测试覆盖率和任何顾虑的简要说明。
