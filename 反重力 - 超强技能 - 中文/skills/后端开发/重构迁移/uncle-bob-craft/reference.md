## Clean Architecture

### 依赖规则

源码依赖只能**指向内层**（指向更高层的策略）。内层不知道外层的存在（如用例不从 web 框架或数据库驱动导入）。

- **实体**（企业业务规则）— 中心，无向外依赖。
- **用例**（应用业务规则）— 仅依赖实体。
- **接口适配器**（presenter、gateway）— 依赖用例/实体。
- **框架和驱动**（UI、DB、HTTP）— 最外层；向内依赖。

### 分层（六边形 / 端口与适配器）

| 层 | 职责 | 依赖于 |
|-------|----------------|------------|
| 实体 | 核心业务实体和规则 | 无（最内层） |
| 用例 | 特定于应用的业务规则、编排 | 实体 |
| 接口适配器 | 在用例与外部世界之间转换数据 | 用例、实体 |
| 框架和驱动 | DB、UI、HTTP、文件 I/O | 接口适配器 |

### 架构上下文中的 SOLID

- **SRP** — 一个模块应只有一个变更理由（一个参与者）。
- **OCP** — 对扩展开放，对修改封闭（使用抽象和边界）。
- **LSP** — 子类型必须可替换其基类型（接口和实现）。
- **ISP** — 多个客户端专用接口优于一个通用接口（在边界处拆分接口）。
- **DIP** — 依赖抽象，而非具体实现（在边界处反转依赖）。

### 组件设计（摘要）

将类分组为组件时：**REP**（复用/发布等价）、**CCP**（共同封闭）、**CRP**（共同复用）；**ADP**（无环依赖）、**SDP**（稳定依赖）、**SAP**（稳定抽象）。简要定义见 [references/clean-architecture.md](./references/clean-architecture.md)。

## The Clean Coder

### 职业素养

- **不造成伤害** — 不要让代码比你接手时更差；能改善时就改善。
- **精通技艺** — 了解你的手艺、练习、保持更新。
- **敢于说不** — 当请求不合理或会损害质量时说不；提供替代方案。
- **说到做到** — 当你说"是"时，要当真；承诺你能完成的截止日期。

### 估算

- **三点估算** — 最佳情况、正常情况、最差情况；用于规划，而非承诺。
- **速率** — 使用历史速率进行迭代规划；不要膨胀承诺。
- **不确定性** — 传达不确定性；避免虚假精确。

### 可持续节奏

- 避免持续加班；它会降低质量和长期产出。
- 测试和重构是工作的一部分，不是"额外的"。

### 测试即要求

- 没有测试的代码在定义上就是遗留代码。编写测试是职业要求。

### 指导与协作

- 帮助他人（指导、结对）以及让团队和代码库变得更好，是职业素养的一部分。

## Clean Agile

### 价值观（来自敏捷宣言，由 Uncle Bob 强调）

- 个体和互动 高于 流程和工具。
- 可工作的软件 高于 详尽的文档。
- 客户协作 高于 合同谈判。
- 响应变化 高于 遵循计划。

### 铁十字（支持上述价值观的四个支柱）

| 价值观 | 含义 |
|-------|--------|
| **沟通** | 偏好面对面和协作；减少信息丢失。 |
| **勇气** | 敢于重构、敢于说不、在需要时改变设计。 |
| **反馈** | 短反馈循环（测试、演示、迭代）。 |
| **简单** | 做可能有效的最简单的事；避免投机性设计。 |

### 实践

- **TDD** — 红、绿、重构；测试先行作为规范和安全网。
- **重构** — 持续小改进；保持测试通过。
- **结对编程** — 两人一机；设计和质量提升。
- **简单设计** — 无重复、表达意图、最少元素、小抽象。

## 代码异味与启发式规则（扩展）

### 设计异味（来自 Clean Code / Clean Architecture）

| 代码 | 名称 | 简述 |
|------|------|--------|
| **僵化** | 难以变更；一次变更触发多处修改。 | 降低耦合；引入边界。 |
| **脆弱** | 在意想不到的地方崩溃。 | 隔离变更；改进测试。 |
| **不可移植** | 难以复用。 | 提取可复用部分；降低耦合。 |
| **粘滞** | 走捷径容易，做正确的事很难。 | 让正确的事变容易（抽象、测试）。 |
| **不必要复杂** | 投机性或未使用的设计。 | YAGNI；需要时再加。 |
| **不必要重复** | 违反 DRY。 | 提取公共逻辑；命名抽象。 |
| **不透明** | 难以理解。 | 重命名、提取、澄清意图。 |

### 启发式规则（审查清单风格）

- **C1 / 命名** — 名称揭示意图；无误导（如 `accountList` 实际不是 list）。
- **C2 / 函数** — 小、单一职责、单一抽象层级；少参数。
- **C3 / 注释** — 优先自解释代码；好注释说"为什么"，而非"是什么"。
- **C4 / 格式** — 报纸隐喻；垂直密度；变量靠近使用处。
- **C5 / 错误处理** — 异常优于返回码；无契约时不返回或传递 null。
- **C6 / 测试** — F.I.R.S.T.；适用时用 TDD。
- **C7 / 类** — 小、单一职责；递降规则。
- **T1 / 边界** — 依赖指向内层；内层无外层细节。
- **T2 / SOLID** — 在上下文中检查 SRP、OCP、LSP、ISP、DIP。
- **T3 / 模式** — 模式用于解决真实问题，而非做样子。

## 设计模式（何时用与何时不用）

- **使用**的时机：有重复的变化（Strategy）、生命周期/创建复杂性（Factory）、明确的横切关注点（如日志、边界处的验证）。
- **避免**的时机：代码简单，模式会增加间接层却未减少重复或澄清意图。
- **盲目套用** — 因为"名字叫这个"到处套用模式（如一切都是 Factory）而无设计理由。偏好"可能有效的最简单做法"，直到出现重复或变更轴。
