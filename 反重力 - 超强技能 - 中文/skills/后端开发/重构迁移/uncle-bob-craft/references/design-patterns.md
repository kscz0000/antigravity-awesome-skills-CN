# 设计模式 — 正确使用 vs 误用

在评估设计模式是否合理，还是盲目套用/过度使用时使用。

## 何时使用模式

- **重复的变化** — 相同算法或结构但行为不同（如不同的折扣规则）→ Strategy 或类似模式。
- **生命周期或创建复杂性** — 对象创建有多种变体或步骤 → 当重复或复杂性确实存在时使用 Factory、Builder。
- **横切关注点** — 边界处的日志、验证、认证 → 边缘处的 Decorator、中间件或适配器。
- **稳定抽象，变化的实现** — 你预期需要替换实现（如 repository、gateway）→ 接口 + 实现；依赖注入。

经验法则：当你感受到**第三次重复**或**第二个变更轴**（第二次打开同一模块的理由）时引入模式。在代码或文档中命名模式以明确意图。

## 何时不使用模式

- **简单、线性代码** — 无重复、只有一个变更理由。在此添加 Factory 或 Strategy 只会增加间接层而无收益。
- **投机性需求** — "我们以后可能需要替换实现。"偏好 YAGNI；在你确实有第二个实现或第二个变更理由时再添加抽象。
- **每个类都是模式** — 不是每个类都需要 Factory 或 Interface 包装。在模式解决真正设计问题的地方使用。

## 盲目套用与误用

- **盲目套用** — 因为"我们就是这么做的"或"企业代码有 Factory"而使用模式，没有明确的设计理由。症状：只调用 `new` 的 Factory、只有一个实现且无第二个计划的 Strategy。
- **过度使用** — 每个类名都包含模式名称；层只做转发不加逻辑；代码比直接版本更难理解。
- **误用** — 用错了模式（如对应该可测试和可替换的东西用 Singleton）；模式隐藏了真正设计（如 Facade 背后的 God Object）。

## 好的信号

- 模式名称出现在设计文档或注释中且确实有帮助（如"折扣计算的 Strategy"）。
- 至少有两个具体变体，或有明确说明的未来变化理由。
- 抽象让测试和调用更简单（如测试注入假的 repository）。

在审查中，当有人提议或已添加 Factory、Strategy、Repository 或其他模式时使用此参考——问"这解决了什么重复或变化？"以及"是否有第二个实现或第二个变更理由？"
