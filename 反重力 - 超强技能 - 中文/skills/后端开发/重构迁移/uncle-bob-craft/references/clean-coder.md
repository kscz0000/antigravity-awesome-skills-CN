# The Clean Coder — 深度参考

基于 Robert C. Martin, *The Clean Coder* (2011)。在讨论职业素养、估算和可持续节奏时使用。

## 职业素养

- **不造成伤害** — 不要让代码库处于更差的状态。如果不得不留下混乱，留下清晰的 TODO 并跟踪它。偏好让代码比你接手时更好（童子军法则）。
- **精通技艺** — 练习（如 kata）、阅读、保持更新。只估算你理解到足以估算的内容。
- **敢于说不** — 当请求会损害质量或不合理时，说不。提供替代方案（如"如果去掉 Z，我们能在 Y 日期前完成 X"或"我们需要多两周做测试"）。
- **说到做到** — 当你承诺时，要当真。不要为了取悦而说"是"然后食言；如果无法履行承诺，尽早沟通。

## 估算

- **三点估算** — 最佳 / 正常 / 最差情况。用于规划和风险，而非作为单一的"承诺"数字。
- **速率** — 使用历史速率进行迭代规划。不要膨胀；如果团队某次走得快，将其作为下次的数据，而非新的永久承诺。
- **不确定性** — 让不确定性可见。偏好范围和置信度，而非虚假精确。
- **拒绝估算** — 当请求模糊或工作未知时，拒绝给日期是职业的做法；提议先拆分或在发现后给范围。

## 可持续节奏

- 持续加班会降低质量和长期产出。偶尔的冲刺可能发生；让它罕见并在之后恢复。
- 测试、重构和学习是工作的一部分。为了"赶日期"而跳过它们会造成技术债务，长期来看是不职业的。

## 测试即要求

- 没有测试的代码就是遗留代码。编写和维护测试是职业交付的一部分，不是可选的。
- 在压力下，首先要保护的是测试套件和安全重构的能力。

## 指导与协作

- **帮助他人** — 职业素养的一部分是指导、结对，以及让代码库（和团队）比你接手时更好。
- **协作** — 与利益相关者和同事清晰沟通；需要时说不，无法说"是"时提供替代方案。

在讨论承诺、估算、说不、可持续开发实践或团队协作时使用此参考。
