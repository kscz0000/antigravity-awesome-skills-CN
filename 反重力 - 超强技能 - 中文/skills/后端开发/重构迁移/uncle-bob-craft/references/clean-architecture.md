# Clean Architecture — 深度参考

基于 Robert C. Martin, *Clean Architecture* (2017)。当你需要依赖方向、分层和边界的详细准则时使用。

## 依赖规则

**依赖只能指向内层。** 中心代码（实体、用例）不得依赖外圈代码（UI、DB、框架）。外圈依赖内圈；内圈定义接口，外圈实现它们。

- **违反示例**：用例从 Express、Django 或具体的 repository 实现导入。
- **正确做法**：用例依赖接口（如 `OrderRepository`）；外层适配器实现该接口并使用 Express/DB。

## 分层（从内到外）

1. **实体** — 企业业务规则。适用于整个应用的纯结构和规则。不依赖框架或 UI。
2. **用例** — 应用业务规则。编排实体并定义特定于应用的工作流。仅依赖实体和 I/O 接口（repository、presenter）。
3. **接口适配器** — 在用例与外部世界之间转换数据。Presenter、gateway、controller 将外部格式翻译为用例格式或反向翻译。依赖用例（且仅通过用例依赖实体）。
4. **框架和驱动** — Web 框架、DB 驱动、消息队列、文件 I/O。实现用例或接口适配器定义的接口。向内依赖。

## 边界

- **边界** = 内层代码依赖的接口或抽象类型；外层代码实现。
- 好的边界使替换实现变得容易（如测试用内存 repo，生产用 SQL repo）。
- 在有变更或替换理由的地方划边界（不同的持久化、不同的 UI、不同的传输方式）。

## 此上下文中的 SOLID

- **SRP** — 每个模块（类/包）只有一个变更理由（一个参与者）。如"报告格式化"和"报告计算"因不同原因变更，应分离。
- **OCP** — 通过接口的新实现（新适配器）扩展行为，而非编辑现有用例或实体代码。
- **LSP** — repository 或 gateway 接口的任何实现必须可替换而不破坏用例。
- **ISP** — 小而聚焦的接口（如读写演化不同时使用 `ReadOrderRepository` 和 `WriteOrderRepository`），而非一个臃肿的 `OrderRepository`。
- **DIP** — 用例依赖 `OrderRepository`（接口）；组合根注入 `SqlOrderRepository`。高层策略不依赖低层细节。

## 依赖反转

- **未反转**：用例直接导入并调用 `SqlOrderRepository` → 用例依赖 DB。
- **已反转**：用例依赖 `OrderRepository`（接口）；`SqlOrderRepository` 实现它并在边缘注入。用例独立于 SQL。

## 组件内聚与耦合（适用于较大系统）

将类分组为**组件**（模块、包）时，Uncle Bob 定义了：

**内聚性：**
- **REP（复用/发布等价）** — 复用的单元就是发布的单元；将一起复用和发布的类分组。
- **CCP（共同封闭）** — 因相同原因变更的类属于同一组件；减少变更影响。
- **CRP（共同复用）** — 一起复用的类应打包在一起；避免迫使依赖方引入不需要的内容。

**耦合性：**
- **ADP（无环依赖）** — 组件依赖图不得有环。
- **SDP（稳定依赖）** — 向稳定方向依赖；不太稳定的组件依赖更稳定的组件。
- **SAP（稳定抽象）** — 稳定的组件应是抽象的；不稳定的组件可以是具体的。

在讨论单层之外的模块/包边界时使用此参考。完整论述见 *Clean Architecture*（Martin, 2017）。
