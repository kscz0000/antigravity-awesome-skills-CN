# Clean Agile — 深度参考

基于 Robert C. Martin, *Clean Agile* (2019)。在讨论敏捷价值观、实践和"铁十字"时使用。

## 敏捷价值观（宣言）

- 个体和互动 高于 流程和工具。
- 可工作的软件 高于 详尽的文档。
- 客户协作 高于 合同谈判。
- 响应变化 高于 遵循计划。

Uncle Bob 强调右侧仍有价值；在存在权衡时优先选择左侧。

## 铁十字（四个支撑价值观）

| 价值观 | 实践中的含义 |
|-------|---------------------------|
| **沟通** | 偏好面对面（或高带宽）沟通；减少信息丢失；保持团队对齐。 |
| **勇气** | 敢于重构、对不合理请求说不、在代码告诉你时改变设计。 |
| **反馈** | 短反馈循环——单元测试、集成测试、演示、小迭代。快速学习什么有效、什么无效。 |
| **简单** | 做可能有效的最简单的事。避免投机性设计和不必要的抽象。 |

## 实践

- **TDD（测试驱动开发）** — 红、绿、重构。先写失败测试，再写最小代码通过，然后重构。测试充当规范和安全网。
- **重构** — 持续小改进。保持测试通过；小步改善结构、命名和设计。
- **结对编程** — 两人一机。改善设计和质量；传播知识。不是每小时都必须，但对困难或关键工作是公认的实践。
- **简单设计** — 无重复；表达意图；最少元素（类、方法）；小而聚焦的抽象。仅在代码要求时才增加复杂性（如第三次重复）。

## 与工程素养的关系

Clean Agile 将敏捷价值观与工程素养联系起来：可持续节奏、测试即要求、重构作为循环的一部分、简单优于投机。在讨论 TDD、重构或结对如何支持敏捷性和质量时使用此参考。
