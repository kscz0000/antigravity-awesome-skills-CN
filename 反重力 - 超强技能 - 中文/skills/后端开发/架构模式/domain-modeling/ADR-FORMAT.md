# ADR 格式

ADR 位于 `docs/adr/` 中，并使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

懒加载创建 `docs/adr/` 目录——只有在需要第一个 ADR 时才创建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：上下文是什么，我们决定做什么，以及为什么。}
```

就是这样。ADR 可以只是一段话。其价值在于记录下做出了*什么*决策以及*为什么*——而不是填满各个章节。

## 可选章节

仅当它们能带来真正价值时才纳入。大多数 ADR 不需要这些。

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——在决策被重新审视时很有用
- **考虑过的备选方案** —— 仅当被否决的备选方案值得记忆时
- **后果** —— 仅当需要指出非显而易见的下游影响时

## 编号

扫描 `docs/adr/` 找到最大的现有编号，并加一。

## 何时提供 ADR

以下三条都必须为真：

1. **难以撤销** —— 事后改变主意的代价很大
2. **脱离上下文会令人惊讶** —— 未来的读者会看到代码并好奇"他们到底为什么要这样做？"
3. **是真正权衡的结果** —— 存在真正的备选方案，你因特定原因选择了其中一个

如果决策易于撤销，就跳过它——你反正会撤销它。如果不令人惊讶，没人会好奇为什么。如果没有真正的备选方案，除了"我们做了显而易见的事"以外没什么可记录的。

### 哪些符合条件

- **架构形态。** "我们在使用 monorepo。" "写模型采用事件溯源，读模型投影到 Postgres。"
- **上下文之间的集成模式。** "Ordering 和 Billing 通过领域事件通信，而不是同步 HTTP。"
- **带有锁定效应的技术选型。** 数据库、消息总线、身份认证提供商、部署目标。不是每个库——只是那些需要花一个季度才能换掉的东西。
- **边界和范围决策。** "客户数据由 Customer 上下文拥有；其他上下文只能通过 ID 引用。" 显式的"不"与"是"同样有价值。
- **有意识地偏离显而易见的选择。** "我们使用手写 SQL 而不是 ORM，因为 X。" 任何合理的读者会做出相反假设的事情。这些能阻止下一位工程师"修复"那些刻意为之的设计。
- **代码中不可见的约束。** "由于合规要求，我们不能使用 AWS。" "由于合作伙伴 API 契约的约束，响应时间必须低于 200ms。"
- **被否决的备选方案，当否决本身不显而易见时。** 如果你考虑了 GraphQL 并出于微妙的原因选择了 REST，请记录下来——否则六个月内又会有人提议 GraphQL。
