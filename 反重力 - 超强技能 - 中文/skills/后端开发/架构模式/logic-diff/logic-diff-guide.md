# 语义对比 — 逐步指南

## 步骤 1：确定共享规格

如果两个版本在共享规格的所有输入下产生相同行为，则它们语义等价。建立规格：
- 两个版本预期处理哪些输入？
- 两者应该产生哪些输出或副作用？
- 是否存在某个版本处理而另一个不处理的输入？（这本身可能就是差异所在。）

如果用户未说明规格，从共享测试、提交信息/PR描述或函数的文档契约中推导。明确陈述推导出的规格。

## 步骤 2：为每个版本构建独立前提

对版本 A 和版本 B **分别**应用 `../_shared/semiformal-checklist.md` 中的完整**前提构建清单**：
- 名称解析：相同标识符，在两者中解析结果一致？
- 类型契约：调用点处的类型假设一致？
- 状态前置条件：所需前置条件一致？
- 控制流：相同输入走相同分支？

此步骤通常在追踪开始之前就能发现差异——重命名的导入、更改的条件、不同的默认值。

## 步骤 3：追踪两个版本的常规用例

```
Scenario: [描述输入]

Version A — Trace:
1. [步骤]
2. [步骤]
Result: [值 / 副作用]

Version B — Trace:
1. [步骤]
2. [步骤]
Result: [值 / 副作用]

Verdict: [等价 / 在步骤N处不等价]
```

如果两者产生相同结果，进入步骤 4。如果出现差异，跳到步骤 5。

## 步骤 4：追踪边界用例

按暴露行为变更的可能性对边界条件排序，然后分别追踪版本 A 和 B 的最高风险场景：
- 空/null/零输入（捕获 L3 差异）
- 最大值（一个版本中的整数溢出而另一个没有）
- 错误输入（哪个版本抛出异常，哪个返回默认值？）
- 集合的首个和末个元素（一个版本中的差一错误）

默认预算：追踪常规用例加最多三个边界场景。仅在用户要求穷举对比、共享规格指定更多必需用例、或已追踪路径暴露出需要确认的新边界时扩展。

## 步骤 5：识别语义差异

语义差异是指 A 和 B 产生不同行为的任何场景。分类每种差异：
- A 中的 bug 在 B 中被修正（有意）
- 回归：B 破坏了 A 原本能处理的功能（无意）
- 行为变更：两者都能工作，但行为不同——用户需决定哪个正确
- 范围扩展：B 处理了 A 未处理的输入

对每个差异，写一个五字段发现（前提 → 追踪 → 差异 → 触发 → 修复），附上最匹配原因的 L-code。

## 步骤 6：分类等价性

- **✅ 语义等价：** 所有追踪场景中未发现差异。
- **⚠️ 条件等价：** 所有常规用例等价；仅在 [特定条件——精确说明] 时不等价。
- **❌ 语义不等价：** 在 [位置] 对 [场景] 确认存在行为差异。

注意："未发现差异"不等于"可证明等价"。说明追踪了哪些场景，并承认未测试场景可能仍有差异。

## 步骤 7：输出

使用 `report-template.md` 中的报告模板。将逻辑评分替换为等价性判定。在摘要中包含：
1. 判定
2. 最显著的差异（或"在已追踪场景中未发现差异"）
3. 任何应人工验证的未追踪场景
