# Fault Locate — 分步指南

## 步骤 1：理解故障

精确刻画故障：
- **观察到的行为：** 确切的错误消息、错误的输出值或症状
- **预期行为：** 来自测试断言、文档或用户描述
- **复现路径：** 哪些输入、环境、操作序列

如果有堆栈跟踪可用，请将最内层的帧（实际的崩溃点）与引起它的函数（可能在堆栈更高层）分开记录。

## 步骤 2：确定入口点

从距离故障最近的代码入口点开始：
- 失败的测试 → 测试函数
- 堆栈跟踪 → 最外层的**应用**帧（非库帧）
- 描述的症状（"API 返回 500"）→ 请求处理器

不要从 `main()` 开始 —— 从距离故障最近的函数开始。

在**故障锥**内开始：
- 应用代码的堆栈跟踪帧
- 直接构建失败状态的失败测试和夹具
- 失败路径调用到的本地被调用方
- 这些帧读取的配置/环境值

除非反向追溯涉及到，否则不要检查无关的模块、备选端点或相邻的测试。

## 步骤 3：从故障点反向追溯

从故障位置开始反向追溯："这个值/状态是从哪里来的？"

每一步都要：
- 将值追溯到其来源：声明、赋值、被调用方返回、外部状态
- 构建前提：对这个值有什么假设？它的实际值是什么？

**最低阈值**（依据 `../_shared/semiformal-guide.md`）：≥ 3 个实质性跳数，≥ 2 个位置锚点。否则将故障置信度下调为低。

**良好的反向追溯：**
```
Symptom: TypeError at service.py:7 — 'NoneType' object is not subscriptable on `result['transaction_id']`.
←1. [service.py:7] `result` came from line 6.
←2. [service.py:6] `result = charge(order)`; `charge` resolves to `payments.gateway.charge`.
←3. [gateway.py:3-4] `charge` returns None when `order.amount == 0`; all other paths return a dict.
←4. Failing test passes an order with `amount == 0`.
Root: gateway.py returns None on amount==0; service.py assumes dict unconditionally — L6.
```

## 步骤 4：正向追溯以确认

从疑似故障位置开始正向追溯：
- 执行是否到达了故障点？
- 该故障是否完全产生了观察到的症状？

如果是的，则假设得到确认。如果不是，则需要修正 —— 实际故障可能在更早的位置或不同的分支。

## 步骤 5：跨过程追踪

如果反向追溯牵涉到某个被调用方，则深入追踪：
- 在观察到的条件下，被调用方是否返回 `None`？
- 被调用方是否抛出了调用方未处理的异常？
- 被调用方是否变更了共享状态从而影响调用方后续行为？

故障通常不在正在查看的函数中 —— 而是在某个表现不符合假设的被调用方中（L6），或在一个解析结果不同的名字中（L1）。

深度预算：先沿着具体的失败路径跟进，并在根因确认后停止。如果需要超过四个本地调用，则陈述剩余调用链的假设并将故障置信度下调，除非有测试或堆栈跟踪直接确认更深的跳数。

## 步骤 6：确定根本偏离点

精确陈述：
- 偏离**起源**的具体行/表达式（不是崩溃点 —— 而是崩溃的原因）
- 被违反的前提
- 实际的值或行为
- 为什么会传播到观察到的故障

贡献因素：促成故障成立的条件（不是发现本身，而是修复的背景）。

## 步骤 7：分类与输出

1. 分配最能描述根本故障的 L-code（L1–L9）。
2. 评估故障置信度：
   - **高（High）：** 追溯完全确认 —— 前提→追溯→偏离形成从原因到症状的完整链条。
   - **中（Medium）：** 追溯强烈牵涉，但其中一步依赖于未验证的假设。
   - **低（Low）：** 故障合理，但在不执行的情况下无法排除其他原因。
3. 输出：主要故障（用于描述根本原因的五字段发现项）+ 贡献因素 + 补救措施。
