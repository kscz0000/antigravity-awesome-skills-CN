# 模式选择指南

> 选择架构模式的决策树。

## 主决策树

```
开始：你的主要关注点是什么？

┌─ 数据访问复杂度？
│  ├─ 高（复杂查询、需要测试）
│  │  → Repository模式 + Unit of Work
│  │  验证：数据源会频繁变更吗？
│  │     ├─ 是 → Repository值得引入间接层
│  │     └─ 否 → 考虑更简单的ORM直连
│  └─ 低（简单CRUD、单一数据库）
│     → ORM直连（Prisma、Drizzle）
│     简单=更好、更快
│
├─ 业务规则复杂度？
│  ├─ 高（领域逻辑、规则随上下文变化）
│  │  → 领域驱动设计
│  │  验证：团队有领域专家吗？
│  │     ├─ 是 → 完整DDD（聚合、值对象）
│  │     └─ 否 → 部分DDD（富实体、清晰边界）
│  └─ 低（主要是CRUD、简单验证）
│     → Transaction Script模式
│     简单=更好、更快
│
├─ 需要独立扩展？
│  ├─ 是（不同组件扩展需求不同）
│  │  → 微服务值得其复杂度
│  │  要求（必须全部满足）：
│  │    - 清晰的领域边界
│  │    - 团队 > 10名开发者
│  │    - 每个服务有不同的扩展需求
│  │  如不全满足 → 改用模块化单体
│  └─ 否（所有组件一起扩展）
│     → 模块化单体
│     证明需要时可后续提取服务
│
└─ 实时性要求？
   ├─ 高（即时更新、多用户同步）
   │  → 事件驱动架构
   │  → 消息队列（RabbitMQ、Redis、Kafka）
   │  验证：能处理最终一致性吗？
   │     ├─ 是 → 事件驱动有效
   │     └─ 否 → 同步+轮询
   └─ 低（最终一致性可接受）
      → 同步（REST/GraphQL）
      简单=更好、更快
```

## 三个问题（选择任何模式前）

1. **解决的问题**：这个模式解决什么具体问题？
2. **更简单的替代方案**：有没有更简单的解决方案？
3. **可延后的复杂度**：能否在需要时再添加？

## 危险信号（反模式）

| 模式 | 反模式 | 更简单的替代方案 |
|------|--------|------------------|
| 微服务 | 过早拆分 | 从单体开始，后续再提取 |
| Clean/六边形架构 | 过度抽象 | 先具体实现，后续再加接口 |
| 事件溯源 | 过度工程化 | 仅追加审计日志 |
| CQRS | 不必要的复杂度 | 单一模型 |
| Repository | 简单CRUD的YAGNI | ORM直连 |
