# 架构示例

> 按项目类型分类的真实架构决策。

---

## 示例1：MVP电商（独立开发者）

```yaml
需求:
  - 初期 <1000 用户
  - 独立开发者
  - 快速上市（8周）
  - 预算有限

架构决策:
  应用结构: 单体（对独立开发者更简单）
  框架: Next.js（全栈，快速）
  数据层: Prisma直连（不过度抽象）
  认证: JWT（比OAuth简单）
  支付: Stripe（托管方案）
  数据库: PostgreSQL（订单需要ACID）

接受的权衡:
  - 单体 → 无法独立扩展（团队规模不需要）
  - 无Repository → 可测试性较低（简单CRUD不需要）
  - JWT → 初期无社交登录（后续可添加）

未来迁移路径:
  - 用户 > 10K → 提取支付服务
  - 团队 > 3 → 添加Repository模式
  - 需要社交登录 → 添加OAuth
```

## 示例2：SaaS产品（5-10名开发者）

```yaml
需求:
  - 1K-100K 用户
  - 5-10名开发者
  - 长期项目（12个月以上）
  - 多领域（计费、用户、核心）

架构决策:
  应用结构: 模块化单体（团队规模最优）
  框架: NestJS（设计即模块化）
  数据层: Repository模式（测试、灵活性）
  领域模型: 部分DDD（富实体）
  认证: OAuth + JWT
  缓存: Redis
  数据库: PostgreSQL

接受的权衡:
  - 模块化单体 → 模块间有耦合（微服务不必要）
  - 部分DDD → 无完整聚合（无领域专家）
  - 后续RabbitMQ → 初期同步（证明需要时再添加）

迁移路径:
  - 团队 > 10 → 考虑微服务
  - 领域冲突 → 提取限界上下文
  - 读性能问题 → 添加CQRS
```

## 示例3：企业级（100K+用户）

```yaml
需求:
  - 100K+ 用户
  - 10+名开发者
  - 多个业务领域
  - 不同的扩展需求
  - 24/7可用性

架构决策:
  应用结构: 微服务（独立扩展）
  API网关: Kong/AWS API GW
  领域模型: 完整DDD
  一致性: 事件驱动（最终一致性可接受）
  消息总线: Kafka
  认证: OAuth + SAML（企业SSO）
  数据库: 多语言持久化（按需选工具）
  CQRS: 选定服务

运维要求:
  - 服务网格（Istio/Linkerd）
  - 分布式追踪（Jaeger/Tempo）
  - 集中式日志（ELK/Loki）
  - 熔断器（Resilience4j）
  - Kubernetes/Helm
```
