# 逻辑原型

一个小型交互式终端应用，让用户手动驱动状态模型。当问题涉及**业务逻辑、状态转换或数据形态**时使用——那种在纸面上看起来合理，但只有推入真实场景才感觉不对的东西。

## 何时选择此形态

- "我不确定这个状态机是否能处理 X 之后 Y 的边界情况。"
- "这个数据模型真的能让我表示……的情况吗？"
- "我想在写 API 之前先感受一下它应该长什么样。"
- 任何用户想要**按按钮看状态变化**的场景。

如果问题是"这应该长什么样"——选错分支了。使用 [UI.md](UI.md)。

## 流程

### 1. 陈述问题

在写代码之前，写下你正在原型化的状态模型和问题。一段话，写在原型的 README 或文件顶部的注释中。回答了错误问题的逻辑原型纯属浪费——让问题明确，以便日后检查，无论用户是在场还是离开后回来。

### 2. 选择语言

使用宿主项目所用的语言。如果项目没有明显的运行时（例如文档仓库），询问。

遵循项目现有的工具约定——不要仅为原型添加新的包管理器或运行时。

### 3. 将逻辑隔离为可移植模块

将实际逻辑——回答问题的那部分——放在一个小而纯粹的接口后面，日后可以提取出来放入真实代码库。围绕它的 TUI 是临时的；逻辑模块不应该是。

正确的形态取决于问题：

- **纯 reducer** — `(state, action) => state`。当动作是离散事件且状态是单个值时适用。
- **状态机** — 显式的状态和转换。当"现在哪些动作是合法的"本身就是问题的一部分时适用。
- **一组纯函数**，操作普通数据类型。当没有隐含的当前状态——只有转换时适用。
- **一个具有清晰方法表面的类或模块**，当逻辑确实拥有持续的内部状态时适用。

选择最符合所提问题的形态，*而非*最容易连接到 TUI 的形态。保持纯粹：无 I/O、无终端代码、无用于控制流的 `console.log`。TUI 导入它并调用它；没有反向数据流。

这正是让原型超越自身生命周期的价值所在。当问题被回答后，验证过的 reducer / 状态机 / 函数集可以被提取到真实模块中——TUI 壳被删除。

### 4. 构建最小的暴露状态的 TUI

构建为**轻量级 TUI**——每次 tick 清屏（`console.clear()` / `print("\033[2J\033[H")` / 等价方法）并重新渲染整个帧。用户应该始终看到一个稳定的视图，而非不断增长的滚动回溯。

每帧有两个部分，按此顺序：

1. **当前状态**，格式化打印且便于 diff（每行一个字段，或格式化 JSON）。字段名或章节标题使用**粗体**，次要上下文（时间戳、ID、派生值）使用**暗色**。原生 ANSI 转义码即可——`\x1b[1m` 粗体，`\x1b[2m` 暗色，`\x1b[0m` 重置。除非项目已有样式库，否则无需引入。
2. **键盘快捷键**，列在底部：`[a] 添加用户  [d] 删除用户  [t] 推进时钟  [q] 退出`。按键用粗体，描述用暗色，或反之——以可读为准。

行为：

1. **初始化状态** — 一个内存中的对象/结构体。启动时渲染第一帧。
2. **读取一个按键（或一行输入）**，分发到修改状态的处理函数。
3. **每次操作后重新渲染**整个帧——不要追加，替换。
4. **循环直到退出。**

整个帧应能放在一屏内。

### 5. 一条命令运行

向项目现有的任务运行器添加脚本（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户运行 `pnpm run <prototype-name>` 或等价命令——无需记住路径。

如果宿主项目没有任务运行器，将命令写在原型 README 的顶部。

### 6. 交给用户

给用户运行命令。他们会自己驱动；有趣的时刻是当他们说"等等，这不应该可能"或"嗯，我以为 X 会不一样"——这些是_想法_中的 bug，而这正是全部意义。如果他们想添加新动作，就添加。原型会演进。

### 7. 记录答案

当原型完成了它的任务，问题的答案是唯一值得保留的东西。如果用户在场，问他们学到了什么。如果不在，在原型旁边留一个 `NOTES.md`，以便在原型被删除前填入答案（或由你填入，如果你观察了整个过程）。

## 反模式

- **不要添加测试。** 需要测试的原型不再是原型。
- **不要连接真实数据库。** 使用内存存储，除非问题明确关于持久化。
- **不要泛化。** 不要"万一以后要支持 X"。原型回答一个问题。
- **不要把逻辑和 TUI 混在一起。** 如果 reducer / 状态机引用了 `console.log`、提示或终端转义码，它就不再可移植。将 TUI 保持为纯模块上的薄壳。
- **不要将 TUI 壳发布到生产环境。** 壳是为从终端手动驱动而优化的。它背后的逻辑模块才是值得保留的部分。
