# UI 原型

在单个路由上生成**多种截然不同的 UI 变体**，通过浮动底栏切换。用户在浏览器中翻阅变体，选择一个（或从各变体中各取所需），然后扔掉其余的。

如果问题是关于逻辑/状态而非外观——选错分支了。使用 [LOGIC.md](LOGIC.md)。

## 何时选择此形态

- "这个页面应该长什么样？"
- "我想在确定之前看看这个仪表盘的几个方案。"
- "试试设置页面的不同布局。"
- 任何用户原本要在脑海中花一天在三个模糊方案之间犹豫的场景。

## 两种子形态——强烈优先选择子形态 A

UI 原型在与应用的其余部分**紧密贴合**时更容易评判——真实的头部、真实的侧边栏、真实的数据、真实的密度。一个独立的临时路由是真空：每个变体在隔离状态下都看起来没问题。只要有合理的已有页面可以承载变体，默认使用子形态 A。只有当原型确实没有邻近的归属页面时才使用子形态 B。

### 子形态 A — 对现有页面的调整（优先）

路由已存在。变体在**同一路由**上渲染，由 `?variant=` URL 查询参数控制。现有的数据获取、参数和认证全部保留——只替换渲染部分。这是默认选项；除非有特定原因，否则选它。

如果原型是给还没有页面但*自然会存在于某个页面内*的东西（仪表盘的新区域、设置页面的新卡片、现有流程中的新步骤）——那仍然是子形态 A。将变体挂载在宿主页面内。

### 子形态 B — 新页面（最后手段）

仅当被原型的东西确实没有已有页面可以嵌入时使用——例如一个全新的顶层界面，或一个无法嵌入任何合理位置的流程。

创建**临时路由**，遵循项目已有的路由约定——不要发明新的顶层结构。命名要明显是原型（例如在路径或文件名中包含 `prototype`）。同样的 `?variant=` 模式。

在确定使用子形态 B 之前，自查一下：真的没有已有页面可以嵌入吗？空路由会隐藏有内容路由会暴露的设计问题。

两种子形态的浮动底栏是相同的。

## 流程

### 1. 陈述问题并选择 N

默认**3 个变体**。超过 5 个就不再是"截然不同"而变成噪音——封顶于此。

在原型位置或文件顶部注释中写下一行计划：

> "设置页面的三个变体，通过 `?variant=` 切换，在已有的 `/settings` 路由上。"

无论用户是否在场可以反馈，这都有效。

### 2. 生成截然不同的变体

起草每个变体。每个变体须遵循：

- 页面的目的及其可访问的数据。
- 项目的组件库/样式系统（TailwindCSS、shadcn、MUI、纯 CSS，任何已有的）。
- 清晰的导出组件名，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须**结构不同**——不同的布局、不同的信息层级、不同的主要交互方式，而非只是不同颜色。三个微调过的卡片网格不是 UI 原型，是壁纸。如果两个草稿太相似，在明确"不要使用卡片网格"的指导下重做一个。

### 3. 将它们连接起来

在路由上创建单个切换器组件：

```tsx
// 伪代码——适配到项目的框架
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

子形态 A（已有页面）：保持所有现有数据获取在切换器之上；每个变体只改变渲染的子树。

子形态 B（新页面）：`/prototype/<name>` 下的临时路由挂载相同的切换器。

### 4. 构建浮动切换器

屏幕底部居中的小型固定定位栏，包含三个部分：

- **左箭头** — 切换到上一个变体（循环）。
- **变体标签** — 显示当前变体键，如果变体导出了名称则也显示名称。例如 `B — 侧边栏布局`。
- **右箭头** — 切换到下一个变体（循环）。

行为：

- 点击箭头更新 URL 查询参数（使用框架的路由器——Next 的 `router.replace`、React Router 的 `navigate` 等），使变体可分享且刷新稳定。
- 键盘：`←` 和 `→` 箭头键也可切换。当 `<input>`、`<textarea>` 或 `[contenteditable]` 获得焦点时不拦截箭头键。
- 视觉上与页面区分（例如高对比胶囊、微弱阴影），使其明显不是被评估设计的一部分。
- 在生产构建中隐藏——用 `process.env.NODE_ENV !== 'production'` 或等价检查控制，防止意外合并将底栏发布给用户。

将切换器放在单个共享组件中，使两种子形态都可以复用。放在项目中共享 UI 所在的位置。

### 5. 交给用户

展示 URL（和 `?variant=` 键值）。用户会在方便时翻阅。有趣的反馈通常是**"我想要 B 的头部加上 C 的侧边栏"**——那就是他们真正想要的设计。

### 6. 记录答案并清理

一旦某个变体胜出，写下哪个及原因（提交消息、ADR、issue，或如果离线运行且用户尚未响应，在原型旁的 `NOTES.md` 中）。然后：

- **子形态 A** — 删除失败的变体和切换器；将胜出者融入已有页面。
- **子形态 B** — 将胜出变体提升为真实路由，删除临时路由和切换器。

不要留下变体组件或切换器。它们腐烂得很快，会困惑下一位读者。

## 反模式

- **仅在颜色或文案上不同的变体。** 那是微调，不是原型。真正的变体在结构上持不同意见。
- **变体之间共享过多代码。** 共享的 `<Header>` 可以；共享的 `<Layout>` 就违背了目的。每个变体应该自由抛弃布局。
- **将变体连接到真实变更操作。** 只读原型即可。如果变体需要变更，指向一个桩——问题是"这应该长什么样"，而非"后端是否能工作"。
- **将原型直接提升到生产环境。** 变体代码是在原型约束下写的（无测试、最少错误处理）。融入时要正确重写。
