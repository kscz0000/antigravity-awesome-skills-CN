# Unity AI 游戏创建器

## 概述

本技能可将一个原始的游戏概念转化为结构完整的 Unity 开发计划，包含 AI 生成的资源、场景、音乐、脚本以及可部署的构建产物。它引导智能体走完一个 5 阶段流水线 —— 从提取游戏核心维度，到为每个资源类别产出开箱即用的 AI 提示词 —— 并使用最新的 Unity 6+ 特性与 AI 工具生态。与泛用的 Unity 参考技能不同，本技能以工作流为驱动：用户提供创意，智能体交付一份完整、可执行的游戏开发路线图。

## 何时使用本技能

- 当用户描述一个游戏创意并希望获得完整的开发计划时使用
- 当需要为 3D 模型、贴图、音乐、音效或 UI 资源生成 AI 提示词时使用
- 当使用现代架构从零搭建一个新的 Unity 项目时使用
- 当需要根据概念描述创建场景蓝图时使用
- 当需要根据粗略想法撰写游戏设计文档（GDD）时使用
- 当在游戏工作流中接入 Unity AI Assistant、MCP 服务器或外部 AI 工具时使用
- 当规划变现策略、性能预算或应用商店上架时使用

## 工作原理

### 主管线

按顺序执行以下阶段。每个阶段必须先产出具体交付物，再进入下一阶段。

```
PHASE 1: IDEATION ──▶ PHASE 2: BLUEPRINT ──▶ PHASE 3: GENERATION ──▶ PHASE 4: ASSEMBLY ──▶ PHASE 5: DEPLOYMENT
  Game Brief            GDD + Scenes           AI Prompts + Assets     Project Setup          Build + Store
  Genre Analysis        Architecture           Scripts + Audio         Core Systems           QA + Submit
  Scope Assessment      Scene Blueprints       Voice + UI              Polish + Juice         Analytics
```

### 阶段 1：构思与深度分析

从用户的游戏创意中提取并澄清以下维度。仅对模糊之处提问，其余从上下文推断。

| 维度 | 要提取的内容 | 不清楚时 |
|------|--------------|----------|
| **类型** | 主类型 + 次类型混搭 | 推荐 3 种类型组合 |
| **平台** | 移动、PC、主机、WebGL、VR/AR | 根据范围推荐 |
| **视角** | 2D、2.5D、3D、俯视、横版、FPS、TPS | 从类型推断 |
| **美术风格** | 写实、风格化、像素、低多边形、动漫、绘画风 | 推荐 3 个选项 |
| **核心循环** | 30 秒一轮的重复玩法 | 从描述中识别 |
| **目标受众** | 年龄段、玩家画像、平台习惯 | 根据类型推荐 |
| **单局时长** | 平均单局游玩时长 | 从平台 + 类型推断 |
| **变现** | 免费游玩、买断制、混合 | 根据平台推荐 |
| **规模** | 独立开发者、小团队、工作室 | 明显时省略，否则询问 |
| **时间表** | MVP 周数、正式发布目标 | 给出切合实际的里程碑 |

**同时提供：**
1. **3 个参考游戏** —— 它们做得好在哪里，用户可以学到什么
2. **市场空白** —— 竞品忽略了什么机会
3. **核心差异点** —— 这个游戏最独特的一点
4. **风险评估** —— 技术风险和市场风险及其缓解措施

**规模校准：**

| 规模 | 时间表 | 团队人数 | 功能预算 |
|------|--------|----------|----------|
| 原型 | 1–2 周 | 独立 | 仅核心循环 |
| 竖切片 | 4–6 周 | 独立–2 人 | 1 个完整关卡 + 打磨 |
| MVP | 8–12 周 | 2–4 人 | 3–5 个关卡 + 存档 + UI |
| 正式发布 | 16–24+ 周 | 3–8 人 | 完整内容 + 多人 |

### 阶段 2：蓝图与设计

#### 游戏设计文档（GDD）

生成结构化的 GDD，覆盖：

1. **执行摘要** —— 电梯演讲、类型、独特卖点
2. **玩法** —— 核心循环图、成长线、技能、胜负条件、难度曲线
3. **世界与叙事** —— 背景设定、角色、关卡结构
4. **美术方向** —— 视觉风格、配色（十六进制色值）、参考描述、UI/UX 风格
5. **音频方向** —— 各场景的音乐风格、音效类别、配音需求
6. **技术规格** —— Unity 版本、渲染管线、性能预算、SDK
7. **变现策略** —— 收入模型、内购设计、广告位
8. **开发路线图** —— 阶段、里程碑、功能优先级矩阵、风险登记册

#### 场景蓝图

为每个场景产出包含以下层级的方案：

- **环境** —— 地面/地形、天空盒、灯光设置、道具及其位置
- **可交互对象** —— 交互时的行为
- **角色 / NPC** —— AI 行为、巡逻路径、对话触发
- **UI 覆盖层** —— HUD 元素、上下文提示
- **音频层** —— 背景音乐（情绪、节奏、循环点）、环境音效、触发音效
- **相机设置** —— 类型（Cinemachine / 固定 / 跟随），行为配置
- **激活的系统** —— 存档检查点、刷怪器、粒子效果

#### 架构蓝图

给出推荐的项目文件夹结构：

```
Assets/_Project/
├── Scripts/ (Core, Gameplay, UI, Data, Audio, Utilities)
├── Prefabs/ (Characters, Environment, UI, VFX)
├── Scenes/
├── Art/ (Models, Textures, Materials, Animations, UI_Assets)
├── Audio/ (Music, SFX, Ambience)
├── ScriptableObjects/
└── Resources/
```

### 阶段 3：AI 驱动的资源生成

为每个资源类别，输出当前最佳 AI 工具的开箱即用提示词。

#### 3D 模型提示词

动态推荐工具（Meshy.ai、Tripo3D、Rodin、Unity AI Assistant）并生成包含以下内容的提示词：
- 资源名称、美术风格、玩法用途
- 多边形预算、贴图分辨率、所需 PBR 贴图
- 是否可动画、缩放参照
- Unity 导入设置（缩放系数、法线、动画类型、材质处理）

#### 贴图与 2D 资源提示词

推荐工具（Leonardo.ai、Midjourney、DALL·E、Unity AI Assistant）并生成包含以下内容的提示词：
- 贴图类型（无缝平铺、精灵、UI 元素、概念图）
- 分辨率、风格、平铺要求、PBR 贴图
- Unity 导入设置（贴图类型、过滤模式、压缩、最大尺寸）

#### 音乐提示词

推荐工具（Suno、Soundraw、Sonauto）并生成包含以下内容的提示词：
- 曲目名称、场景上下文、情绪、节奏（BPM）、时长
- 乐器、参考曲目、循环点、动态分层
- Unity 导入设置（格式、加载类型、压缩、循环配置）

#### 音效提示词

推荐工具（ElevenLabs、OptimizerAI/SFX Engine）并生成包含以下内容的提示词：
- 音效名称、类别、玩法上下文、时长
- 变体数量、特征
- Unity 导入设置（格式、加载类型、3D 音效配置、变体数组）

#### 配音与旁白提示词

推荐工具（ElevenLabs、Play.ht、Coqui）并生成包含以下内容的提示词：
- 角色名、声音画像、对白台词、情绪、上下文

#### UI/UX 资源提示词

为 Unity UI Toolkit 或 AI 图像生成产出规格说明，包含：
- 元素名称、所在界面、视觉风格
- 状态（普通、悬停、按下、禁用）
- 尺寸、锚点行为、动画描述

### 阶段 4：组装与开发

#### 项目初始化清单

- Unity 版本（推荐最新版 Unity 6 LTS）
- 渲染管线（URP 用于移动/风格化，HDRP 用于高保真，Built-in 用于 2D）
- Build Settings、Player Settings（色彩空间、脚本后端、API 兼容）
- 必备包（Input System、Cinemachine、TextMeshPro、Addressables、Unity AI Assistant）
- Git LFS 配置（用于大体量资源）

#### 开发顺序（按周）

1. **基础** —— 项目搭建、GameManager、事件系统、场景管理、输入配置
2. **核心玩法** —— 玩家控制器、相机、核心循环、基础敌人
3. **内容与系统** —— 关卡搭建、UI 系统、音频管理器、存档/读档
4. **打磨与反馈** —— 特效/细节、成长线、新手引导、设置
5. **平台与发布** —— 性能分析、变现、数据分析、平台适配、提交

#### 脚本架构模式

- GameManager → 单例或服务定位器
- 事件 → 观察者模式（C# event + ScriptableObject Event）
- 存档系统 → JSON 序列化 + 加密
- 对象池 → 通用池（子弹、敌人、特效）
- 状态机 → 玩家状态、游戏状态、敌人 AI
- 音频 → 单例 + Audio Mixer Groups
- UI → UI Toolkit 结合 MVVM 或事件绑定
- 场景加载 → Addressables + 异步加载（带进度）
- 输入 → 新版 Input System 配合 Action Maps

#### Unity AI Assistant 与 MCP 集成

- Unity AI Assistant（Unity 6.3+）的安装步骤
- 用于外部 IDE/智能体控制的 MCP 服务器配置
- 用于场景操控、脚本编写、性能分析的编辑器内提示词示例

### 阶段 5：质量与部署

#### 性能预算

|                    | 移动      | PC        | 主机      |
|--------------------|-----------|-----------|-----------|
| 目标帧率           | 30/60     | 60/120    | 60        |
| Draw Call          | < 100     | < 500     | < 300     |
| 每帧三角形         | < 10 万   | < 200 万  | < 100 万  |
| 贴图内存           | < 150 MB  | < 1 GB    | < 512 MB  |
| 包体大小           | < 150 MB  | < 2 GB    | < 4 GB    |

#### 测试清单

核心循环稳定性、UI 响应性、存档/读档持久化、音频平衡、内存泄漏检查、帧率稳定性、输入设备覆盖、边缘情况（低电量、中断）、无障碍、本地化。

#### 商店上架指南

应用图标、特性图、截图、宣传视频、描述、关键词、隐私政策、年龄分级、内容分级 —— 附带用于生成营销资源的 AI 提示词。

## 示例

### 示例 1：休闲农场游戏

**用户：** "我想做一款带魔法元素的休闲农场游戏，面向移动端"

**智能体交付：**
1. **游戏简报** —— 休闲农场模拟 × 魔法生物，移动竖屏，风格化低多边形，5 分钟一局，F2P + 外观内购
2. **3 个参考游戏** —— Stardew Valley（循环）、Merge Magic（移动端 UX）、Moonstone Island（魔法农场融合）
3. **核心循环** —— 种植 → 照料 → 收获 → 出售 → 升级 → 发现魔法种子
4. **场景蓝图** —— 农场（主场景）、村庄市集、魔法森林、玩家小屋
5. **AI 资源套件** —— 15 个作物模型、8 个建筑模型、5 个角色模型、四季音乐轨道、农场环境音效、UI 套件
6. **技术方案** —— Unity 6 LTS + URP，移动端优先 30 fps，使用 Addressable 进行季节性更新
7. **6 周路线图** —— 基础 → 核心玩法 → 内容 → 变现 → 打磨 → 软启动

### 示例 2：多人射击游戏

**用户：** "做一款节奏快、霓虹视觉的 PC 竞技场射击游戏"

**智能体交付：**
1. **游戏简报** —— 竞技场 FPS，PC/Steam，风格化霓虹赛博朋克，10 分钟一局，买断制 $9.99
2. **核心循环** —— 出生 → 拾取 → 战斗 → 消灭 → 得分 → 复活
3. **技术方案** —— Unity 6 + HDRP、Netcode for GameObjects、专用服务器、120 fps 目标
4. **AI 资源套件** —— 竞技场模型、武器模型、霓虹材质/Shader 提示词、电子音乐轨道、武器音效

### 示例 3：2D 益智手游

**用户：** "像 Wordle 那样简单但换成颜色的益智游戏"

**智能体交付：**
1. **游戏简报** —— 休闲益智，移动竖屏，扁平极简，2 分钟一局，F2P + 激励广告
2. **规模** —— 1 周出原型，3 周出 MVP
3. **场景蓝图** —— 主菜单、棋盘、结算界面、每日挑战
4. **AI 资源套件** —— UI 精灵、庆祝粒子、平静环境音乐、点击/滑动音效

## 最佳实践

- ✅ **从核心循环开始** —— 在做任何其他事之前先把 30 秒循环打磨好
- ✅ **先分析再优化** —— 用 Unity Profiler 找出真正的瓶颈
- ✅ **用 ScriptableObject 管理数据** —— 数据与逻辑解耦，更灵活
- ✅ **生成多份 AI 资源变体** —— 从 3–5 个生成结果中挑最好的
- ✅ **尽早用真机测试** —— 模拟器会漏掉平台特有的问题
- ✅ **从第一天起就接入版本控制** —— Git + LFS，频繁提交
- ❌ **不要硬编码工具选择** —— 始终向用户给出多种替代方案
- ❌ **不要跳过 GDD** —— 即使是独立项目，书面设计也很有价值
- ❌ **不要过早优化** —— 先让它跑起来，再做对，再做快
- ❌ **不要未经审核就上线 AI 资源** —— 检查每个工具的授权条款

## 局限性

- 本技能不能替代环境相关的验证、测试或专家审核。
- AI 生成资源用于商业用途前，需根据每个工具的服务条款进行法律审查。
- 性能预算仅作参考 —— 始终在实际目标硬件上做性能分析。
- AI 工具及其能力迭代迅速 —— 推荐前需确认其当前可用性与定价。
- 若缺少必要输入、权限、安全边界或成功标准，应停下来向用户确认。

## 安全与合规说明

- 本技能不包含 shell 命令、网络请求或凭证处理。
- 所有 AI 工具推荐均为外部服务；本技能不执行 API 调用也不存储令牌。
- 资源生成提示词仅为文本指引 —— 不会发生自动下载或文件变更。

## 常见陷阱

- **问题：** 用户给出的创意过于模糊，例如"做个游戏"
  **解决方案：** 在生成第一版前，先问 2–3 个有针对性的问题（类型、平台、规模）

- **问题：** 范围蔓延 —— 试图一口气做完所有东西
  **解决方案：** 使用规模校准表和功能优先级矩阵（必须有 / 应该有 / 可以有 / 不做）

- **问题：** AI 生成的资源与美术风格不匹配
  **解决方案：** 在每个资源提示词中都带上 GDD 的美术方向关键词，保持一致性

- **问题：** 目标平台上出现性能问题
  **解决方案：** 尽早设定平台专属预算，并按周进行性能分析对比
